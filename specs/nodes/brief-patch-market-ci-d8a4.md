## Pinned decisions (binding constraints this lane carries)

- **Graft C — fire on the artifact that merges the winner's code; close C's specs-only hole.** The rejected Candidate C folded the gate into `pr-evidence.yml`, whose in-job skip waves through specs-only PRs — the exact "a multi-patch brief whose PR skips the comparison node" case the intent's headline acceptance names — so the patch market's graph-only steps merged green and ungated (decision "Why each rejected candidate lost", C). This lane's `patch-comparison.yml` **must therefore run the gate on the PR that actually merges the winner's patch code**, and **must NOT inherit `pr-evidence.yml`'s specs-only/docs-only skip**. The safe default is **no event-level `paths:` filter and no specs-only in-job skip** — the workflow runs the gate on every PR (the gate itself decides, from PR→brief mapping, whether a market applies; that decision is domain-backend's, not the workflow's). If any path/scope guard is added at all, it must be proven to still cover a code-bearing patch-merge PR (a PR touching `tools/**`, `.claude/**`, app code, etc.) — never skip it.
- **Fix 5 — document the out-of-diff branch-protection wiring.** The named `patch-comparison` check blocks nothing until a repo admin marks it required in GitHub branch-protection settings, which live outside this (or any) diff (`docs/branch-protection.md` lines 3-7). This lane must document that wiring in two places: (a) a header comment in `patch-comparison.yml` stating the workflow only reports and that blocking requires the branch-protection setting, and (b) a short addition to `docs/branch-protection.md` adding `patch-comparison` to the required-checks record. Until that admin step lands, the workflow runs and reports but blocks nothing.
- **Always-report invariant (inherited from the precedent, reinforced by release-critic).** Because `patch-comparison` will be a *required* check, the workflow must post a status on **every** PR — including PRs with no patch market. A non-market PR must report **success** (the gate passes when there is provably ≤1 live competitor / no mapped multi-patch market), never "skipped/never-run", or marking the check required strands every non-market PR. The verdict logic that distinguishes "no market here → pass" from "unresolved market → fail" is the **domain-backend** gate's `evaluatePatchGate`; this lane's only obligation is to *run* the gate on every PR so a status is always posted.
- **`fetch-depth: 0` + `GATE_BASE` are mandatory, not optional.** Modelling on `pr-evidence.yml` is not enough; the diff-aware gate *throws* (a CI infra-error red, conflated with a real "market unresolved" red) on a shallow checkout where `resolveBase` cannot resolve a base. This lane MUST set `fetch-depth: 0` on checkout and pass `GATE_BASE: ${{ github.event.pull_request.base.sha }}` (reliability-ops finding 3).
