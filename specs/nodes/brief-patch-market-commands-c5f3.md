- **The implementation-agent invocation `/propose-patches` reuses per strategy.** `/implement-brief` is the existing "implement exactly what an approved brief specifies — code and project files only; this command performs no graph writes and delegates to no agent" pattern (`implement-brief.md` lines 4-11), including its scope-discipline "If the brief seems wrong … STOP and ask the human — never expand scope silently" (lines 9-10). `/propose-patches` runs this same implementation work per strategy with an injected strategy directive on a dedicated branch.
- **The reviewer `/compare-patches` invokes.** `contract-reviewer` is the existing judgement-only agent that "writes nothing — graph-maintainer records" and summarises candidates + consequences (`.claude/agents/contract-reviewer.md` lines 7-8, 22-24). `/compare-patches` invokes it per candidate branch and pairs it with the existing `/detect-drift` command (`detect-drift.md`), which deterministically maps a branch/PR diff and judges observable-behaviour drift, returning a `drift-finding` or "no drift" — `/compare-patches` consumes that drift verdict as one of its six scoring axes.
- **The comparison-node + `compares`-edge authoring convention.** `/review-contracts` already records "exactly ONE `comparison` node — body sections: the candidate trade-off table, the critic findings grouped by perspective, and the case against each candidate — with one `compares` edge per live candidate", REPLACING any existing comparison rather than authoring a second (`review-contracts.md` lines 28-35, step 4). `/compare-patches` reuses this exact one-comparison-per-market, replace-not-author idempotency, with `compares` edges now pointing at `patch` nodes (the widened target the data-migration lane authors).
- **The selection-decision + `selects`-edge + scope-integrity convention.** `/approve-contract` records the `decision` node, its `selects` edge, and the lifecycle status changes; its body "must record the accepted trade-off and why each rejected candidate lost" and applies the work-class quorum pre-check (`approve-contract.md` lines 12-23). `/select-patch` mirrors this decision-authoring shape (now `selects → patch`) and carries the Phase-6 scope-integrity branches verbatim from CLAUDE.md rule 5 (supersede the brief / capture a follow-up intent / return to human approval). The winner's branch then proceeds to `/prepare-evidence` (`prepare-evidence.md`), which authors the evidence node, its `evidences` edge, and the `touches` edges — `/select-patch` hands off to it, never duplicating that work.
- **Scope-integrity prose source.** CLAUDE.md "Scope-integrity" rule 5 (the three branches: brief boundary wrong → supersede the brief; contract incomplete → `/capture-intent` a follow-up; selected work changes intended behaviour → return to human approval). `/capture-intent` (`capture-intent.md`) is the existing command `/select-patch` names for the follow-up-intent branch.
- **The override/escape-hatch the operator hits at the merge wall.** The `patch-comparison` named check, the `waives → patch-comparison` override, and the `patch-comparison.yml` workflow that fails the merge live in OTHER lanes (data-migration registers the check; domain-backend builds `spec:patch-gate`; observability-release wires the workflow). ## Pinned decisions (binding constraints this lane carries)

- **Fix 4 (the lane's headline binding):** ALL FOUR commands MUST end with the closing status-report contract every existing command guarantees — the ids of every node/edge touched plus each updated node's status, then "Stop there." This is non-negotiable per `decision-patch-market-ci-gate-8a2f` mandatory fix 4 (ux). No command may end on the delegated graph-write without re-stating what changed.
- **Thin commands, fat agents (repo convention, from the decision's "thin commands delegating to fat agents/graph-maintainer" framing):** these four files orchestrate and delegate; they author no graph data themselves and contain no diff/gate logic. The merge gate, schema, and validate rules are explicitly out of this lane.
- **Never select (intent lines 33-34, contract Acceptance example 1):** `/compare-patches` ends by asking for a human decision and NEVER selects or ranks — selection is `/select-patch`'s human-recorded `decision`, exactly as `/review-contracts` never selects and `/approve-contract` records the human choice.
- **Within-lane synthesis only; across-lane combination is integration (intent lines 39-41, contract Non-scope):** `/synthesize-patches` combines patches that all `competes-for` the SAME lane brief; it states explicitly that across-lane combination is `/integrate`'s job, never synthesis.
- **Selection supersedes losers AND the parents of a selected synthesis patch; keeps their branches; closes their draft PRs (intent lines 42-48, contract Acceptance example 1 & 5):** `/select-patch` sets every losing patch — including a selected synthesis patch's parent patches — to `superseded`, closes their draft PRs via `gh`, and KEEPS their branches.
- **Scope-integrity in patch review (Phase-6 / CLAUDE.md rule 5, contract Scope line 47):** `/select-patch` carries all three branches — if the comparison exposed the brief boundary was wrong, supersede the brief (never widen the winner); if the contract was incomplete, `/capture-intent` a follow-up; if the selected patch changes intended behaviour, STOP and return to human approval before proceeding.
- **Graph-first PR→brief mapping is NOT this lane's mechanism (Fix 3 lives in domain-backend), but its data is:** `/propose-patches` is the command that AUTHORS the `patch` node carrying `branch` (= the exact head branch `patch/<brief-slug>/<strategy>`) and the `competes-for` edge that the gate later maps on. This lane must keep `patch.branch` byte-equal to the real created branch so the domain-backend graph-first mapping has a sound join key — the command sets `branch` to the same string it `git checkout -b`'d.

## Ordered implementation steps

Within-lane the four files are independent flat markdown and may be authored in any order; the recommended order matches the operator lifecycle (propose → compare → optionally synthesize → select). Each file is the established thin-command shape: `description:` frontmatter, an `Input:` line keying off `$ARGUMENTS`, a short numbered orchestration body, a delegate-to-graph-maintainer step ending in the mutating reminder, and the Fix-4 closing report.

1. **Create `propose-patches.md`** — `/propose-patches <brief-id> <n> <strategy-list>`.
   - `Input:` brief node id (first token), `n`, and a comma-separated strategy list. Locate the brief via `specs/indexes/by-type.yaml`; confirm it is a LANE brief or a single brief (read its `lane`/single-brief shape) — stop and report if the id is not a brief.
   - For each named strategy: `git checkout -b patch/<brief-slug>/<strategy>` off the lane's base; invoke the implementation agent (the `/implement-brief` flow) against the brief PLUS an injected strategy directive that biases HOW the brief is implemented (the strategy), carrying `/implement-brief`'s scope-discipline ("if the brief seems wrong … STOP and ask the human — never expand scope silently"). Open a DRAFT PR for the branch via `gh pr create --draft`.
   - Delegate to graph-maintainer to create one `patch` node per strategy — frontmatter `status: candidate`, `branch:` set BYTE-EQUAL to the created branch name `patch/<brief-slug>/<strategy>`, `strategy:` set to the strategy token, a body holding the evidence summary (what the strategy did) — plus one `competes-for` edge (`patch —competes-for→ brief`) per patch, and set `patch_market: true` on the brief frontmatter. Then regenerate indexes and validate; nothing committed on red.
   - **Closing report (Fix 4):** each new `patch` id + its branch + its `candidate` status, the brief id now carrying `patch_market: true`, the draft-PR urls, and a pointer that the market is now OPEN — its merge is gated by the `patch-comparison` check until `/compare-patches` then `/select-patch` resolve it. Stop there.
   - DEPENDS ON the data-migration lane's `patch` node type, `competes-for` edge type, and `patch_market` brief flag EXISTING in the schema, else graph-maintainer's `pnpm spec:validate` reds (unknown node/edge type). Call this dependency out at the top of the file as a precondition graph-maintainer enforces.

2. **Create `compare-patches.md`** — `/compare-patches <brief-id>`.
   - `Input:` brief node id. Locate the brief and its live (non-superseded) candidate patches via the brief's `competes-for` incomings in `specs/indexes/incoming.yaml`, reading only the named patch node files.
   - For each candidate branch: run `contract-reviewer` (judgement-only, writes nothing) and run `/detect-drift <patch-branch>` to obtain that branch's observable-behaviour drift verdict.
   - Delegate to graph-maintainer to author exactly ONE `comparison` node for the market — body sections: a candidate trade-off table SCORING each patch on the six axes the intent names (contract fit, scope control, simplicity, test quality, drift risk, rollback safety), the reviewer/drift findings grouped per candidate, and the case against each candidate — with one `compares` edge per live candidate patch (`comparison —compares→ patch`). If a `comparison` already covers this market, REPLACE it (regenerate body + re-author `compares` to the current live candidate set) rather than author a second; never leave two comparison nodes for one market (the `/review-contracts` step-4 idempotency, reused). Regenerate indexes and validate; nothing committed on red.
   - **Closing report (Fix 4):** the comparison id, the patch ids it `compares`, each candidate's six-axis line, the `patch-comparison` check named by its exact literal as the gate the operator must clear, and the next command `/select-patch`. End by asking for a HUMAN DECISION. NEVER select or rank. Stop there.
   - DEPENDS ON the widened `compares.target` (`[contract, patch]`, data-migration lane) so `comparison —compares→ patch` validates.

3. **Create `synthesize-patches.md`** — `/synthesize-patches <brief-id> <patch-id-list> <instruction>`.
   - `Input:` brief node id, a comma-separated patch-id list (the parents to combine), and a free-text human instruction for HOW to combine them. Locate the brief and confirm every named parent patch `competes-for` THIS brief (reject any parent competing for a different brief — across-lane combination is integration, not synthesis — and stop).
   - `git checkout -b patch/<brief-slug>/synthesis`; combine the named parent patches per the human instruction (the implementation work). Open a draft PR via `gh`.
   - Delegate to graph-maintainer to create a synthesis `patch` node (`status: candidate`, `branch: patch/<brief-slug>/synthesis`, `strategy: synthesis`, body carrying the evidence summary of the combination) with a `synthesizes` edge to EACH named parent patch (`patch —synthesizes→ patch`) AND a `competes-for` edge to the SAME lane brief the parents compete for. Regenerate indexes and validate; nothing committed on red. (The `synthesis_parentage` structural rule the domain-backend lane builds reds a synthesis patch with fewer than two `synthesizes` edges — so the command must require ≥2 parents up front.)
   - State explicitly in the body: a synthesis patch competes for the SAME lane brief as its parents; ACROSS-LANE combination is `/integrate`, never synthesis.
   - **Closing report (Fix 4):** the synthesis patch id + its branch + `candidate` status, the parent patch ids it `synthesizes`, the brief it `competes-for`, the draft-PR url, and that it now competes in the same market (re-run `/compare-patches`, then `/select-patch`). Stop there.
   - DEPENDS ON the `synthesizes` edge type and the synthesis-parentage rule (data-migration declares the edge; domain-backend builds the rule).

4. **Create `select-patch.md`** — `/select-patch <patch-id> <rationale>`.
   - `Input:` winner patch id (first token); the remainder is the free-text rationale for the decision body. Locate the winner via `specs/indexes/`, its brief via the winner's `competes-for` edge, and the sibling competing patches via the brief's `competes-for` incomings.
   - **Scope-integrity gate FIRST (CLAUDE.md rule 5, Phase-6):** before recording anything, apply the three branches — if the comparison exposed the brief boundary was wrong (contract intact), instruct graph-maintainer to SUPERSEDE the brief with a corrected brief (`supersedes` edge), never edit the approved brief in place and never widen scope inside the winner; if the contract was incomplete (intended behaviour unchanged), STOP and `/capture-intent` a follow-up intent for the missing scope, do not widen the winner; if the selected work CHANGES intended behaviour, STOP and return to human approval (a new `decision` is required) before proceeding.
   - Delegate to graph-maintainer to author one `decision` node (`decided_by` required; rationale + accepted trade-off + why each loser lost in its body, citing the market's `comparison` node id), its `selects` edge to the winner (`decision —selects→ patch`), and the status changes: every LOSING patch — INCLUDING the parent patches of a selected synthesis patch — set to `superseded`; the winner set to `selected`. Regenerate indexes and validate; nothing committed on red.
   - Close the losing patches' DRAFT PRs via `gh pr close`, but KEEP their branches (do not delete the refs).
   - Hand the winner's branch off to `/prepare-evidence` (which authors the evidence node, the `evidences` edge, and the `touches` edges — not this command).
   - **Closing report (Fix 4):** the decision id, the winner patch id now `selected`, every superseded loser id (and superseded synthesis-parent ids), the brief id (and any superseding-brief id / follow-up-intent id if a scope-integrity branch fired), the closed PR numbers, and the next step `/prepare-evidence <winner-branch>`. Stop there — do not prepare evidence inline.
   - DEPENDS ON the widened `selects.target` (`[contract, patch]`, data-migration lane) and on Fix 1's `intentsForContract` type-guard (domain-backend) so a `selects → patch` edge does not red `class_market_quorum`/`comparison_required`. State this as a precondition the graph-maintainer validate step relies on.

5. **Cross-file consistency pass.** Confirm all four files: (a) end with the Fix-4 closing-status-report block; (b) delegate every graph write to graph-maintainer and end its mutating step with the `pnpm spec:index && pnpm spec:validate` reminder (env form `node_modules/.bin/tsx tools/spec.ts index && … validate` per the project memory) and "nothing committed on red"; (c) name the `patch-comparison` check by its exact literal where the operator meets the gate; (d) never select except in `/select-patch`'s human-recorded decision. No `spec:index`/`spec:validate` is triggered by creating these files themselves — `.claude/commands/` is outside the validated graph — but each command's BODY must instruct that step for its own runtime mutations.
