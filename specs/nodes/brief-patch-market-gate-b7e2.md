## Grounding (reuse, don't reinvent)

- **The pure-decision / git-runner split is the established gate shape.** `tools/gate.ts` exports a pure `evaluateGate(spec, input: GateInput): GateResult` and a thin `runGate(spec)` that derives `addedEdgeIds`/`addedNodeIds`/`today` (`gate.ts:63-136`); `tools/checkdiff.ts` mirrors it exactly â€” `evaluateCheckDiff(spec, input: CheckDiffInput): GateResult` + `runCheckDiff(spec)` (`checkdiff.ts:100-158`). `evaluatePatchGate`/`runPatchGate` MUST follow this split verbatim so the core is offline unit-testable (Graft A).
- **Git adapter helpers, fail-closed.** `tools/gitdiff.ts` exports exactly four primitives: `resolveBase()` (reads `$GATE_BASE`, else `git merge-base origin/HEAD HEAD`, verifies the base is a real commit, throws otherwise â€” `gitdiff.ts:25-48`), `addedNodeIds(spec, base)` (`:90-108`), `addedEdgeIds(spec, base)` (`:80-88`), and `changedFiles(base)` (`git diff --name-only <base>...HEAD`, `:126-135`). NONE of them returns the current branch name â€” the head-branch leg of the PRâ†’brief mapping (Fix 3) is net-new git plumbing this lane must add (see step 6), failing closed in the same throw-on-error style.

## Pinned decisions (binding constraints this lane carries)

- **Graft A â€” one predicate, two consumers, no two-places drift.** A single exported live-competitor/resolved predicate is reused by BOTH `evaluatePatchGate` AND the two structural rules. It is built on `liveSourcesByEdge(... "competes-for" ...)` (extend `coverage_traversal.ts` or add a small sibling module that re-exports/wraps it); it must NOT be re-implemented inside `patch_gate.ts`. "Live competitor" = distinct, resolved `competes-for` sources excluding `superseded` AND `selected` (Fix 2).
- **Graft A â€” pure gate core.** `evaluatePatchGate(input)` is a pure function over an immutable input â€” the live `{graph}` (`LoadedSpec`), the `resolvedBrief` (the brief the PR maps to, plus enough to count its competitors), and `today` â€” returning a `GateResult` (`{ pass, reason }`). No git, no `process.env`, no `new Date()` inside the core; all impurity lives in `runPatchGate()`.
- **Graft C â€” reuse, don't reinvent.** Reuse the existing `override`/`waives` mechanism, `toDateString`, and the `gitdiff.ts` helpers (`addedNodeIds`, `changedFiles`, `resolveBase`). The override is a DEDICATED `patch-comparison` waiver (the `PATCH_COMPARISON_CHECK` constant) â€” never overload `pr-evidence`. The override must be diff-bound (the `override` node + `waives â†’ patch-comparison` edge ADDED in this PR, same as `evaluateGate` requires its override be in `addedNodeIds`/`addedEdgeIds`), so a pre-existing untouched waiver does not suppress a later market.
- **Graft C â€” fire on the artifact that merges the winner's code.** The gate evaluates the PR actually merging the winner; `runPatchGate()` derives PR state from `resolveBase()` + the diff helpers (the workflow wiring `GATE_BASE`/`fetch-depth: 0` belongs to the observability-release lane, NOT this one).
- **Fix 1 â€” type-guard `intentsForContract` callers.** In BOTH `class_market_quorum.ts` and `comparison_required.ts`, before treating a `selects` target as a contract, guard `asString(target.data["type"]) === "contract"` (skip patch-typed targets). The widened `selects`/`compares.target` must never feed patch ids into the contract-only logic.
- **Fix 2 â€” exclude `selected` as well as `superseded`.** The shared live-competitor predicate excludes BOTH `superseded` and `selected` patches (a chosen winner must not count as a live competitor, or the post-selection steady state never reaches a clean â‰¤1-competitor pass). This is the single place that count is defined, so the gate and the rules inherit it identically.
- **Fix 3 â€” graph-first, fail-closed PRâ†’brief mapping.** Map graph-first: the `patch` node whose `branch` field == the PR head â†’ its `competes-for` brief. Fail-closed default: when the head maps to NO patch node AND the branch does not parse as `patch/<brief-slug>/<strategy>`, the PR is NOT a patch-market PR â†’ the gate PASSES (it must never strand an unrelated PR). When the head maps to a patch node (or parses to a brief) but the mapping is AMBIGUOUS (â‰Ą2 distinct briefs reachable) OR resolves to a brief with >1 live competitor and no covering `comparison`+`selects`, the gate FAILS (it must never silently pass an unmapped/under-resolved multi-patch merge). These two opposite cells are pinned here, not left to the implementer, because the critic panel flagged the no-match default as the single load-bearing decision.
