## Candidate trade-off table

Both candidates ship an identical coverage-and-coherence core; they diverge on exactly one axis — **does `spec:validate` machine-check the `integration` node body, or is body completeness left to agent discipline plus a CLAUDE.md convention?** Candidate A (`contract-lane-integration-convention-body-4c1f`) leaves the body to convention; Candidate B (`contract-lane-integration-validated-body-7b2e`) adds a second handler that machine-checks required section presence.

| Axis | Candidate A — convention body | Candidate B — validated body |
|---|---|---|
| **`spec:validate` machine-checks the integration body** | No. Body is governed only by `requires_body: true` (non-empty) plus the CLAUDE.md convention and the `integration-reviewer` agent. A one-character body is structurally complete. | Yes. A second handler parses the markdown body and emits a finding (red) for each required section that is absent or empty. |
| **New validate rules added** | One (the shared coverage-and-coherence rule). | Two (the shared coverage rule **plus** `integration_body_sections`). |
| **Validator surface / blast radius** | Smallest. Stays a graph/frontmatter-shape checker; no new parsing primitive. | Largest. Introduces the codebase's **first markdown-body-prose parser** inside `spec:validate` — no existing handler reads `node.body` as structured data. New parsing/heading-match/fenced-code surface, novel `HANDLERS` registration, and a new false-red/false-green failure class on the merge gate. |
| **Fit with the existing "node bodies are convention, not validated" precedent** | Consistent. Preserves the flat doctrine (`requires_body` only) that CLAUDE.md fixes for every body-bearing type, including the analogous `comparison` body. | Breaks it — and breaks it *asymmetrically*: `integration` becomes the lone section-checked body while the structurally-analogous `comparison` stays convention-only. Establishes a case-by-case per-type body-validation policy the codebase has so far avoided. |
| **Body-completeness guarantee** | None at the gate. An `integration` whose COMBINED-test-run / joint-compliance-verdict section is hollow or absent still goes green; completeness is process-gated only. | Section *presence* is gated: an omitted required section turns the tree red. Stops at presence — a present-but-hollow section (one sentence, no commands) still passes. Raises the floor from "empty" to "one non-empty line under each heading." |
| **Honesty bound (what green actually asserts)** | Green asserts "an integration node exists and is wired to final evidence for every live brief." Risk: downstream readers over-read this as "the combined change was verified." A documents this gap honestly. | Green asserts "all required sections are present and non-empty." Risk: this reads as stronger assurance ("the integration was verified / tests ran") than it certifies; a *form* gate can launder a hollow body into a more authoritative-looking green. B documents this (example 8) honestly. |
| **IDENTICAL in both (the shared core)** | The coverage-and-coherence handler, scoped to `selects`-edged intents; the new `integration` node type and `integrates` edge; the laned `/decompose-lanes` → `/write-tests` → `/integrate` lifecycle and the laned-brief-`implemented`-while-intent-`open` hand-off; the `/prepare-evidence` STOP-and-ask + `touches` wiring; the `capability-spec-tooling-1a2b` widening to own `CLAUDE.md` + `tests/**`; the `specs/{nodes,graph,indexes}/**`-intentionally-unowned resolution. **All four acceptance lines (L1–L4)** are the same. The **`intent-status-coherence-d4f2` subsumption** (driven `open → addressed` via a note in the selecting decision, replacing d4f2's `unresolved.yaml` reporting with hard-fail `Finding[]`) is identical. The **capability-recursion resolution** (this PR wires its own evidence's `touches` to the widened tooling capability) is identical. The **status-safety constraints** (`/integrate` refuses `final` until every lane is covered; commit only on green) are identical. |

## Critic findings by perspective

- **Spec.** *A:* the "d4f2 driven to `addressed`" claim is internally contradictory — the rule is scoped to `selects`-edged intents but d4f2 has no incoming edges, so the rule structurally skips it; and `/integrate` writing `touches` edges from the integration node collides with the `touches` `source: evidence` constraint the contract relies on as "free." *B:* shares both shared-core defects (d4f2-unreachable, `touches`-source collision, and the `unresolved.yaml`-reporting-replaced-not-generalised point); B's distinguishing body-section rule adds a novel markdown parser whose parse direction (iterate required-config vs scan found-headings) and false-red behaviour must be pinned, and whose `kind` string is the first body-parsing handler with no precedent to copy.

- **Product.** *A:* solves the coverage half faithfully but makes machine-true a cheaper proxy ("an integration node exists and is wired") than the intent's stated value ("the combined change was actually verified as a whole"), leaving the body — the part the intent specified precisely because it is the value — to agent trust. *B:* closes the specific body-omission gap (a missing COMBINED-test-run section turns the tree red — a real, falsifiable gain over A) but its presence-only check can be satisfied by boilerplate, so a green tree asserts "sections present" while downstream users over-read it as "the change was verified"; also risks training authors to fill legitimately-empty sections ("N/A") to pass.

- **UX.** *A:* the integration-completeness failure is invisible to the operator — a thin/untested body fails silently behind a green `spec:validate` every downstream human trusts; plus the shared `/integrate` closing report, partial-integration remediation path, and `/prepare-evidence` STOP-and-ask prompt content are unspecified. *B:* turns A's silent gap into a discoverable red (a real UX win) but ships a new false-red experience (operator wrote the section, parser insists it's missing) whose finding-message clarity and authoring-time section checklist are unspecified, and a green that can falsely read as "integration sound." Shared-core gaps stand against both.

- **Architecture.** *A:* the integration node is bound to its contract only transitively through `integrates → evidence` with no one-integration-per-contract invariant, and the coverage handler is a third hand-rolled copy of the proposes/selects/decomposes traversal rather than a shared helper; the `capability-spec-tooling-1a2b` widening makes a cohesive capability a grab-bag. *B:* inherits all of A's architecture findings verbatim, and additionally makes `spec:validate` parse body prose for the first time, breaks the "bodies are convention" doctrine asymmetrically (integration validated, the analogous comparison not), and scatters the required-heading vocabulary across three files (config, agent prompt, CLAUDE.md) with no single source of truth.

- **Security-privacy.** Both: **no concern on the personal/sensitive-data axis** — this change handles no PII or secrets; all findings are integrity/CI-trust. *A:* the agent-only integration body sits on the sensitive-path (`specs/schema/**`) coverage chain with no machine backstop, and the `/integrate`-authored `touches` edge and widened tooling capability are unspecified on which capability may govern a sensitive change. *B:* inherits A's coverage-chain trust gaps and adds a new cost — the first untrusted-markdown-body parser inside the CI merge gate — unspecified on fail-closed behaviour and input hardening (heading-spoofing false-green, pathological-input DoS).

- **Compliance-risk.** *A:* the contract-level audit record (the integration body carrying the joint compliance verdict) is unenforced by the gate while d4f2 is driven to `addressed` with no provenance chain and the graph-data tree is exempted from coverage with no durable authorization record — three audit-trail gaps in a class-3 lifecycle. *B:* strengthens the audit record by gating section presence (a real improvement on this axis), but inherits the d4f2-provenance and graph-data-exemption gaps and adds a machine-green-but-substance-blind compliance section plus a parser whose false-greens would certify a missing audit record as present.

- **QA-test.** *A:* the COMBINED-test-run claim has no oracle (agent-trust only — A's explicit design choice), and the test plan omits the `lane`-enum validity case, two coverage boundary rows (draft-integration status, brief-set-bounded coverage), a dispatch-pinning assertion, and a self-dogfood red-path. *B:* closes A's body-oracle gap but adds a markdown parser whose hardest failure (heading drift across config, agent, and CLAUDE.md producing false-reds) is named but untested; still leaves the lane enum and the same two coverage boundaries unverified, and guarantees section presence not substance.

- **Reliability-ops.** *A:* the combined-test-run signal is machine-unverifiable, and the shared `/integrate` / `/prepare-evidence` multi-write flows have no specified crash/retry/idempotency recovery — a botched retry can silently fork the coverage artifact while the graph stays green. *B:* banks A's missing combined-run signal as a machine check but introduces the codebase's first prose-parsing rule, whose brittle heading-matching can take the whole non-short-circuiting gate falsely red (blocking correct merges) or falsely green (manufacturing confidence), while inheriting all of A's unspecified crash-recovery gaps.

- **Cost-maintainability.** *A:* the body-completeness guarantee is an untested, three-way hand-synced prose contract (CLAUDE.md convention, agent prompt, intent) that can silently diverge with no CI signal, and the change overloads the tooling capability with docs/tests it does not describe. *B:* buys a presence-only (form-not-substance) guarantee at the price of the codebase's first markdown-body parser, exact-heading brittleness that produces false-red merge blocks on routine prose edits, and a deliberate break of the "bodies are convention" precedent that every future body-shape decision must navigate.

- **Release.** *A:* **blocking** — the coverage rule lands ungated, with no `_from` cutoff or staged-rollout lever (unlike the `comparison_required` rule it models on), so on merge it retroactively judges all six pre-existing selected intents and can turn main red with no rollback short of reverting the migration; fix is a dated `coverage_coherence_from` cutoff or a committed pre-merge audit. *B:* **blocking** — carries A's ungated-coverage hazard in full **and** adds a second irreversible gate enforced by a net-new body parser whose first subject is this PR's own integration node, coupling the migration's merge to parser teething with no rollback but full revert; the body rule should be staged or flagged so it cannot self-block the coverage core's merge.

## The case against each candidate

**Strongest reason a reviewer could reject A.** A makes a *weaker* promise than the intent's headline value while occupying the same visual position. The intent introduces the integration node specifically because the combined test run "catches defects present in no single lane" — yet A's gate proves only that a `final` integration node *exists and is wired to the right evidence*, never that its body contains the combined-test-run or joint-compliance-verdict section at all. By A's own concession, a `final` integration with a one-character or section-omitting body passes `spec:validate`, the contract is marked covered, the intent flips `addressed`, and a release later `includes` this node as its coverage artifact — all green. For a class-3, production-sensitive lifecycle, the single most consequential failure ("was the integrated whole actually verified?") has no machine signal and is invisible to every downstream human who trusts the green check; the security framing sharpens this from a quality gap into a gate-integrity gap, because that same hollow node carries the `touches` edges check-diff trusts to govern `specs/schema/**`. A reviewer who holds that the intent requires the body's *content* to be gated — not merely a node to be present and wired — has grounds to reject A.

**Strongest reason a reviewer could reject B.** B pays a permanent, novel cost — the codebase's first and only markdown-body-prose parser running inside the merge-gating validator — for a guarantee B itself concedes is presence-only, not substance. That parser is a new bidirectional failure surface on the gate: a *false red* (a section genuinely present but written with a slightly different heading, trailing whitespace, a different heading level, or inside a fenced code block) takes the entire non-short-circuiting `spec:validate` red and blocks an otherwise-correct multi-lane merge on a formatting nit, with the operator's only recovery being to reverse-engineer the parser's match semantics; a *false green* (a heading spoofed inside a code fence, or a hollow one-sentence section) manufactures confidence a responder may over-trust. The required-heading vocabulary must stay byte-aligned across three hand-synced files (the `validation-rules.yaml` config, the `integration-reviewer` agent prompt, and CLAUDE.md) with no test binding them, so routine prose edits or agent drift red the whole graph. This risk binds at the worst moment: B's defining gate fires first on *this PR's own* hand-authored integration node, coupling the merge of the high-value coverage core to brand-new parser teething with no rollback but reverting everything — and it buys only section *presence*, still requiring the same human read A needs to confirm the section is real. A reviewer who judges that a presence-only check does not justify a permanent markdown-parser maintenance surface, an asymmetric break of the "bodies are convention" precedent, and a second ungated, self-blocking gate on the merge path has grounds to reject B.
