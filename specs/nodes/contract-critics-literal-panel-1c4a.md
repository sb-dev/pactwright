## Problem interpretation

This intent (`intent-specialist-critics-comparison-7ada`) is a Class 3, multi-surface change: it edits the schema (`node-types.yaml`, `edge-types.yaml`, `validation-rules.yaml`), the loader and validator (`tools/`), `CLAUDE.md`, two lifecycle commands (`/review-contracts`, `/approve-contract`), and adds new agent files under `.claude/agents/`. That class is why this market carries ≥2 candidates; this is candidate **1 of 3**.

The intent asks for three things that compose into one capability — *make contract review produce a durable, multi-perspective record of why the losers lost, and forbid a class-≥2 selection from standing without one.* (a) A `comparison` node type + `compares` edge + a `comparison_required_from` cutoff, enforced by a new `spec:validate` rule. (b) Nine specialist critic agents that attack candidate contracts each from a single perspective. (c) A `/review-contracts` rewrite that routes and invokes critics by work-class, and an `/approve-contract` update whose decision cites the comparison.

The schema/validator half is a near-mechanical extension of patterns already in the tree: the cutoff is read exactly like `sensitivePaths` (`tools/loader.ts`), and the new rule is a `class_market_quorum` clone that adds a date guard and a "comparison exists and covers the candidates" predicate. The critic half is where this market's three candidates genuinely diverge. **This candidate takes the intent's enumeration literally:** nine bespoke standalone agent files, one per perspective, with routing written as `CLAUDE.md` prose plus explicit per-class invocation logic in the command. It contrasts with `contract-critics-registry-driven` (one parameterized `specialist-critic` agent + a declarative critic-registry — which deviates from the literal nine-named-files enumeration) and `contract-critics-named-thin-rubric` (nine named but THIN agent files delegating rules and routing to a shared rubric/registry).

## Scope

### Shared core (identical across all three candidates)

1. **Schema migration (one atomic commit).** In `node-types.yaml` add `comparison` with `required_fields: [id, type, title, created]`, `requires_body: true`, **no** `status_values`, **no** `class`. This is safe by existing precedent: `nodes-status-in-enum` skips any type lacking `status_values` (exactly as it does for `decision`/`override`), and `nodes-class-in-range` is scoped to `types: [intent, contract]` only (`validation-rules.yaml` line 39), so a class-less, status-less node type validates cleanly. In `edge-types.yaml` add `compares` with `source: comparison`, `target: contract`. In `validation-rules.yaml` add a top-level `comparison_required_from: "2026-06-18"` — a date string (the loader parses with CORE_SCHEMA, so it stays a string; ISO `YYYY-MM-DD` sorts lexically == chronologically). The cutoff is the day **after** this market's bootstrap contracts (`created: 2026-06-17`), so a lexical `>=` strictly grandfathers every existing contract and dodges same-day ambiguity.
2. **Loader change.** Extend `tools/loader.ts` to surface `comparison_required_from` on `LoadedSpec`, parallel to the existing `sensitivePaths` read (loader.ts:128 reads `sensitive_paths` as optional data; add a sibling string read after it, and a `comparison_required_from?: string` field on the `LoadedSpec` interface beside `sensitivePaths`). Absent ⇒ the rule is disabled.
3. **New validate rule `comparison-required`** (kind `comparison_required`), placed as the **last data rule** — after `class-market-quorum`, before `indexes-fresh`. It traverses resolved `selects`/`proposes`/`compares`, so it must follow `edges-references-resolve` and `edges-endpoint-types`. New handler `tools/handlers/comparison_required.ts`, registered in `tools/validator.ts` `HANDLERS`, signature `(rule, spec) => Finding[]`, **reusing `toDateString` from `tools/gate.ts`** for the date compare and modeled on the `class_market_quorum` traversal (`nodesById`, iterate `selects`, resolve target contract, find the proposing intent via `proposes`, read `class` as a `number`). Semantics: for each `selects` → contract C, skip if `C.class < 2`, or `toDateString(C.created) < comparison_required_from`, or the date/cutoff is unparseable (grandfather / fail-open). Otherwise valid only if there exists a `comparison` node K whose resolved `compares` targets **cover** the live (non-`superseded`) candidate set proposing C's intent — the *identical* live-candidate predicate `class_market_quorum` uses (count `proposes` edges targeting the intent whose source contract is not `superseded`) — **and** K has ≥2 resolved `compares` edges. No backfill; no graph writes to existing nodes.
4. **`/review-contracts` rewrite.** Route critics by intent class + each candidate's declared `## Scope`: class 0–1 → spec-critic only; class 2 → spec-critic + the specialists whose surface the candidates' scope touches; class 3 → spec-critic + the full panel. Invoke each critic as a **real subagent** (replacing today's inline "Act as spec-critic"). Each appends a perspective-labelled `## Critique` to every candidate. Then graph-maintainer creates **one** `comparison` node (body sections: trade-off table / critic findings by perspective / the case against each candidate) + one `compares` edge per candidate; regenerate indexes; validate; end by asking for a human decision; never select or rank.
5. **`/approve-contract` update.** The decision body cites the comparison node id and records the accepted trade-off + why each rejected candidate lost. Keep the existing class-market-quorum pre-check; still writes no brief.
6. **CLAUDE.md.** Add the critic-routing-by-class table, the comparison mechanism, and the grandfathering rule (the cutoff; pre-cutoff / class-≤1 skip; no backfill).
7. The `comparison` body **structure** is a command / graph-maintainer convention, **not** a validate rule — there is no body-section-validation precedent among the existing data rules, and inventing one is out of scope.

### What distinguishes this candidate

- **Nine bespoke standalone critic agent files** under `.claude/agents/`, exactly as the intent enumerates: `product-critic.md`, `ux-critic.md`, `architecture-critic.md`, `security-privacy-critic.md`, `compliance-risk-critic.md`, `qa-test-critic.md`, `reliability-ops-critic.md`, `cost-maintainability-critic.md`, `release-critic.md`. Each carries frontmatter `name`, `description`, `tools: Read, Grep, Glob`; a body of numbered review rules **specialized to that single perspective** — the specific missing-cases, risks, and strongest-argument-against it hunts on its one axis — ending by drafting a perspective-labelled `## Critique` for graph-maintainer to append, never selecting or ranking, all writes via graph-maintainer. They mirror `spec-critic.md`'s shape (compact, ~25–40 lines each).
- **Routing as prose + per-class invocation lists.** The Class 0–3 routing table lives as **prose in CLAUDE.md**, and `/review-contracts` carries explicit per-class invocation logic. For class 2 it matches each candidate's `## Scope` text against a surface→critic mapping: UI → ux-critic; payments / personal-data → security-privacy-critic + compliance-risk-critic; schema / service-boundary → architecture-critic; testing → qa-test-critic; runtime / ops → reliability-ops-critic; cost / maintainability → cost-maintainability-critic; release / rollout → release-critic; product / value → product-critic. For class 3 it invokes the full panel regardless of surface.

## Non-scope

- **No comparison-body schema rule.** The trade-off-table / by-perspective / case-against structure is a command convention; it is never machine-checked (no body-section-validation precedent — Shared core item 7).
- **No backfill and no graph writes to existing nodes.** The cutoff grandfathers every pre-cutoff selection; no comparison node is retro-fitted to a historical market.
- **No parameterized single critic and no critic-registry data file.** A registry-driven agent is `contract-critics-registry-driven`'s mechanism, deliberately not adopted here — routing is prose + command logic, not data.
- **No thin-agent / shared-rubric indirection.** Delegating the nine perspectives' rules and routing to a shared rubric is `contract-critics-named-thin-rubric`'s mechanism, deliberately not adopted; each agent owns its full prompt.
- **No new CI workflow.** `spec-validate.yml` already runs `spec:validate`; `comparison-required` rides it.
- **No automated critic selection or ranking, ever.** Selection stays a human decision recorded by a `decision` node citing the comparison.

## Trade-offs

- **+** Maximal fidelity to the intent's literal enumeration: nine bespoke standalone files named exactly as written, nothing inferred or collapsed.
- **+** Deepest per-axis specialization — nine independently tunable prompts, each free to hunt the missing-cases / risks / strongest-argument that matter on its one surface, with no shared-prompt lowest-common-denominator and no registry indirection to read through.
- **+** No new abstraction to build or validate: the agents are flat markdown like `spec-critic.md`, and routing is plain prose + command logic — zero new data surface beyond the schema migration the whole market shares.
- **−** Nine prompt files to maintain, with real **drift risk across them** — a convention change (how to label a `## Critique`, how to defer writes to graph-maintainer) must be hand-applied nine times and can rot unevenly.
- **−** Routing knowledge is **duplicated in two places** — the CLAUDE.md prose table *and* the command's per-class invocation logic — and the two can diverge silently; nothing validates that they agree.
- **−** The class-2 surface→critic match is **heuristic free-text matching** on each candidate's `## Scope` prose, encoded by hand in the command. A scope that says "checkout" but never "payments" can route past the security/compliance critics; the mapping is only as good as its keyword list.

## Risks

- **Nine-file drift.** Convention changes applied unevenly across nine bespoke prompts leave critics behaving inconsistently (one still selecting/ranking, one writing directly instead of deferring to graph-maintainer). *Mitigation:* pin a shared skeleton in the brief — every critic file opens with the same "draft only / never select / all writes via graph-maintainer / end-of-step reminder" boilerplate copied from `spec-critic.md`, differing only in the numbered per-axis review rules; the brief lists the nine files so a reviewer can diff their skeletons against each other.
- **CLAUDE.md / command routing divergence.** The prose table and the command's invocation logic are two encodings of one mapping and can drift apart. *Mitigation:* the brief makes `/review-contracts` quote the surface→critic mapping inline and CLAUDE.md the normative source; an acceptance example exercises a representative class-2 and class-3 route so a divergence shows up as a wrong critic set, not a silent miss. (This duplication is the explicit axis on which `contract-critics-registry-driven` and `contract-critics-named-thin-rubric` differ — a recorded trade-off, not a defect to hide.)
- **Per-critic prompt-quality variance.** Nine hand-written prompts will not be uniformly sharp; a weak prompt yields a shallow `## Critique` that still gets recorded in the comparison, giving false confidence on that axis. *Mitigation:* each critic's numbered rules must name the concrete missing-cases / risks / strongest-argument it hunts (mirroring how `spec-critic.md` demands "name concrete failure scenarios, not generalities"); the comparison node's by-perspective section makes a thin critique visibly thin to the human selector.
- **Heuristic match misroutes a class-2 review.** Free-text scope matching can route a payments-touching class-2 candidate past the security/compliance critics. *Mitigation:* document the keyword list in CLAUDE.md; when scope is ambiguous, `/review-contracts` errs toward invoking more critics, never fewer; and any genuinely security/privacy/payments-touching change is class 3 by CLAUDE.md's own routing table, which invokes the full panel regardless of the heuristic.

## Acceptance examples

1. **Schema first, green.** After the migration commit, `pnpm spec:validate` is green; `comparison` appears in `node-types.yaml` (with `requires_body: true`, no `status_values`, no `class`), `compares` in `edge-types.yaml`, `comparison_required_from: "2026-06-18"` as a top-level string in `validation-rules.yaml`, and a `comparison` group exists in `specs/indexes/by-type.yaml`.
2. **Loader surfaces the cutoff.** `LoadedSpec.comparison_required_from === "2026-06-18"` on the real tree; deleting the key from `validation-rules.yaml` leaves it `undefined` and the `comparison-required` rule emits zero findings (disabled), proving fail-open on absence.
3. **Post-cutoff class-2 selection is blocked until a covering comparison exists.** Construct a class-2 intent with two live candidate contracts (`created` ≥ `2026-06-18`) and a `decision —selects→` one of them, with **no** comparison node. `pnpm spec:validate` fails on rule **`comparison-required`** with a detail naming the selected contract's intent and the missing comparison. Add one `comparison` node with two `compares` edges covering both live candidates → the rule flips green.
4. **Coverage, not mere count.** A comparison node with two `compares` edges that both target the *same* candidate (leaving a live sibling uncovered) does **not** satisfy the rule for a two-candidate market — `comparison-required` requires the `compares` targets to *cover* the live candidate set, identical to `class_market_quorum`'s live-candidate predicate, plus ≥2 `compares` edges.
5. **Every pre-cutoff selection still validates (grandfathered).** On the real tree the four existing `selects` edges and this market's own bootstrap selection (all resolving to contracts with `created` ≤ `2026-06-17` < the cutoff) emit zero `comparison-required` findings; the migration PR stays green with no comparison node authored for any historical market.
6. **Class-2 UI route invokes ux-critic, not the payments critics.** `/review-contracts` on a class-2 intent whose candidates' `## Scope` describes a UI surface invokes spec-critic + ux-critic and does **not** invoke security-privacy-critic or compliance-risk-critic; each invoked critic appends a perspective-labelled `## Critique` to every candidate.
7. **Class-3 invokes the full panel.** `/review-contracts` on a class-3 intent invokes spec-critic + all nine specialists regardless of the candidates' surface, producing one labelled `## Critique` per perspective per candidate.
8. **The comparison is the durable record.** After review, exactly one `comparison` node exists for the market with one `compares` edge per candidate; its body carries the trade-off table, critic findings by perspective, and the case against each candidate; `/approve-contract`'s `decision` cites that comparison node id and records why each loser lost. The comparison node is never superseded by selection and persists as the durable record.

## Critique

This critique was drafted by spec-critic against the real graph and tools/ on 2026-06-17; it reviews contract-critics-literal-panel-1c4a on its own terms and does not rank it against the other candidates.

**Shared-core gaps (inherited; this candidate carries them with no added defence).**

- *The `comparison_required` coverage predicate is named "identical to `class_market_quorum`'s live-candidate predicate" but the two predicates are not the same shape, and the candidate never resolves the mismatch.* `class_market_quorum` (handlers/class_market_quorum.ts:26-38) only *counts* live `proposes` edges into the intent (the `liveCandidates` closure returns a number). The new rule must instead build the *set of contract ids* that live-propose the intent and check that the `compares` targets *cover* it. "Cover" is left underspecified for the realistic cases: (a) a comparison whose `compares` targets a *superseded* candidate plus one live candidate — the candidate requires both "≥2 resolved `compares` edges" and coverage of the live (non-`superseded`) set, but never says whether a compares edge pointing at a superseded contract legitimately counts toward the ≥2; acceptance example 4 only exercises duplicate-target-on-a-live-candidate, so the superseded-target case is unspecified. (b) A market that legitimately grows from two to three live candidates *after* the comparison was authored: the original comparison now covers 2 of 3 live candidates, so by the stated coverage rule the previously-green selection silently goes red on the next unrelated PR. The contract promises "no backfill, no graph writes to existing nodes" yet its own rule can retroactively invalidate a standing selection when the candidate set changes — a class-3 graph that "cannot merge" on an edit nobody reviewed against this rule.

- *Cutoff date semantics: only `C.created` is normalized in the shown compare, leaving the cutoff string un-normalized despite the prose claim of fail-open on an unparseable cutoff.* The candidate's stated comparison is `toDateString(C.created) < comparison_required_from` (Shared core item 3), routing only `C.created` through `toDateString` (gate.ts:42, whose regex is `^\d{4}-\d{2}-\d{2}$`). The contract's prose adds "or the date/cutoff is unparseable ... fail-open," but the actual compare expression it shows never runs the *cutoff* itself through `toDateString` — only `C.created`. Since the cutoff is read raw from `validation-rules.yaml` "parallel to `sensitivePaths`" (loader.ts:128), and the loader's existing idiom passes such values only through `asString` element-level (never `toDateString`), a malformed cutoff like `"2026-06-18 "` (trailing space) or `2026-6-18` (un-padded) is a non-empty string that `asString` accepts, is not `undefined`, and is then compared *un-normalized* against a `toDateString`-normalized `C.created` — silently mis-grandfathering. This is a class-3 enforcement rule whose own prose intent (fail-open on an unparseable cutoff) is not realized by the compare expression it specifies, so a one-character schema typo could disable the gate with no finding emitted.

- *Loader claim is half-verified and the contract over-states the precedent.* `comparison_required_from` is *not* on `LoadedSpec` today — the candidate proposes adding it. The real `sensitivePaths` field (loader.ts:47 declares it `string[]`; loader.ts:128 reads it) is a *list* read via `asList` (mapping each element through `asString`), not a scalar string read; there is no existing scalar-string-from-`rulesDoc` precedent in the loader. So "parallel to the existing `sensitivePaths` read" overstates the fit: a new scalar read with its own absent/empty/non-string handling has to be written and tested, and the contract's acceptance example 2 (`=== "2026-06-18"`) presumes a read path that does not yet exist and whose absent-case (`undefined` vs `""`) it never pins.

- *Command/agent rewrites are entirely unverifiable as "scope".* Shared-core items 4 and 5 ("`/review-contracts` rewrite", "`/approve-contract` update") describe behaviour of `.claude/commands/` files that no validation rule touches and that the contract never quotes. There is no acceptance example that a machine can run for the routing or the decision-cites-comparison requirement — examples 6, 7, 8 are "recorded transcript" checks (Verification needs), i.e. manual. For a class-3 change the contract leans on human-audited transcripts for its single most behaviourally-load-bearing surface.

**Weaknesses specific to this candidate's axis (nine bespoke files + prose/command routing).**

- *The routing-duplication risk is acknowledged in Trade-offs but the contract resolves it with "the brief makes it agree" — i.e. it ships an invariant with no enforcement.* Nothing in the schema or the new rule validates that the CLAUDE.md prose table and the `/review-contracts` per-class invocation list name the same critic set. The candidate explicitly declines a registry (Non-scope: "no critic-registry data file"), so the two encodings are free text in two files. Concrete failure: a future PR adds a tenth critic to the CLAUDE.md table but not to the command (or vice versa); `spec:validate` is green, every acceptance example still passes, and a real class-3 review silently omits a critic. The contract's own Risk mitigation ("an acceptance example exercises a representative route") catches only the two routes the example happens to test, not the gap.

- *The class-2 surface→critic heuristic is free-text keyword matching with a named bypass that the contract treats as a feature.* Risk 4's mitigation is "any genuinely security/privacy/payments-touching change is class 3 by CLAUDE.md's own routing table." But CLAUDE.md's class-3 trigger is itself a *human classification at capture*; nothing forces a payments-touching change to be filed as class 3. So a candidate filed as class 2 with `## Scope` prose saying "checkout" but never "payments"/"personal data" routes past security-privacy-critic and compliance-risk-critic, and the comparison node then records *false confidence* — its by-perspective section simply has no security row, which reads as "no concern found" rather than "never looked." The contract names this scenario and then defers it to a classification step it does not control.

- *"Mirror `spec-critic.md`'s shape (~25-40 lines each)" plus nine files is a maintenance invariant with no diff gate.* The verification need "a manual diff of the nine skeletons confirms they have not drifted apart" is, again, manual. `spec-critic.md` (the real file, 26 lines) is a specific six-step numbered prompt ending in the index-then-validate reminder, and step 5 is the "never select or rank candidates" line; the contract requires nine copies of that boilerplate to stay aligned by hand. Concrete drift: one critic file omits the "never select or rank" line (or keeps an older inline phrasing), and that critic starts ranking candidates inside a comparison whose whole purpose is to *not* rank — a behaviour no rule can catch because agent-body content is never validated.

- *Scope creep in the routing mapping itself.* The intent's CLAUDE.md ask lists only a few example mappings ("UI scope -> ux-critic; payments or personal data -> security-privacy-critic and compliance-risk-critic; schema or service-boundary -> architecture-critic; and so on"). This candidate hard-codes a full nine-way keyword table (distinguishing item 2: "release / rollout -> release-critic; cost / maintainability -> cost-maintainability-critic", etc.) into the command. Each of those keyword choices is an unstated product decision the intent did not make; baking them into command logic (rather than leaving them as reviewer judgement, which the intent's "and so on" arguably invites) commits the graph to a specific taxonomy that will need re-litigation the first time a candidate's scope doesn't use the blessed nouns.

**Net.** The candidate's schema/validator half is plausible and its concrete code references (`toDateString` at gate.ts:42, the `class_market_quorum` count traversal, the rule-ordering slot after `class-market-quorum` and before `indexes-fresh`) are real — but its central coverage predicate is under-specified for the superseded-target and growing-candidate-set cases and can retroactively red a standing class-3 selection, and its shown cutoff compare normalizes only `C.created`, leaving the cutoff string un-normalized so a schema typo can silently disable a class-3 gate against the contract's own fail-open intent. On its distinguishing axis it concentrates two unenforced invariants — prose/command routing agreement and nine-file skeleton parity — into hand-maintained free text whose only guard is "the brief will say so" and "manual diff," with a self-acknowledged heuristic that can route a mislabelled payments change past the security and compliance critics. Every behaviourally load-bearing surface (routing, decision-cites-comparison, agent uniformity) rests on manual transcript audits rather than machine checks, which is a thin spine for a change the contract itself classifies class 3.
