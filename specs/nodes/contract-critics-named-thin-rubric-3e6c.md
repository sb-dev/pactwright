## Problem interpretation

The intent (`intent-specialist-critics-comparison-7ada`, class 3) replaces the single inline "Act as spec-critic" review with a routed panel of nine perspective-specialist critics, makes their collective output a durable `comparison` node, and gates a post-cutoff class-≥2 selection on that node existing. The mechanism splits cleanly into a deterministic spine — a new node type, a new edge, a dated cutoff, a loader field, a validate rule, two command rewrites — and one genuinely contested design axis: **how the nine critic perspectives are encoded and routed.** The three candidates in this market share the spine byte-for-byte and differ only on that axis.

This candidate is the compromise between its two siblings. `contract-critics-literal-panel` writes nine bespoke standalone agent files, each carrying its own rules and surface-trigger as prose, with routing as CLAUDE.md per-class invocation lists — maximally literal to "new agents in `.claude/agents/`," but nine independent copies of review discipline that drift apart. `contract-critics-registry-driven` collapses all nine into one parameterized `specialist-critic` agent driven by a declarative registry — one source of truth, but it deviates from the intent's literal nine-named-files enumeration and loses name-invocable critics. **This candidate keeps the nine NAMED, name-invocable agent files (literal-compliant) AND a single rubric source of truth: each named file is THIN — a perspective label plus a pointer — and delegates its numbered rules and surface-trigger to one shared rubric/registry doc that every critic and the router read.**

## Scope

### Shared core (identical across all three candidates)

1. **Schema migration, one atomic commit.** In `node-types.yaml` add `comparison` with `required_fields: [id, type, title, created]`, `requires_body: true`, and **no** `status_values` and **no** class — `comparison` is a non-status, non-classed record like `decision`/`override`. This is safe by construction: `nodes-status-in-enum` skips any type lacking `status_values`, and `nodes-class-in-range` scopes only to `[intent, contract]`, so neither rule touches the new type. In `edge-types.yaml` add `compares: {source: comparison, target: contract}`. In `validation-rules.yaml` add a top-level `comparison_required_from: "2026-06-18"` — a date *string* (CORE_SCHEMA keeps an unquoted `YYYY-MM-DD` a string anyway, and ISO dates sort lexically == chronologically). The cutoff is the day **after** this market's bootstrap contracts (`created: 2026-06-17`), so a lexical `>=` strictly grandfathers every contract that exists at migration time, these three included.
2. **Loader change.** Extend `tools/loader.ts` to surface `comparison_required_from` on `LoadedSpec`, exactly parallel to the existing `sensitivePaths` field (read from `validation-rules.yaml`, absent ⇒ `undefined` ⇒ rule disabled).
3. **New validate rule `comparison-required`** (`kind: comparison_required`), placed as the **last data rule** — after `class-market-quorum`, before `indexes-fresh`, following `edges-references-resolve` and `edges-endpoint-types` so it may assume endpoints resolve and are correctly typed. New handler `tools/handlers/comparison_required.ts`, registered in the `HANDLERS` map in `tools/validator.ts`, signature `(rule, spec) => Finding[]`. It **reuses `toDateString` from `tools/gate.ts`** and is **modeled on `class_market_quorum`**: build `nodesById`, iterate `selects` edges, resolve the target contract, find the proposing intent via `proposes`, read `class` as a `number`. Semantics — for each `selects` → contract C: **skip** if `C.class < 2`, or `toDateString(C.created) < comparison_required_from`, or either date is unparseable (grandfather / fail-open). Otherwise the selection is valid only if there exists a `comparison` node K whose resolved `compares` targets **cover the live (non-`superseded`) candidate set** proposing C's intent — the *identical* live-candidate predicate `class_market_quorum` already uses — **and** K has **≥2** resolved `compares` edges. No backfill; no writes to any existing node.
4. **`/review-contracts` rewrite.** Route critics by the intent's `class` plus the candidates' declared `## Scope`: class 0–1 → spec-critic only; class 2 → spec-critic + the specialists whose surface the candidates touch; class 3 → spec-critic + the full nine-critic panel. Invoke each as a **real subagent** (replacing today's inline "Act as spec-critic"). Each critic appends a perspective-labelled `## Critique` to every candidate. Then graph-maintainer creates exactly **one** `comparison` node (sections: trade-off table / findings by perspective / case against each candidate) and **one** `compares` edge per candidate; regenerates indexes; validates; asks for a human decision; never selects.
5. **`/approve-contract` update.** The decision body cites the `comparison` node id and records the accepted trade-off plus why each rejected candidate lost. Keep the existing `class-market-quorum` pre-check. Still authors no brief.
6. **CLAUDE.md.** Add the critic-routing-by-class table, the comparison mechanism, and the grandfathering rule.
7. The `comparison` body **structure** is a command/graph-maintainer convention, not a validate rule — the rule checks only the node's existence, edge count, and candidate coverage.

### What distinguishes this candidate

- **Nine NAMED but THIN agent files** in `.claude/agents/`: `product-critic.md`, `ux-critic.md`, `architecture-critic.md`, `security-privacy-critic.md`, `compliance-risk-critic.md`, `qa-test-critic.md`, `reliability-ops-critic.md`, `cost-maintainability-critic.md`, `release-critic.md`. The nine names exist and are invocable-by-name, satisfying the intent's literal "new agents in `.claude/agents/`." Each file is THIN: frontmatter `name` / `description` / `tools: Read, Grep, Glob`, and a SHORT body that (a) declares its single perspective and (b) **delegates** its numbered review rules and its surface-trigger to a single shared rubric doc. The critic reads its own pinned section of the shared doc at review time. It never selects; it writes nothing directly — all graph writes go through graph-maintainer.
- **A single shared rubric/registry** — `.claude/critic-rubric.md` — holds, once, every perspective's numbered review rules and its surface-trigger (the `## Scope` signals that route it in at class 2). The thin agent files carry no rules of their own, only a perspective label and a pinned pointer (a stable section anchor) into this doc. **Routing reads the shared doc's surface-triggers as the single routing source** — CLAUDE.md's routing table and `/review-contracts` both reference the rubric for *which* critics a class-2 change earns — while *invocation* is by the nine agent names. One rule list, one trigger list, nine name-invocable front doors.

## Non-scope

- No node/edge/rule additions beyond the shared core (`comparison`, `compares`, `comparison-required`).
- No backfill: existing pre-cutoff selections are grandfathered by the date check, never retro-fitted with comparison nodes.
- No validation of the `comparison` body's section structure — that is a command convention; the rule checks existence, edge count, and coverage only.
- No status lifecycle on `comparison` (it has no `status_values`); it is a durable record, never superseded in place — a corrected comparison is a new node, as with `decision`.
- **Rejected sibling (a): nine standalone bespoke agents** (`contract-critics-literal-panel`) — rejected here because nine independent copies of review discipline drift apart and there is no single place to amend a rule or a trigger.
- **Rejected sibling (b): one parameterized `specialist-critic` + registry** (`contract-critics-registry-driven`) — rejected here because it abandons the intent's literal nine-named-files enumeration and the name-invocable critics; this candidate keeps both.
- No automatic enforcement that a thin agent's perspective label matches its rubric section — coherence between the two layers is a convention, watched by spec-critic, not a validate rule (see Risks).

## Trade-offs

- **+** Keeps the nine NAMED, name-invocable critic agents — literal fidelity to "new agents in `.claude/agents/`," which `contract-critics-registry-driven` forfeits — *and* a single rubric source of truth, so review rules and surface-triggers cannot drift nine ways, which `contract-critics-literal-panel` cannot promise.
- **+** Amending a perspective's rules or its routing trigger is a one-place edit in `.claude/critic-rubric.md`; the nine front doors are untouched. Routing and review read the same triggers, so "which critics does this class-2 change earn" has exactly one answer.
- **+** Genuine middle ground: a reviewer who wants to invoke `ux-critic` by name still can, while the maintainer who wants one rubric to audit still has it.
- **−** The **most moving parts** of the three: nine files *plus* a shared doc *plus* the indirection between them. Both siblings have fewer artifacts to keep coherent (one has nine self-contained files; the other has one agent + one registry).
- **−** A thin agent can drift from the rubric — a name file's perspective label saying one thing while the rubric section it points at says another — a coherence failure neither sibling has in the same form.
- **−** Two layers (name files + rubric) must be kept coherent, and a name file pointing at a stale or renamed rubric section is a **new failure mode** absent from both siblings: a critic that silently reviews against the wrong section, or no section at all.

## Risks

- **Thin-agent / rubric divergence.** A name file's declared perspective and its pointed-at rubric section fall out of sync. *Mitigation:* the thin files carry **only** a perspective label and a pointer — zero duplicated rules — so there is nothing in the agent file to contradict the rubric except the pointer itself; the pointer is the single coupling to police.
- **Rubric section-naming coupling.** Renaming or reordering a rubric section silently breaks the agents that point at it. *Mitigation:* pin the section anchors the agents reference (stable slugs, listed once at the top of the rubric), and have spec-critic check anchor integrity during a class-3 review of any change to the rubric or the agents.
- **Silent no-op critic.** If a critic's rubric section is missing or its anchor unresolved, the critic could run and produce an empty or generic `## Critique` — a perspective silently dropping out of a class-3 panel. *Mitigation:* the rubric lists all nine anchors in one block; a missing anchor is visible there, and `/review-contracts` enumerates the expected nine critiques for a class-3 intent so an absent perspective is caught by count, not by reading nine appended sections.
- **Grandfather correctness (shared).** The cutoff must lexically grandfather every existing contract. *Mitigation:* `comparison_required_from = 2026-06-18` is strictly after the latest existing `created` (`2026-06-17`); `toDateString` rejects malformed dates and the rule fails-open (skips) on an unparseable date or cutoff, exactly as the spine specifies.

## Acceptance examples

1. **Schema first, green.** After the atomic migration commit, `pnpm spec:validate` is green; `comparison` appears in `node-types.yaml` (no `status_values`, no class), `compares` in `edge-types.yaml`, `comparison_required_from: "2026-06-18"` in `validation-rules.yaml`, and a `comparison` group exists in `specs/indexes/by-type.yaml`.
2. **Nine named files + one rubric.** Nine critic agent files exist under `.claude/agents/` (`product-critic.md` … `release-critic.md`), each invocable by name and THIN (perspective label + pointer, `tools: Read, Grep, Glob`, no embedded rules); a single `.claude/critic-rubric.md` holds all nine perspectives' numbered rules and surface-triggers, with pinned section anchors listed once.
3. **Post-cutoff class-2 selection blocked until a comparison exists.** A new class-2 intent created after the cutoff, with two live candidate contracts and a `decision —selects→` one of them but no `comparison` node, fails `pnpm spec:validate` on rule **`comparison-required`** with a detail naming the intent and the missing/under-covered comparison. Adding one `comparison` node with ≥2 `compares` edges covering both live candidates flips the rule green.
4. **Every pre-cutoff selection still validates.** On the real tree post-migration, all existing `selects` edges (every existing contract has `created` ≤ `2026-06-17` < `2026-06-18`) are skipped by the date check; `comparison-required` emits zero findings for them. No backfill is performed.
5. **Routing reads the shared doc — UI change.** `/review-contracts` on a UI-touching class-2 intent invokes `ux-critic` (its surface-trigger in the rubric matches the candidates' `## Scope`) and does **not** invoke the payments-adjacent critics (`compliance-risk-critic`, no matching trigger). The set of critics is derived from the rubric's surface-triggers, not from prose duplicated in nine files.
6. **Full panel — class 3.** `/review-contracts` on a class-3 intent invokes spec-critic plus all nine specialists; each appends a perspective-labelled `## Critique` to every candidate; the expected count of critiques is enumerated so a silently-dropped perspective is caught.
7. **Durable record.** After review, exactly one `comparison` node and one `compares` edge per candidate exist; the node persists as the standing record cited by the eventual `decision` body, and `/approve-contract` writes a decision citing that comparison id with the accepted trade-off and per-rejected-candidate reasoning.
8. **Coverage, not mere count.** A `comparison` node with two `compares` edges that both target the *same* candidate (or a superseded one) does **not** satisfy `comparison-required` for an intent with two distinct live candidates — coverage of the live candidate set is required, identical to the `class_market_quorum` live predicate.

## Critique

This critique was drafted by spec-critic against the real graph and tools/ on 2026-06-17; it reviews contract-critics-named-thin-rubric-3e6c on its own terms and does not rank it against the other candidates.

### Ambiguity

- **The cutoff is never normalized through `toDateString`, and the contract's "fails-open on an unparseable cutoff" promise has no stated detection mechanism.** `toDateString` (gate.ts:42) normalizes a *node field* to `YYYY-MM-DD` and returns `undefined` for anything that fails the shape/calendar round-trip. The candidate's comparison rule (Scope §3) computes `toDateString(C.created) < comparison_required_from` — the left operand is normalized, the right operand is the raw schema string, never itself passed through `toDateString`. The candidate *does* claim the rule "fails-open (skips) on an unparseable date or cutoff" (Risk "Grandfather correctness"; Verification §(e)), so this is not undefined behaviour as a stated intent — but the contract never says *how* a malformed cutoff is detected, and the only normalizer it cites runs on `C.created`, not on the cutoff. If a maintainer ever writes `comparison_required_from: "2026-6-18"` (single-digit month) or with a trailing time component, the left side is normalized and the right is not; the lexical `<` then compares unlike shapes rather than skipping, contradicting the fail-open promise. The contract asserts "ISO dates sort lexically == chronologically" but never binds the *cutoff* to a `toDateString` round-trip before comparison.
- **"Cover the live candidate set" is asserted as "the *identical* live-candidate predicate `class_market_quorum` already uses" — but that predicate computes a *count*, not a *set*.** `liveCandidates` in class_market_quorum.ts (lines 26-38) counts non-superseded `proposes` sources and returns a `number`; it never collects their ids. The candidate's coverage check (Acceptance §8: two `compares` edges to the *same* candidate must fail) needs set membership — resolve each `compares` target, dedupe, and prove the live candidate ids are a subset. That is a genuinely new traversal the contract describes as already-existing reuse. The word "identical" papers over the part that actually has to be written, and Verification §(g) ("superseded candidate excluded") only deepens the new logic without budgeting it.
- **"Pinned section anchor" is undefined as a mechanism.** Markdown has no enforced anchor syntax; section slugs are derived from heading text by whatever renderer. The contract says agents "read its own pinned section of the shared doc at review time" and that anchors are "stable slugs, listed once at the top of the rubric" (Scope §-distinguishing; Risk "Rubric section-naming coupling") — but never says how an agent *resolves* a slug to a section at runtime, nor what a critic does when the slug resolves to nothing. The entire distinguishing axis rests on a pointer indirection whose resolution semantics are left to convention.

### Missing cases

- **The `comparison-required` rule's count is ambiguous between raw and resolved `compares` edges.** Scope §3 requires "≥2 **resolved** `compares` edges" AND coverage, but Scope §7, Non-scope, and Acceptance §3 describe the rule as checking "edge count" / "≥2 `compares` edges" without the "resolved" qualifier. If both `compares` edges point at ids that do not resolve (deleted/renamed contracts), `edges-references-resolve` reports them but, per the modeled handler's own comment (class_market_quorum.ts:33, "unresolved source: skip, references_resolve owns it"), the rule order does not remove them. Whether the "2" includes unresolved edges is left unreconciled across the contract's own sections.
- **No handling of multiple `comparison` nodes for one intent.** The contract permits a "corrected comparison" to be a new node (Non-scope: "a corrected comparison is a new node, as with `decision`"). Nothing says the rule passes if *any* one comparison covers the set, versus *all* of them must. Two comparisons — an early one covering one candidate, a later one covering both — leave the rule's quantifier unstated. A stale comparison node is never superseded (the type has no `status_values`), so it lingers; the rule must say it tolerates the extra node.
- **Loader default mismatch undermines the "parallel to `sensitivePaths`" claim.** `sensitivePaths` defaults to `[]` (loader.ts:128, via `asList(...).map(...).filter(...)`), never `undefined`. The candidate says `comparison_required_from` absent ⇒ `undefined` ⇒ rule disabled (Scope §2). That is a *different* default contract than the field it claims exact parallelism with. The handler's fail-open path therefore depends on a loader behaviour (`undefined` on absence) that the cited precedent does not exhibit — an untested seam, only partly covered by Verification's "(absent)" loader test.
- **No machine validation that the nine thin agents and the rubric stay in sync — by the contract's own admission — yet the failure is silent and catastrophic.** Risks/Non-scope concede "no automatic enforcement that a thin agent's perspective label matches its rubric section." The mitigation is "spec-critic checks anchor integrity during a class-3 review" — but spec-critic is the agent being reshaped by this very change, and is not invoked on edits to `.claude/` files outside a contract review. A renamed rubric section (Risk "Rubric section-naming coupling") produces a critic that "silently reviews against the wrong section, or no section at all" (the contract's own Trade-off −3 and Risk "Silent no-op critic"). The only guard offered is `/review-contracts` "enumerates the expected nine critiques" by *count* — but a critic that resolved to the wrong section still emits a (wrong) critique, so the count is satisfied while the perspective is corrupted. The count guard catches absence, never substitution. This is a class-3 panel (including security-privacy and compliance-risk) silently reviewing under the wrong rubric with a green graph.

### Scope creep

- **The shared rubric doc (`.claude/critic-rubric.md`) is an artifact the intent never asked for.** The intent (lines 22-28) specifies "New agents in `.claude/agents/` … body of numbered review rules ending by writing findings." It places the rules *in the agent bodies*. This candidate *removes* the rules from the bodies and invents a second file plus a pointer indirection between the two layers. That indirection — and the "anchor integrity" guard, the "all nine anchors listed in one block" convention, the silent-no-op mitigation — are all machinery that exists only to manage a structure the intent did not request. The contract's own Trade-off (−1) admits it is "the most moving parts of the three." Every one of those moving parts is scope the intent is silent on.
- **Routing source-of-truth relocation is unrequested.** The intent says routing reads "the candidates' scope" (lines 30-37) and CLAUDE.md carries the routing table. This candidate makes "the shared doc's surface-triggers the single routing source" (Scope §-distinguishing) — relocating the routing authority from CLAUDE.md/contract-scope into the new rubric file. That is a design decision the intent did not authorize and that couples routing correctness to the same anchor-resolution fragility flagged above.

### Shared-core gaps inherited

- **The atomic-migration "green by construction" argument is asserted, not demonstrated against the rule order.** Scope §1/§3 place `comparison-required` "after `class-market-quorum`, before `indexes-fresh`." Verified: `class-market-quorum` then `indexes-fresh` are the last two rules (validation-rules.yaml:58-61), so the insertion point is real. But `indexes_fresh` regenerates the indexes in-memory and byte-compares against the committed files (indexes_fresh.ts); adding a `comparison` node type means `by-type.yaml` gains a group (Acceptance §1). If the migration commit adds the schema and rule but the index regeneration is not part of the *same* atomic commit, `indexes-fresh` fails. The contract says "one atomic commit" but never names index regeneration as part of that atom — the CI step "ends with `pnpm spec:index && pnpm spec:validate`" is mentioned only under Verification, not bound into the migration acceptance.
- **The handler registration claim is correct but the contract under-specifies the malformed-rule path.** `HANDLERS` is the right map (validator.ts:24-34) and an unknown `kind` produces a finding (validator.ts:68). So registering `comparison_required` is necessary or the graph self-reports — verified true. But the contract never states what `comparison_required_from` being a *non-string* (e.g. a YAML date object, which CORE_SCHEMA may or may not keep as a string) does to the handler. The contract leans on "CORE_SCHEMA keeps an unquoted `YYYY-MM-DD` a string anyway" — yet then recommends quoting it (`"2026-06-18"`), leaving the unquoted-vs-quoted decision and its parse consequence unsettled.

### Net

The shared spine is plausible and the rule-ordering / handler-registration claims check out against the real files, but two load-bearing assertions are overstated: the coverage predicate is described as reuse of `class_market_quorum` when it is in fact new set-based logic that handler never wrote (it returns a count, not a set), and the cutoff comparison runs `toDateString` on only one of its two operands, so the contract's "fails-open on an unparseable cutoff" promise has no stated detection mechanism — a non-normalized cutoff compares unlike shapes rather than skipping. The candidate's distinguishing axis — nine thin files plus a shared rubric plus a pointer indirection — introduces a substitution failure mode (a critic silently reviewing against the wrong or absent rubric section) that the contract itself names yet guards only by a critique *count*, which cannot detect substitution; and the rubric-as-routing-source and the rubric file itself are structure the intent never requested. The grandfathering math (`2026-06-18` strictly after the `2026-06-17` bootstrap contracts) is sound and the pre-cutoff `selects` edges in the live graph would all be skipped, so the green-graph claim holds; the risk is concentrated in the unbudgeted coverage traversal and the two-layer coherence that has no machine guard.
