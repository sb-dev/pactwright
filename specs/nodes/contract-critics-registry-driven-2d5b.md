## Problem interpretation

The intent (`intent-specialist-critics-comparison-7ada`, class 3) wants three things working together: (a) a durable `comparison` node type plus a `compares` edge and a dated `comparison_required_from` cutoff, with a validate rule that blocks a post-cutoff class-≥2 `selects` until a covering comparison node exists; (b) per-axis *adversarial* critique of every candidate contract — each perspective hunting that axis's missing cases, risks, and strongest argument against each candidate; and (c) class-and-scope routing that decides which perspectives review a given market, plus the `/review-contracts` and `/approve-contract` rewrites that wire critics → one comparison node → a human decision.

The intent's text, however, does not merely describe nine *perspectives* — it literally enumerates **nine named agent files** in `.claude/agents/`: `product-critic, ux-critic, architecture-critic, security-privacy-critic, compliance-risk-critic, qa-test-critic, reliability-ops-critic, cost-maintainability-critic, release-critic`. **This candidate does not create those nine files.** It creates one parameterized `specialist-critic` agent and a declarative `critic-registry` holding the nine perspectives and their routing triggers and rubrics. It therefore serves the intent's *purpose* — per-axis adversarial attack, class/scope routing, and the durable comparison — while diverging from the *letter* of the enumeration. That literal-fidelity gap is the central, deliberate tension of this candidate, and it is named here, in Trade-offs, and in an acceptance example so a reviewer cannot miss it. It is the strongest case *against* this candidate and the thing the architecture-critic, spec-critic, and the human must weigh. This contract does not claim to satisfy the nine-named-files enumeration; the sibling `contract-critics-literal-panel` does that literally (nine bespoke files), and `contract-critics-named-thin-rubric` splits the difference (nine thin named files delegating to a shared rubric).

## Scope

### Shared core (identical across all three candidates)

The schema, loader, validate-rule, command-rewrite, and CLAUDE.md work below is identical in all three candidate contracts of this market. The candidates differ *only* in how the critic agents and their routing are encoded (see "What distinguishes this candidate").

1. **Schema migration — one atomic commit.** In `node-types.yaml` add `comparison`: `required_fields: [id, type, title, created]`, `requires_body: true`, **no** `status_values`, **no** `class`. This is safe by existing-handler construction: `nodes-status-in-enum` skips any type lacking `status_values` (the `decision`/`override` precedent), and `nodes-class-in-range` (`class_range`) scopes only to `[intent, contract]`, so a `comparison` node needs neither status nor class. In `edge-types.yaml` add `compares: {source: comparison, target: contract}`. In `validation-rules.yaml` add a top-level key `comparison_required_from: "2026-06-18"` — a date string, kept a string because the loader parses with `CORE_SCHEMA`; an ISO `YYYY-MM-DD` value sorts lexically == chronologically. The cutoff is the day *after* this market's bootstrap contracts (`created: 2026-06-17`), so a lexical `>=` strictly grandfathers every existing contract, including these three candidates.
2. **Loader change.** Extend `tools/loader.ts` to surface `comparison_required_from` on `LoadedSpec`, parallel to the existing `sensitivePaths`. Absent ⇒ the rule is disabled (fail-open by omission).
3. **New validate rule `comparison-required`** (`kind: comparison_required`), placed as the **last data rule** — after `class-market-quorum` and `edges-endpoint-types`, before `indexes-fresh` — so it can assume edge endpoints already resolve. New handler `tools/handlers/comparison_required.ts`, registered in the `HANDLERS` map in `tools/validator.ts`, signature `(rule, spec) => Finding[]`. It **reuses `toDateString` from `tools/gate.ts`** and models its traversal on `class_market_quorum`: build `nodesById`, iterate `selects` edges, resolve the target contract, find the proposing intent via the contract's outgoing `proposes` edges, read `class` as a number. Semantics, per `selects` → contract `C`: **skip** if `C.class < 2`, or `toDateString(C.created) < comparison_required_from`, or either date is unparseable (grandfather / fail-open). Otherwise the selection is valid only if there exists a `comparison` node `K` whose resolved `compares` targets **cover the live (non-`superseded`) candidate set** proposing `C`'s intent — the *identical* live-candidate predicate `class_market_quorum` uses — **and** `K` has ≥2 resolved `compares` edges. No backfill; no writes to existing nodes.
4. **`/review-contracts` rewrite.** Route critics by intent class + the candidates' declared `## Scope`: class 0–1 → spec-critic only; class 2 → spec-critic + scope-matched specialists; class 3 → spec-critic + the full panel. Invoke each critic as a **real subagent** (replacing today's inline "Act as spec-critic" prose). Each critic appends a perspective-labelled `## Critique` to *every* candidate. Then graph-maintainer creates **one** `comparison` node (sections: trade-off table / findings by perspective / the case against each candidate) plus one `compares` edge per candidate; regenerates indexes; runs validate; asks for a human decision; never selects.
5. **`/approve-contract` update.** The decision body cites the `comparison` node id and records the accepted trade-off plus why each rejected candidate lost. Keep the existing `class-market-quorum` pre-check. Still authors no brief.
6. **CLAUDE.md.** Add the critic-routing-by-work-class table, the comparison mechanism, and the grandfathering rule.
7. The `comparison` body *structure* is a command/graph-maintainer convention, **not** a validate rule (the rule checks coverage and count, never section headings).

### What distinguishes this candidate

- **One parameterized `specialist-critic` agent file** in `.claude/agents/` (frontmatter `name: specialist-critic`, a `description`, `tools: Read, Grep, Glob`; never selects; all graph writes go through graph-maintainer). Its body instructs it to attack every candidate from the **single perspective named in its invocation prompt** — the caller passes the perspective name and that perspective's rubric — hunting that axis's missing cases, risks, and the strongest argument against each candidate, and ending in a perspective-labelled `## Critique`. The perspective is data, not a separate agent identity.
- **A declarative `critic-registry`** at `.claude/critic-registry.yaml` (chosen over `specs/` because it governs *agent invocation*, a `.claude/` concern alongside `agents/` and `commands/`, and must not become a spec node validated by the graph). It lists the nine perspectives, each with (a) **surface-triggers** — UI → ux; payments/personal-data → security-privacy + compliance-risk; schema/service-boundary → architecture; testing → qa-test; runtime/ops → reliability-ops; cost/maintainability → cost-maintainability; release/rollout → release; product/value → product — and (b) a **review rubric**: the numbered rules for that perspective.
- **Routing reads the registry.** `/review-contracts` computes the routed perspective set from intent class + the candidates' `## Scope` matched against the registry's surface-triggers, then invokes the one `specialist-critic` agent **once per routed perspective**, passing the perspective name and its rubric. CLAUDE.md names the registry as the **single source of truth** for routing; the command and CLAUDE.md both point at the same file rather than restating the nine-way mapping in prose.

## Non-scope

- **No nine bespoke agent files.** This candidate deliberately does not create the nine named agents the intent enumerates — that is `contract-critics-literal-panel`'s mechanism, and the thin-file variant is `contract-critics-named-thin-rubric`'s.
- **No change to the comparison-required rule's semantics across candidates.** The schema, loader, handler, and rule wording are the shared core; this candidate adds nothing to them.
- **No `status` or `class` on `comparison`** — its existence is its meaning, and adding either would pull it under `nodes-status-in-enum` / `class_range` for no benefit.
- **No backfill and no writes to existing nodes.** The cutoff grandfathers all prior selections; the rule simply skips them.
- **No machine validation of the comparison body's sections.** Section structure stays a convention enforced by the command and graph-maintainer, not a rule.
- **The registry is not a spec node.** It lives under `.claude/`, is not loaded by `tools/loader.ts`, and no validate rule reads it.

## Trade-offs

- **+** A single routing source of truth: CLAUDE.md and `/review-contracts` both read `.claude/critic-registry.yaml`, so the surface-trigger mapping cannot drift between the doc and the command.
- **+** Trivially extensible: adding a tenth perspective is one registry entry (triggers + rubric), with zero new agent files and no command edit.
- **+** No nine-way prompt drift: one agent body holds the shared "attack from one perspective, never select, end in a labelled `## Critique`" contract; only the per-axis rubric varies, and it varies as *data*.
- **+** Far fewer files to create, review, and keep consistent (one agent + one registry vs nine agents).
- **−** **Deviates from the intent's literal nine-named-agents enumeration** — this is the primary case against this candidate. The intent says "New agents in `.claude/agents/` ... product-critic, ux-critic, ...", and a reviewer who reads that as a binding file list will see this candidate as non-compliant on its face.
- **−** Weaker per-perspective specialization: one parameterized prompt is a less sharply-tuned reviewer than nine hand-written bespoke prompts could be, even with the rubric passed in.
- **−** Introduces a new file format and surface (the registry schema) that must be defined and kept stable, and which lives outside the validated graph, so no `spec:validate` rule guards it against rot.

## Risks

- **The literal-fidelity gap is rejected outright in review.** A reviewer or the architecture-critic may hold the intent's enumeration as binding and reject this candidate for not producing nine named files. **Mitigation:** the registry preserves every perspective, every surface-trigger, and the full routing the intent asked for; the divergence is purely *how* the perspectives are materialized, and this contract states the gap plainly rather than burying it — so the human decision is made on an honest comparison, not a hidden one. If the enumeration is binding, a sibling candidate is the right choice and that is an acceptable market outcome.
- **Registry-format bikeshedding.** A new YAML format invites debate over keys, nesting, and where rubrics live. **Mitigation:** the brief pins a minimal schema (a top-level map: perspective → `{triggers: [...], rubric: [...]}`) and one worked example entry; the format is intentionally flat and unvalidated-by-graph to keep the surface small.
- **A single parameterized prompt is a weaker specialist than a bespoke one.** **Mitigation:** the caller passes the *full* per-perspective rubric from the registry into each invocation, so the agent reviews against the same numbered rules a bespoke agent would carry — the specialization lives in the rubric data, and the rubric is as detailed as a bespoke prompt's body would be.
- **Routing mis-classification from scope text.** Surface-trigger matching reads free-text `## Scope`; a candidate that touches payments without saying "payments" could miss the security-privacy/compliance perspectives. **Mitigation:** class 3 invokes the full panel regardless of apparent surface (the intent's own backstop), so the highest-risk class never depends on scope-text matching.

## Acceptance examples

1. **Schema first, green.** After the migration commit, `pnpm spec:validate` is green; `comparison` appears in `node-types.yaml` (no `status_values`, no `class`), `compares` in `edge-types.yaml`, `comparison_required_from: "2026-06-18"` in `validation-rules.yaml`, and a `comparison` group exists in `specs/indexes/by-type.yaml`.
2. **Cutoff blocks a new class-2 selection.** Create a post-cutoff (`created: 2026-06-19`) class-2 intent with two live candidate contracts and a `decision —selects→` one of them, but **no** comparison node. `pnpm spec:validate` fails on rule **`comparison-required`** with a detail naming the selected contract and the missing covering comparison. Add a `comparison` node with `compares` edges covering both candidates → the rule flips green.
3. **Coverage and count both required.** A post-cutoff class-2 selection with a comparison node that has only **one** `compares` edge fails `comparison-required` (the ≥2-edges clause); a comparison node with two `compares` edges that do **not** cover the live candidate set also fails (the coverage clause). Only a node satisfying both passes.
4. **Grandfathering.** Every pre-cutoff selection — all Phase 1–6 contracts and this phase's own three bootstrap contracts (`created: 2026-06-17`) — still validates with no comparison node, because `toDateString(C.created) < "2026-06-18"`, so `comparison-required` skips them and the whole graph stays green.
5. **Class-1 needs no comparison.** A post-cutoff class-1 selection passes `comparison-required` with no comparison node (the `C.class < 2` skip).
6. **Scope-routed critics on a class-2 UI market.** `/review-contracts` on a UI-touching class-2 intent computes the routed set from the registry's surface-triggers and invokes `specialist-critic` with the **ux** perspective — and **not** with the security-privacy or compliance-risk perspectives, because no candidate's `## Scope` mentions payments or personal data.
7. **Full panel on a class-3 market.** `/review-contracts` on a class-3 intent invokes `specialist-critic` once per perspective for **all nine** registry perspectives regardless of apparent surface, each appending a perspective-labelled `## Critique` to every candidate, after which graph-maintainer produces one comparison node + one `compares` edge per candidate.
8. **The durable record persists.** After a human decision, the comparison node remains in `specs/nodes/` as the standing record of why each loser lost, referenced by the decision body via its id.
9. **The literal deviation is visible.** `.claude/agents/` contains `specialist-critic.md` and **not** nine files named `product-critic.md` … `release-critic.md`; routing data lives in `.claude/critic-registry.yaml`. A reviewer must explicitly accept that this candidate satisfies the intent's *purpose* but not the *letter* of its nine-named-files enumeration. (If that trade is unacceptable, a sibling candidate is the intended choice.)

## Critique
This critique was drafted by spec-critic against the real graph and tools/ on 2026-06-17; it reviews contract-critics-registry-driven-2d5b on its own terms and does not rank it against the other candidates.

### Inherited common-core gaps (schema / handler / cutoff / command rewrites)

1. **The schema migration and the handler registration are not actually "one atomic commit" the way the contract describes them, and the contract never says which file lands first.** `tools/validator.ts` (HANDLERS, lines 24-34) has no `comparison_required` key, and the unknown-kind branch (lines 68-76) emits a finding for any rule whose `kind` has no handler. So the instant the `comparison-required` rule entry appears in `validation-rules.yaml` *without* the handler also present and imported, `spec:validate` reddens with a finding of the form `rule comparison-required has unknown kind 'comparison_required'`. The contract calls this "one atomic commit" but never states that the handler file, the `import`, the HANDLERS entry, and the rule entry must all be in that single commit — a brief that splits them (e.g. schema + rule first, handler second) breaks the tree. Concrete failure: a reviewer approves the schema diff alone, CI goes red, and the "schema first, green" acceptance example (item 1) is unreachable.

2. **The loader change is asserted as "parallel to `sensitivePaths`" but the fail-open semantics are weaker than claimed.** `tools/loader.ts` reads `sensitive_paths` as a list (declared on `LoadedSpec` at line 47, assigned at lines 128-130); `comparison_required_from` is a scalar string and is *not* surfaced on `LoadedSpec` today. The contract says "Absent ⇒ the rule is disabled (fail-open by omission)," but it does not specify behaviour when the key is *present but malformed* (e.g. `comparison_required_from: 2026-13-99`, or a non-string, or an accidental list). `toDateString` (defensively rejecting impossible months/days via a `Date.UTC` round-trip) would return `undefined` for such a cutoff, and the contract's own handler semantics say "either date unparseable ⇒ skip" (Scope item 3) — meaning a typo'd cutoff silently disables the entire comparison gate with a green graph. There is no acceptance example asserting that a *malformed* cutoff is loud rather than silently fail-open, and no validation that the cutoff key itself is well-formed. For a class-3 security/compliance-adjacent mechanism, silent disablement on a one-character typo is a real hole.

3. **The cutoff date `2026-06-18` is hardcoded to "the day after this market's bootstrap" but the contract is itself `created: 2026-06-17` and not yet selected.** If selection or merge slips past 2026-06-18 and any *new* class-≥2 contract is created on/after that date in the interim, that contract is no longer grandfathered and instantly requires a comparison node that may not exist — reddening the tree on an unrelated market. The contract treats the cutoff as a static fact of "this migration's timestamp" (matching the intent), but never addresses the window between proposal and merge. Concrete failure: a parallel class-2 intent gets a `selects` on 2026-06-19 before this migration merges, and once it does merge the quorum-style backstop fails retroactively.

4. **The coverage predicate is borrowed from `class_market_quorum` but the two rules can disagree on "the live candidate set" at different points in time.** `class_market_quorum` (its `liveCandidates` helper, lines 26-38) counts live `proposes` edges at validate time, skipping only `superseded` sources. The comparison node's `compares` edges are a *snapshot* captured when graph-maintainer built it. If a fourth candidate is proposed after the comparison node is written but before the `selects` decision, the live set grows to four while the comparison covers three — the contract's coverage clause then *correctly* reddens, but the contract never names this race or tells graph-maintainer to regenerate the comparison when a late candidate appears. Conversely, if a covered candidate is later superseded, the live set shrinks and the comparison now over-covers; the contract says coverage must "cover the live set" but is silent on whether *extra* `compares` edges to a now-superseded candidate are tolerated. Acceptance example 3 tests under-coverage (the 1-edge count clause) and wrong-coverage (a node not covering the live set) but not the superseded-after-comparison case.

5. **The command rewrites (`/review-contracts`, `/approve-contract`) are described but their failure and idempotency behaviour is unspecified.** What happens when `/review-contracts` is re-run on an intent that already has a comparison node and per-candidate `## Critique` sections? The contract says graph-maintainer "creates **one** comparison node" (Scope item 4) but never says the command must detect and refuse/replace an existing one — a second run would author a second comparison node and a second set of `compares` edges, and nothing in the schema forbids two comparison nodes for one intent. The coverage clause would still pass (both cover the set), so the duplicate is invisible to validate. Concrete failure: a re-run leaves two comparison nodes; the decision body cites one id; the other is orphaned but green.

### Weaknesses specific to the registry-driven axis

6. **The single source of truth lives *outside* the validated graph, so the contract's own central "+" is unguarded by the toolchain it relies on everywhere else.** `.claude/critic-registry.yaml` is explicitly "not loaded by `tools/loader.ts`, and no validate rule reads it" (Non-scope). That means the registry can drift from CLAUDE.md, the routing logic in `/review-contracts`, and the nine perspectives the intent enumerates, with *zero* `spec:validate` signal. The contract claims "the surface-trigger mapping cannot drift between the doc and the command" because both read the file — but CLAUDE.md is prose a human maintains, and nothing forces CLAUDE.md's table to match the registry. A perspective deleted from the registry (or a triggers key typo'd to `trigers:`) silently stops routing that critic, and the only backstop is the class-3 full-panel fallback (the "Routing mis-classification from scope text" risk) — which by construction does *not* protect class-2 markets. So a class-2 payments market with a typo'd `security-privacy` trigger ships with no security review and a green tree.

7. **Routing on free-text `## Scope` is brittle, and the contract's mitigation only covers class 3.** The contract concedes (Risk: "Routing mis-classification from scope text") that a class-2 candidate touching payments without the word "payments" misses the security-privacy/compliance perspectives, and mitigates only by noting class 3 runs the full panel regardless. But the intent's whole class-2 design is *selective* routing by scope text. Concrete failure: a class-2 candidate whose `## Scope` says "store the customer's card token" never matches a `payments`/`personal-data` trigger keyed on those literal words, so no security or compliance critic is invoked, and the durable comparison node records a confident "case against each candidate" that simply never examined the payments risk. The "single perspective passed in the prompt" design makes this worse than nine bespoke agents would be: there is no agent whose standing identity is "I always look for payments," only a parameterized prompt that runs only if routing fired.

8. **"The specialization lives in the rubric data" is an untested assumption about subagent behaviour.** The contract's mitigation for weaker specialization is that the caller passes "the *full* per-perspective rubric" so the one agent reviews "against the same numbered rules a bespoke agent would carry." But a single agent body that says "attack from the perspective named in your invocation prompt" plus injected rubric text is empirically a weaker prime than a dedicated agent file whose entire body and name reinforce one perspective — and the contract offers no acceptance test that the parameterized agent actually produces perspective-distinct critiques rather than nine near-identical generic ones. Verification need "Agent contract" checks the *frontmatter and invocation wiring* (that `specialist-critic.md` declares `tools: Read, Grep, Glob`, never selects, and is invoked once per routed perspective), not the *quality or distinctness* of the resulting critiques, so the central trade-off is asserted, not demonstrated.

9. **Registry schema and the agent prompt format are new, unversioned surfaces with no migration story.** The contract pins "a minimal schema (perspective → `{triggers, rubric}`)" in the brief (Risk: registry-format bikeshedding), but a YAML format outside the graph has no schema-version field, no validation, and no test that the *command's* parser and the *agent's* expectations stay in sync. If the registry format later needs a third key (say, `severity` or `applies_to_class`), there is no mechanism that forces the command, CLAUDE.md, and the agent prompt to migrate together — exactly the drift the in-graph schema files (`node-types.yaml`, `edge-types.yaml`) are designed to prevent for nodes/edges.

10. **Literal-fidelity gap, on its own terms.** The intent does not say "nine perspectives"; it says "New agents in .claude/agents/ ... product-critic, ux-critic, ..." and enumerates nine file-named identities. This candidate ships *one* file, `specialist-critic.md`, and a data file. A reviewer holding the enumeration as the contract's deliverable list will read this as not delivering the named artifacts at all — and acceptance example 9 confirms `.claude/agents/` will *not* contain `product-critic.md` … `release-critic.md`. The contract is admirably honest that this is the case against it, but honesty does not close the gap: if any downstream tooling, doc, or future intent references `security-privacy-critic` as an agent name (the intent's own wording does), that reference dangles, because no such agent file exists. (Today `.claude/agents/` holds only generic agents — `spec-critic`, `contract-reviewer`, `graph-maintainer`, `spec-writer` — so no critic-named agent file exists in either form yet.)

### Net

The candidate is internally coherent and its code claims are accurate — `toDateString` is genuinely exported from `tools/gate.ts` (line 42), the `class_market_quorum` live-candidate predicate (skipping `superseded`, line 34) and numeric-class read (`typeof cls !== "number"`, line 65) are real, and the `decision`/`override` no-`status_values` precedent and the `class_range` rule's scoping to `[intent, contract]` (validation-rules.yaml line 39) do support a status-less, class-less `comparison` type. Its strongest exposures are (a) shared-core silent-failure modes the wording papers over — non-atomic handler registration reddening the tree, a malformed-cutoff fail-open that disables the gate on a typo, an unspecified cutoff-window race, and non-idempotent `/review-contracts` re-runs — and (b) the axis-specific decision to put the single routing source of truth and the per-perspective specialization *outside* the validated graph, which trades enforceability for extensibility precisely on a class-3, security/compliance-touching mechanism where the missing-perspective failure mode (class-2 payments scope that never trips a trigger) ships green and silent. The literal nine-named-files deviation is the loudest objection but the most honestly disclosed; the quieter unguarded-registry and silent-fail-open holes are the ones a reviewer should weigh hardest.
