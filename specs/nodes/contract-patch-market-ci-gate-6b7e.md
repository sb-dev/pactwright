This intent (`intent-patch-market-synthesis-3b1e`, class 3) is multi-surface — schema,
validator, four commands, `CLAUDE.md`, and CI. It is a **Class 3** change, so this market
carries three candidate contracts (this is candidate **B** of three). All three share an
identical common core and differ on one axis: **where the multi-patch merge gate lives.**
Candidate **B** realises that gate as a **new diff-aware subcommand** `spec:patch-gate`
(modelled on `tools/gate.ts`/`tools/checkdiff.ts`) consumed by a dedicated
`patch-comparison.yml` workflow, overridable through the existing `override` + `waives`
mechanism — the most literal reading of the intent's "merging its implementation PR
requires … overridable via an override node." `spec:validate` stays pure (only the two
structural rules).

## Problem interpretation

The patch market the intent describes (per-lane candidate patches → one `comparison` →
human `/select-patch`; within-lane synthesis via `synthesizes` edges; cross-lane
combination is integration, never synthesis; judged in lane isolation) is **identical in
all three candidates.** So is the irreducible common core, which is the same as candidate
A (`contract-patch-market-validate-authoritative-9d41`); see its Problem interpretation
for the full text. Restated in brief:

- **Schema (`node-types.yaml`).** New `patch` node (`required_fields [id, type, title,
  status, branch, strategy, created]`, `status_values [candidate, selected, superseded]`,
  `requires_body: true`); optional boolean `patch_market` flag on `brief` (unset = no
  market, like the `lane` precedent).
- **Schema (`edge-types.yaml`).** New `competes-for` (`patch → brief`) and `synthesizes`
  (`patch → patch`); widen `compares.target` and `selects.target` to `[contract, patch]`.
  **No `edge_endpoint_types` change needed** — list targets are already supported (the
  `flags` edge uses `target: [evidence, capability]`). (The intent's "`selects` already
  allows decision → patch" is inaccurate; today it is `decision → contract` only — the
  widening is in-scope work, not a scope change.)
- **Commands (`.claude/commands/`).** `/propose-patches`, `/compare-patches`,
  `/synthesize-patches`, `/select-patch` with exactly the behaviours the intent specifies
  (branch `patch/<brief-slug>/<strategy>`, draft PRs, `patch` nodes + `competes-for`
  edges, `patch_market: true`; comparison scoring contract fit / scope control /
  simplicity / test quality / drift risk / rollback safety; synthesis on
  `patch/<brief-slug>/synthesis` competing for the SAME lane brief; selection supersedes
  losers + closes their PRs + keeps branches, winner → `/prepare-evidence`), carrying the
  Phase-6 scope-integrity rules into patch review.
- **CLAUDE.md.** Per-lane patch-market prose; the by-class row (0–1 single/no market, 2
  optional, 3 per lane); within-lane synthesis vs across-lane integration; scope-integrity
  in patch review.
- **Two intent-mandated `spec:validate` extensions (structural).** `synthesis_parentage`
  (a synthesis patch needs ≥2 `synthesizes` edges) and selected-patch comparison coverage
  (a `selects`-edged patch's brief must have a `comparison` covering its competing
  patches) — pure `Finding[]` handlers after `edges-references-resolve`, counting only
  non-superseded competitors.

What distinguishes **candidate B**: the **merge gate** is a **diff-aware
`spec:patch-gate` subcommand** in `tools/`, structured exactly like `tools/gate.ts`
(`evaluateGate`) and `tools/checkdiff.ts` (`evaluateCheckDiff`): given the PR base
(`$GATE_BASE`), it identifies the brief the PR implements — from the head branch
`patch/<brief-slug>/<strategy>` and/or the `patch` node whose `branch` equals the head —
counts that brief's **live** `competes-for` edges, and if >1 requires a `comparison`
covering them plus a `selects` decision in the graph; else it **fails the merge**. It is
overridable through the existing pattern: an added `override` + `waives → patch-comparison`
(a new entry in `specs/schema/checks.yaml`), with the same non-expired-`expires` check the
other gates use. A dedicated `patch-comparison.yml` workflow runs it on every PR.
`spec:validate` is untouched beyond the two structural rules — it stays diff-agnostic.

## Scope

- **Common core** — schema, four commands, CLAUDE.md, the two structural validate rules,
  as above (identical to candidates A and C).
- **`tools/patch_gate.ts` (new) + a `patch-gate` case in `tools/spec.ts` + a `spec:patch-gate`
  script** — a pure `evaluatePatchGate(input)` over an immutable input (added/changed ids,
  the PR head branch, the live graph, today), wrapped by a `runPatchGate()` that derives
  git state via the existing `gitdiff.ts` helpers (`addedNodeIds`, `changedFiles`,
  `resolveBase`) and the override-expiry helper (`toDateString`) reused from `gate.ts`. The
  PR→brief mapping is graph-first (the `patch` node whose `branch` == head → its
  `competes-for` brief), with the branch-name convention as the fallback.
- **`specs/schema/checks.yaml`** — add the `patch-comparison` named check.
- **`.github/workflows/patch-comparison.yml`** — runs `pnpm spec:patch-gate` with
  `GATE_BASE: ${{ github.event.pull_request.base.sha }}`, modelled on `pr-evidence.yml`;
  a required check that fails the PR when the gate fails.
- **`tools/validator.ts` + `tools/handlers/*` + `validation-rules.yaml`** — only the two
  structural rules; no third "resolved" rule (that is candidate A's mechanism).
- **Capability ownership, this PR** — per the Phase-8 capability-wiring flow.

## Non-scope

- **No merge-gate logic in `spec:validate`.** The gate is diff-aware CI, never a graph-state
  validate rule. This is B's defining choice and is candidate A's mechanism, not adopted.
- **No fold into `pr-evidence`.** B keeps a **separate** gate and workflow; reusing the
  existing `spec:gate` is candidate C's mechanism, not adopted.
- **No PR-state tracking in the graph;** no across-lane synthesis; no auto-selection; no
  release/`includes` work (as in all candidates).

## Trade-offs

- **+** Mirrors the established `pr-evidence`/`check-diff` pattern exactly: "block the
  merge PR" and "overridable via an override node" fall out for free from the existing
  gate + `waives` machinery — **the most literal reading of the intent.**
- **+** `spec:validate` stays diff- and override-agnostic — no architectural impurity; the
  two structural rules remain pure graph-shape checks.
- **+** WIP branches stay **green** in `spec:validate` mid-market; the merge gate is a
  separate concern that fires only on the PR actually being merged, with real PR context.
- **+** Reuses `gitdiff.ts` + `toDateString` + the `override`/`waives` path — the new
  surface is one gate module + one workflow + one checks entry.
- **−** Patch-market knowledge now lives in **two places** — `tools/patch_gate.ts` (the
  merge gate) and `tools/handlers/*` (the structural rules) — a drift surface (the same
  con Phase-6's `command-discipline` carried).
- **−** Needs a robust **PR→brief mapping**; a branch rename or a missing patch node could
  misidentify the brief. (Mitigated by graph-first mapping, but it is real complexity A
  does not have.)
- **−** Largest new **tool surface** of the three (new subcommand + module + workflow +
  named check), and the gate is testable only with synthetic diff inputs, not pure graph
  fixtures.

## Risks

- **PR→brief misidentification.** If `spec:patch-gate` can't map the PR to a brief, it must
  **fail closed** (treat as ungated only when there is provably ≤1 competitor), never
  silently pass. **Mitigation:** prefer the graph mapping (`patch.branch` == head →
  `competes-for` brief); fall back to branch-name parse; unit-test both and the
  no-match case.
- **Two-places drift.** The structural rules and the gate could diverge on "what counts as
  a live competitor." **Mitigation:** share one helper (counting non-superseded
  `competes-for` sources) between `patch_gate.ts` and the handlers; test parity.
- **Status-blind competitor count (the recurring trap).** **Mitigation:** count only
  non-superseded patches as competitors, in both the gate and the structural rules.
- **Override expiry / date parsing.** **Mitigation:** reuse `toDateString` and the
  `expires >= today` check verbatim from `gate.ts`; test an expired override does not
  suppress.

## Acceptance examples

1. **(Two real candidate patches → compare → select, fully traced.)** As in all candidates:
   `/propose-patches <brief> 2 strategyA,strategyB` → two `patch` nodes + `competes-for`
   edges; `/compare-patches` → one `comparison` with `compares` edges to both;
   `/select-patch <winner>` → a `decision` + `selects` edge, loser superseded; chain fully
   walkable.
2. **(Multi-patch brief whose PR skips comparison is blocked by `patch-comparison.yml`.)**
   A PR on `patch/<brief-slug>/winner` for a brief with two live `competes-for` edges and
   **no** `comparison`/`selects` makes `spec:patch-gate` **exit non-zero**, failing the
   `patch-comparison` required check. Recording the `comparison` + a `selects` makes the
   gate pass. This is the intent's headline acceptance, met by the literal named workflow.
3. **(Override.)** An added `override` + `waives → patch-comparison` (non-expired) makes
   the gate pass with the market still unresolved; an expired override does not.
4. **(`spec:validate` stays green mid-market.)** While the market is open (patches proposed,
   not yet compared), `spec:validate` is **green** — the structural rules don't fire, and
   the merge gate is not a validate rule. Only `patch-comparison.yml` blocks the merge.
5. **(Synthesis.)** `/synthesize-patches <brief> p1,p2 "<instruction>"` → a synthesis
   `patch` on `patch/<brief-slug>/synthesis` with `synthesizes` edges to p1,p2 and a
   `competes-for` to the same brief; selecting it supersedes p1,p2; a synthesis patch with
   one `synthesizes` edge fails `synthesis_parentage`.

## Critique (spec)

**1. The defining strength — keeping `spec:validate` pure — is real and correctly matches the existing override mechanism; this is the cleanest fit with how gates work here.** B routes the merge gate through a diff-aware `spec:patch-gate` modelled on `tools/gate.ts`, reusing `resolveBase`/`addedNodeIds` from `gitdiff.ts`, `toDateString`, and the `override` + `waives → <check>` pattern. That is exactly the shape of `evaluateGate`: a pure `(spec, input)` decision with `input.today` supplied by the runner, expiry checked via `toDateString(node.data["expires"]) >= today` (gate.ts lines 109-120). Unlike candidate A, B's override carries a real run-date and a real expiry semantics, and `spec:validate`'s `HANDLERS` map stays all-pure. **No concern on the merge-gate-location axis itself** — siting the date-dependent, override-aware logic in a diff gate rather than in `spec:validate` is the architecturally consistent choice. The concerns below are about completeness of that choice, not its location.

**2. The new `patch-comparison` check must be added to `checks.yaml` or the override silently can't be authored — and `references_resolve` will mis-handle it.** B's Scope says "add the `patch-comparison` named check" to `specs/schema/checks.yaml` (today lines 8-16: `ci, spec-index, spec-validate, pr-evidence, check-diff, drift`). This is load-bearing and must not be hand-waved: per edge-types.yaml lines 30-36, a `waives` edge has `target: any` and `references_resolve` "treats a known check name as resolved." If the new `patch-comparison` literal is *not* registered in `checks.yaml`, a `waives → patch-comparison` override edge has an **unresolved target** and the graph goes red on `edges-references-resolve` — so the very override B relies on for WIP cannot be expressed in a green graph. **Fix/question:** confirm B's PR adds `patch-comparison` to `checks.yaml` *and* that `spec:patch-gate` matches that exact literal (mirroring `gate.ts`'s `PR_EVIDENCE_CHECK = "pr-evidence"` constant, line 15), so the registry, the gate constant, and the workflow's check name are byte-identical.

**3. The PR→brief mapping is genuinely new logic with no precedent in `evaluateGate`, and "fail-closed on no match" can wedge ordinary PRs red.** `evaluateGate` never maps a PR to a brief — it scans `addedEdgeIds` for an `evidences` edge and walks `decomposes` (gate.ts lines 67-94); it needs no head-branch parsing at all. B's `spec:patch-gate` must newly identify "the brief the PR implements" from the head branch `patch/<brief-slug>/<strategy>` and/or the `patch` node whose `branch` field equals the head. B's Risk section says map graph-first, fall back to branch-name, and "fail closed." But fail-closed here means: any PR the mapper *can't* resolve to a single brief is treated as gated — so a perfectly ordinary non-patch PR with an unrecognized branch name could fail `patch-comparison` for a market it isn't part of. **Question for the selector:** what is the precise default when the mapper finds zero patch nodes for the head branch — does the gate *pass* (treating "no patch market here" as the common case) or *fail* (B's stated fail-closed)? Those are opposite behaviours for every non-market PR in the repo.

**4. (Shared core) Two-places "live competitor" counting is a real drift surface, and the shared widening of `selects`/`compares` collides with existing handlers neither contract patches.** B's `spec:patch-gate` counts "live `competes-for` edges" and the structural `synthesis_parentage`/coverage handlers count the same thing — B owns this as "two-places drift" and proposes a shared helper. Good, but note the precise precedent: `coverage_traversal.ts`'s `liveSourcesByEdge` already centralizes "distinct, resolved, non-superseded sources of an edge type," so the right move is to reuse it (with `edgeType: "competes-for"`), not to write a second counter — B should name that helper. Separately, and unaddressed by B exactly as by A and C: widening `selects.target` to `[contract, patch]` (edge-types.yaml line 13) makes `class_market_quorum.ts` (lines 27-43) and `comparison_required.ts` (line 61) iterate over `selects` edges that now point at *patches*. `intentsForContract` returns `[]` for a patch, so `class_market_quorum` emits its hard "targets contract X which proposes no intent" finding on **every patch selection**. **Fix:** require the shared core (in whichever candidate wins) to guard both existing handlers to `type === "contract"` selects targets before they treat the target as a contract.

## Critique (product)

Reviewed on the product / user-value axis only.

- **B is the only candidate that delivers the intent's acceptance criterion as written — and on my axis that literal fidelity is a genuine user-value, not a formality.** The intent's acceptance (lines 72-73) is concrete: "a multi-patch brief whose PR skips the comparison node is **blocked by patch-comparison.yml**." B is the only candidate that produces a workflow and required check literally named `patch-comparison` (body lines 57-67, 80-83, 141-145). This matters for product because the *user* of this feature is a future contributor or reviewer staring at a red PR: with B they see a check named `patch-comparison` failing and immediately know "I skipped the patch market," a self-describing, teachable signal. A folds it into `spec:validate` (a red graph for many possible reasons) and C folds it into `pr-evidence` (a check named for a *different* concern). **No concern on the literal-fidelity axis** — B alone makes the failure legible to the human at the moment they hit it; the selector should weigh how much that named-check legibility is worth.

- **The PR→brief mapping B introduces is net-new behaviour the existing gate does not have, and if it misfires the user-facing failure mode is the worst kind: a silent pass.** B's distinguishing mechanism reads the head branch `patch/<brief-slug>/<strategy>` and/or the `patch` node whose `branch` == head to identify the brief (body lines 57-66, 73-79). I verified this mapping does not exist today: `evaluateGate` in `tools/gate.ts` keys entirely on *added `evidences` edges in the PR diff* (gate.ts lines 63-126) — it never maps a PR to a brief by branch name. So B is inventing a new identification step. From the product perspective the risk is asymmetric: a *false fail* (gate blocks a legitimate merge) is annoying but visible and self-correcting; a *false pass* (gate can't map the PR, treats it as ungated, lets a multi-patch merge through with no comparison) silently destroys the exact guarantee the user is buying. B's Risks section says "fail closed" (lines 120-124), but "fail closed only when there is provably ≤1 competitor" still passes the PR whenever the mapping *fails to find a brief at all* — a renamed branch or a missing/late `patch` node. **Fix / question for the selector:** demand the explicit truth table for "branch doesn't parse AND no patch node matches head" — does B block the merge (safe, may annoy) or pass it (silently voids the guarantee)? The intent's whole value rests on this one cell.

- **Two-places enforcement means the user can get conflicting green/red signals for the same market, eroding trust in the gate.** B keeps the structural rules in `spec:validate` (`synthesis_parentage`, coverage) and the merge gate in a *separate* `spec:patch-gate` (body lines 51-67, 108-110). These two read "what counts as a live competitor" independently. If they ever diverge on superseded-counting (the recurring status-blind trap, flagged in B's own Risks lines 128-129), a contributor could see `spec:validate` green but `patch-comparison` red (or vice versa) for the same brief — and have no way to reconcile which is telling the truth. From a product standpoint, two gates that can disagree about the same fact teach users to distrust both and to merge on whichever is green. **Fix / question:** require B to share one `liveCompetitors()` helper between `patch_gate.ts` and the handlers (B proposes this, lines 126-127) and to state that the gate and the rules can never reach opposite verdicts on the same brief — otherwise the user faces an unresolvable mixed signal.

- **Shared-core: B inherits the "nodes exist ≠ human judged" gap, and its extra workflow surface raises the bar for the value to be worth it.** Like A and C, B's gate checks that a `comparison` node and `selects` exist, not that the comparison was substantive (the same honest bound). But B pays the *most* for this — a new subcommand, a new module, a new workflow, a new named check (body lines 73-83, 114-116) — so the product question is sharpest here: the user-visible benefit over A/C is exactly one self-describing named check (first finding), bought with the largest new surface. **Question for the selector:** is the legibility of a check literally named `patch-comparison` worth B's added surface, given that the underlying guarantee ("a human actually compared on the six axes," intent line 32-33) is no stronger in B than in A or C?

For the operator this means the local, always-available signal (`spec:validate`, which every other command's closing report leans on per `prepare-evidence.md` line 28 etc.) reports GREEN on an unresolved market, and the only thing that ever says "you must compare and select before merging" is a CI check that runs on the PR. An override that only the implementer knows about is not a user-reachable escape hatch.

## Critique (architecture)

**1. The "reuse `gate.ts`/`checkdiff.ts` pattern" claim hides a genuinely new git surface: nothing in the codebase reads the PR head branch name, which B's mapping depends on.** B's distinguishing mechanism maps the PR to its brief "from the head branch `patch/<brief-slug>/<strategy>` and/or the `patch` node whose `branch` equals the head" (line 60-61, Scope line 77-79). But `gitdiff.ts` exposes exactly four primitives — `resolveBase`, `addedEdgeIds`, `addedNodeIds`, `changedFiles` — and *none* returns the current branch name; `GateInput` (gate.ts:17-24) and `CheckDiffInput` (checkdiff.ts:22-31) carry only added-id sets, changed files, and `today`. The branch-name leg of B's mapping is net-new git plumbing (`git rev-parse --abbrev-ref HEAD` or equivalent) that must also fail closed in the `gitdiff.ts` style, plus a new dependence on a branch *naming convention* (`patch/<brief-slug>/<strategy>`) that no validation rule enforces and a branch rename silently breaks. B's "+ Reuses `gitdiff.ts`" trade-off (line 106-107) overstates the reuse. *Fix/question: ask B to enumerate the new `gitdiff.ts`/`patch_gate.ts` git calls precisely, and to state what `spec:patch-gate` does when the head branch does not match `patch/.../...` and no `patch` node's `branch` equals the head — fail-closed how, exactly?*

**2. The `patch` node's `branch` field is the load-bearing join key for B's whole gate, yet nothing keeps it equal to the real git branch — a single point of silent failure A does not have.** B's graph-first mapping pivots on "the `patch` node whose `branch` == head" (line 60, 77-78). But `branch` is a free-text frontmatter string (`node-types.yaml`, B's own schema: `required_fields [..., branch, ...]`) with no validation rule tying it to an actual ref; `/propose-patches` writes it once at creation and git history can diverge (rename, rebase to a new branch, force-push). When `branch` and the real head disagree, B's gate either misidentifies the brief (counts the wrong `competes-for` set) or finds no match and must fall back to fragile branch-name parsing. A reads `competes-for`/`compares`/`selects` edges directly and is "immune to branch renames" (A's trade-off, line 143-144); B reintroduces exactly the PR→artifact mapping problem the edge-based design was meant to retire. *Fix/question: ask B what validates that a `patch` node's `branch` still names a live ref at gate time, and what the gate does on the (common) rebase/rename case — does it fail open (market slips through) or fail closed (legitimate merges blocked)?*

**3. Two-places enforcement is a real drift surface on the definition of "live competitor", and the shared helper B proposes does not eliminate it.** B keeps the structural rules in `tools/handlers/*` and the merge gate in `tools/patch_gate.ts` (Trade-offs, line 108-110, conceded as "a drift surface"). Both must agree on "count only non-superseded `competes-for` sources" — exactly the status-blind trap a prior critique caught on the quorum rule (B's Risks, line 128-129). The existing precedent for "count live sources" is `coverage_traversal.ts`'s `liveSourcesByEdge` (excludeStatus `"superseded"`); but that helper takes a `byId` map and walks `spec.edges` — it is a *validate*-side primitive. For `patch_gate.ts` to share it, the gate must construct the same `LoadedSpec`/`byId` and call the same traversal, which is feasible but means the gate and the handlers are coupled to one traversal module whose semantics (superseded-skip, unresolved-skip) must hold identically in a diff context. If B instead re-implements the count in `patch_gate.ts`, the two definitions can silently diverge the day someone adds a new patch status. *Fix/question: ask B to commit to importing `liveSourcesByEdge` (or a `competes-for` analogue) into `patch_gate.ts` verbatim, with a parity test, rather than re-deriving the live-competitor count — and to show the gate can build the graph it needs without duplicating loader logic.*

**4. Shared-core: the widened `selects.target` collides with the existing contract-only `selects` consumers, and B's "validate stays pure" stance means B never touches them.** As with all three candidates, widening `selects.target` to `[contract, patch]` makes `selects → patch` a legal edge (`edge_endpoint_types.ts:65-67` handles the list for free), but `class_market_quorum.ts:28-42`, `comparison_required.ts:62-73`, and `coverage_coherence.ts` all call `intentsForContract(spec, <selects-target>)` assuming a contract; a patch target yields the spurious red *"selects edge … targets contract `<patch-id>` which proposes no intent"* (class_market_quorum.ts:40). ## Critique (security-privacy)

Reviewed from the security-and-privacy axis only (trust boundaries, fail-closed behaviour, abuse/injection paths in the diff-aware gate). B introduces the largest new enforcement-code surface, so the boundary of concern is **what the new `spec:patch-gate` trusts as input and how it fails when that input is attacker- or accident-controlled.**

1. **The PR→brief mapping makes the *head branch name* a trusted security input, and a branch name is attacker-/typo-controllable — the contract names no fail-closed default that survives a non-conventional branch.** B derives the gated brief "from the head branch `patch/<brief-slug>/<strategy>` and/or the `patch` node whose `branch` equals the head" (contract lines 58-60, 78-79). But no existing gate input carries a branch name: `GateInput`/`CheckDiffInput` and the `gitdiff.ts` helpers expose only added-id sets, `changedFiles`, and `today` (`tools/gitdiff.ts`; `tools/gate.ts` lines 17-24) — the branch must be threaded in fresh from `$GATE_BASE`-adjacent CI context. That means the gate's verdict now depends on a string an author chooses. Concrete abuse path: a multi-patch winner pushed on a branch that does *not* match `patch/<slug>/<strategy>` (renamed, or named to a *different* brief's slug) maps to the wrong brief or to no brief — and B's stated fallback "treat as ungated only when there is provably ≤1 competitor" (Risks line 121) means a mapping miss can land a multi-patch merge with no comparison. Fix: define the *only* trusted mapping as graph-state (the `patch` node whose `branch` == head, authored by `/propose-patches`, not by the merger), and on any mapping ambiguity or miss fail CLOSED (block), never "treat as ungated."

2. **A new `spec:patch-gate` that re-implements override/expiry parsing is a second copy of security-critical date logic that can drift from the audited original.** B reuses `toDateString` and "the same non-expired-`expires` check the other gates use" (lines 64-66), which is correct in intent — but it also adds a *new* `evaluatePatchGate` with its own override-scan loop modelled on `gate.ts` lines 100-121. The security risk is divergence: if the new loop omits the near-miss/`continue` behaviour (a malformed or expired override must NOT hard-pass and must keep scanning — `gate.ts` lines 110-120), an attacker or a sloppy author could slip a malformed-`expires` override past the new gate that `evaluateGate` would have rejected. B's own Trade-offs flag "patch-market knowledge now lives in two places… a drift surface" (line 108-110), but frames it as maintainability; on this axis it is a *waiver-bypass* surface. Fix: do not re-implement the override scan — extract the existing `gate.ts` override-evaluation (node-is-override, `expires` parse, `expires >= today`, expired-does-not-suppress) into one shared helper consumed by both gates, and assert parity in tests, including the expired-override and malformed-`expires` cases.

3. **A new always-on `patch-comparison.yml` required check that runs on *every* PR widens the blast radius of a gate bug into a repo-wide merge freeze.** B adds a dedicated workflow run "on every PR" as a required check (lines 80-83, 66). Combined with finding 1's branch-name input, a defect in branch parsing or in `resolveBase`/`changedFiles` (which fail CLOSED by design — `gitdiff.ts` throws rather than yielding empty sets, lines 14-21, 33-47) means any infra hiccup blocks *all* merges, not just patch-market ones. That is the correct security posture (fail closed) but the selector should weigh that a brand-new always-on required check is a new single point of merge-failure with no graceful degradation. Sharp question: is the dedicated `patch-comparison` check gated only on PRs that actually carry patch nodes/branches, or does every PR now pay the failure surface of branch-name parsing it has no patches to parse?

4. **Shared-core, on-axis: the override waiver is a same-PR token in `evaluateGate` — confirm B preserves that bound rather than honouring a standing graph waiver.** Because B routes through a diff-aware gate like `evaluateGate`, it *can* keep the override bound to `addedEdgeIds`/`addedNodeIds` (`gate.ts` lines 100-108), which is the safer semantics than candidate A's standing-graph waiver. The contract says "overridable through the existing pattern" (line 64) but does not state explicitly that the `override` node and `waives → patch-comparison` edge must be *added in this PR*. Fix: make B's gate require the override be diff-bound (same-PR) exactly as `evaluateGate` does, and test that a pre-existing, untouched `waives → patch-comparison` left in the tree does NOT suppress a later market's gate.

## Critique (compliance-risk)

Candidate B realises the merge gate as a new diff-aware `spec:patch-gate` subcommand (modelled on `tools/gate.ts`/`tools/checkdiff.ts`), a dedicated `patch-comparison.yml` workflow, and a new `patch-comparison` named check in `checks.yaml`, overridable via `waives -> patch-comparison`; `spec:validate` stays pure.

If they diverge — say the gate counts a `selected` patch as live but the handler does not, given `patch.status` is tri-valued `[candidate, selected, superseded]` — a PR can pass the merge gate while the permanent graph asserts the market is *not* covered, or vice versa. Fix: the selector should ask whether B will pin gate/handler parity with an actual test in the required set, and how a future contributor is prevented from editing one definition without the other.

- **A dedicated `patch-comparison` waiver is finer-grained than C's overload — a genuine compliance *strength* — but B does not pin who may sign it or that it cannot be reused across markets.** Adding `patch-comparison` to `checks.yaml` (line 80) and waiving via `waives -> patch-comparison` (line 64) means a patch-market exception no longer also waives evidence provenance (the win over C). But the `override` node only requires `reason, approved_by, expires` (node-types.yaml:47); like every existing gate, `spec:patch-gate` will honour any added override whose `expires >= today` (the `toDateString` / `expires >= input.today` pattern, gate.ts:109-118, which B reuses verbatim per line 130). Nothing scopes a `patch-comparison` override to a particular brief/market — so one signed waiver can clear the comparison gate for any multi-patch PR opened before it expires. For a Class 3 control gating privacy/compliance-sensitive change selection, a market-scoped, single-use waiver would be the defensible record. Question: does B's `patch-comparison` override name the brief/market it excuses, or is it a blanket time-boxed pass that the next contributor can ride?

- **No concern that B leaves any market-resolution event un-recorded.** Because B runs `patch-comparison.yml` on **every** PR (`on: pull_request`, no path filter, modelled on `pr-evidence.yml` lines 6-7 which run unconditionally and decide scope in-job), B does not inherit C's specs-only-PR coverage hole, and because the merge gate is a separate always-on check it does not inherit A's red-WIP-forces-overrides problem. On the pure axis of "is every multi-patch merge a recorded, gated, waivable event," B's mechanism is the soundest of the three — this finding is an explicit clean bill on coverage completeness, distinct from the real PR->mapping and two-places-drift findings above.

## Critique (reliability-ops)

Candidate B realises the merge gate as a new diff-aware `spec:patch-gate` subcommand (modelled on `tools/gate.ts`/`evaluateGate`) behind a dedicated `patch-comparison.yml` workflow with its own `patch-comparison` named check; `spec:validate` stays pure. The contract says fail-closed but then qualifies "(treat as ungated only when there is provably ≤1 competitor)" — which is fail-*open* for the ambiguous case. So on no-match the gate must choose: fail-closed (red every PR it cannot map, including unrelated PRs, since `patch-comparison.yml` runs `on: pull_request` with no path filter like its siblings) or fail-open (let an unmapped multi-patch PR merge ungated — defeating the whole gate). 2. **Two homes for "what is a live competitor" is a genuine drift-into-incident surface — the count that reds CI and the count that validate enforces can disagree silently.** B puts the merge gate in `tools/patch_gate.ts` and the structural coverage rule in `tools/handlers/*` (Scope, Trade-offs "−"). If they drift (one excludes a status the other includes), you get the worst observability outcome: `spec:validate` green but `patch-comparison` red (or vice-versa) on the same graph, and an on-call operator cannot tell from the two exit codes which count is "right." Fix: make the single shared live-competitor helper a contract-level invariant with an enforced parity test, so the gate and the validate rule provably cannot diverge.

3. **The diff/git surface inherits `gitdiff.ts`'s shallow-checkout and base-resolution fail-modes; the contract must carry those forward, not just reuse the happy path.** `resolveBase()` (gitdiff.ts lines 25-48) throws when `GATE_BASE` is unset *and* `git merge-base origin/HEAD HEAD` cannot resolve (shallow checkout). B's `patch-comparison.yml` must therefore set `fetch-depth: 0` and pass `GATE_BASE: ${{ github.event.pull_request.base.sha }}` exactly as `pr-evidence.yml` does (lines 13-14, 47) — otherwise the gate *throws* (CI infra-error red, not a clean gate verdict) on every PR. The contract names modelling on `pr-evidence.yml` but does not pin `fetch-depth: 0`. This is a real operability checklist item: a diff-aware gate that A (a pure graph rule) never needs. Fix/question: confirm `patch-comparison.yml` carries `fetch-depth: 0` + `GATE_BASE`, and that a `resolveBase` throw is surfaced as a distinguishable infra failure, not conflated with "market unresolved."

4. **Positive on the core axis: blast radius and recovery are well-contained.** Unlike A, B keeps `spec:validate` green mid-market (so an open market does not red unrelated spec PRs), gives the gate real PR context (it fires only on the PR actually merging the winner), and provides a *dedicated* `waives → patch-comparison` override (checks.yaml-registered) so an operator can wave exactly the patch-market gate without also waving `pr-evidence` — the right blast-radius scoping for the recovery lever. The status-blind-count trap is named and mitigated (Risks) and the override-expiry path reuses `toDateString` + the `expires >= today` check verbatim from `gate.ts` (lines 42-54, 114), so expiry semantics are battle-tested. Net: my only real reliability concerns are findings 1-3 above; the containment design itself is sound.
