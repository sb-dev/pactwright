This intent (`intent-patch-market-synthesis-3b1e`, class 3) is multi-surface — schema,
validator, four commands, `CLAUDE.md`, and CI. It is a **Class 3** change, so this market
carries three candidate contracts (this is candidate **C** of three). All three share an
identical common core and differ on one axis: **where the multi-patch merge gate lives.**
Candidate **C** does **not** add a new gate or workflow at all — it **folds the
multi-patch requirement into the existing `spec:gate`/`pr-evidence` merge gate** that
already runs on every code PR. This is the maximal-reuse candidate; it deliberately
**deviates from the intent's literal "New workflow patch-comparison.yml"** and owns that
gap honestly, the way Phase-7's `registry-driven` candidate openly did not create the nine
named critic files the intent enumerated.

## Problem interpretation

The patch market itself (per-lane candidate patches → one `comparison` → human
`/select-patch`; within-lane synthesis via `synthesizes`; cross-lane combination is
integration; judged in lane isolation) is **identical in all three candidates**, as is the
irreducible common core — the same as candidate A
(`contract-patch-market-validate-authoritative-9d41`); see its Problem interpretation for
the full text. Restated in brief:

- **Schema (`node-types.yaml`).** New `patch` node (`required_fields [id, type, title,
  status, branch, strategy, created]`, `status_values [candidate, selected, superseded]`,
  `requires_body: true`); optional boolean `patch_market` flag on `brief`.
- **Schema (`edge-types.yaml`).** New `competes-for` (`patch → brief`) and `synthesizes`
  (`patch → patch`); widen `compares.target`/`selects.target` to `[contract, patch]` (list
  targets already supported by `edge_endpoint_types`; the intent's "`selects` already
  allows decision → patch" is inaccurate — widening it is in-scope work).
- **Commands (`.claude/commands/`).** `/propose-patches`, `/compare-patches`,
  `/synthesize-patches`, `/select-patch` with exactly the intent's behaviours, carrying the
  Phase-6 scope-integrity rules into patch review.
- **CLAUDE.md.** Per-lane patch-market prose; the by-class row; within-lane synthesis vs
  across-lane integration; scope-integrity in patch review.
- **Two intent-mandated `spec:validate` extensions (structural).** `synthesis_parentage`
  (≥2 `synthesizes` edges) and selected-patch comparison coverage — pure `Finding[]`
  handlers after `edges-references-resolve`, counting only non-superseded competitors.

What distinguishes **candidate C**: the **merge gate** is implemented by **extending the
existing `tools/gate.ts` (`evaluateGate`)** — the gate behind `pr-evidence.yml` that
already blocks every code PR lacking evidence-or-override — with one additional clause: if
the PR implements a brief that has >1 live `competes-for` edge, the gate **also** requires
a `comparison` covering those patches plus a `selects` decision, else it fails. Override is
**already wired**: the existing `override` + `waives → pr-evidence` path suppresses the
whole gate, the new clause included. **No `patch-comparison.yml`, no new named check, no
new subcommand** — one merge gate enforces both evidence-or-override and patch-comparison.

## Scope

- **Common core** — schema, four commands, CLAUDE.md, the two structural validate rules
  (identical to candidates A and B).
- **`tools/gate.ts` (extend `evaluateGate`)** — add the multi-patch clause: derive the
  brief the PR implements (graph-first via the `patch` node whose `branch` == head →
  `competes-for` brief; branch-name fallback); if that brief has >1 live `competes-for`
  edge, require a `comparison` covering them + a `selects` in the added/live graph. The
  clause reuses the existing `gitdiff.ts` helpers, the `override`/`waives → pr-evidence`
  suppression, and `toDateString` already in the module.
- **`tools/validator.ts` + `tools/handlers/*` + `validation-rules.yaml`** — only the two
  structural rules.
- **`CLAUDE.md`** — additionally documents that the multi-patch merge gate is enforced by
  the existing `pr-evidence` check (not a separate workflow), so a reader is not left
  expecting a `patch-comparison.yml`.
- **No new workflow file; no `checks.yaml` change** (reuses `pr-evidence`).
- **Capability ownership, this PR** — per the Phase-8 capability-wiring flow.

## Non-scope

- **No `patch-comparison.yml` workflow and no `patch-comparison` named check.** This is C's
  defining choice and the source of its literal-fidelity gap (below); candidates A and B
  both create the named workflow.
- **No merge-gate logic in `spec:validate`** (candidate A's mechanism, not adopted).
- **No PR-state tracking in the graph;** no across-lane synthesis; no auto-selection; no
  release/`includes` work.

## Trade-offs

- **+** Maximal reuse — **one** merge gate (`pr-evidence`) enforces evidence-or-override
  **and** patch-comparison; zero new workflow files; smallest CI surface of the three.
- **+** Override support is **already there** — `waives → pr-evidence` suppresses the new
  clause with no new check, no new wiring.
- **+** No PR→brief-mapping code beyond what the gate already does to find the implemented
  brief; the new logic is a small clause in a module that already maps PRs to the graph.
- **−** **Deviates from the intent's literal "New workflow patch-comparison.yml"** and from
  acceptance criterion 3's wording ("blocked by `patch-comparison.yml`"). The merge IS
  blocked — by `pr-evidence`, not by a file of that name. This is a real literal-fidelity
  gap (the Phase-7 `registry-driven` archetype), stated upfront and owned, not hidden. The
  reviewer must decide whether the intent's value ("a multi-patch PR without comparison
  cannot merge") is met by *the merge being blocked* or specifically by *a workflow named
  patch-comparison.yml*.
- **−** **Overloads `pr-evidence`'s single responsibility** — it now gates two distinct
  concerns (evidence provenance AND patch-market resolution); a future reader of one gate
  must know it enforces both, and a `waives → pr-evidence` override now waives **both** at
  once (coarser than a dedicated `patch-comparison` waiver).
- **−** **Coverage hole:** `pr-evidence.yml` skips specs-only PRs (it runs only when code
  changes exist). A patch-proposal step that lands only graph nodes (no code) would not hit
  the gate. **Owned** in Risks with a mitigation, but it is a genuine gap A (a validate
  rule, always run) and B (a dedicated always-on workflow) do not have.

## Risks

- **Specs-only-PR bypass (C's defining risk).** A PR that adds patch nodes/edges but no
  code skips `pr-evidence`, so the multi-patch clause never runs. **Mitigation:** the
  winner's branch always carries the implementation (code), so the *merge that matters* —
  the one bringing the chosen patch's code to main — does hit the gate; document that
  graph-only PRs are not the gated artifact. The reviewer should weigh whether that
  reasoning is airtight.
- **Overloaded override.** `waives → pr-evidence` now waives patch-comparison too.
  **Mitigation:** document the coarser semantics; if finer control is later needed, this is
  the upgrade path to candidate B's dedicated check.
- **Status-blind competitor count (the recurring trap).** **Mitigation:** count only
  non-superseded patches as competitors in both the gate clause and the structural rules;
  share one helper; test parity.
- **PR→brief mapping inside the gate.** **Mitigation:** graph-first mapping with
  branch-name fallback, fail-closed on no match; unit-test all three paths.

## Acceptance examples

1. **(Two real candidate patches → compare → select, fully traced.)** As in all candidates:
   propose two patches, `/compare-patches` writes one `comparison` with `compares` edges to
   both, `/select-patch <winner>` records the `decision` + `selects`, loser superseded;
   chain fully walkable.
2. **(Multi-patch brief whose impl PR skips comparison is blocked.)** The winner's
   implementation PR for a brief with two live `competes-for` edges and **no**
   `comparison`/`selects` makes `spec:gate` (the `pr-evidence` check) **exit non-zero**.
   Recording the `comparison` + `selects` makes it pass. The merge is blocked — by
   `pr-evidence`, satisfying the intent's *spirit*; C explicitly does **not** provide a
   workflow literally named `patch-comparison.yml` (its owned deviation).
3. **(Override.)** An added `override` + `waives → pr-evidence` (non-expired) makes the
   gate pass with the market unresolved; this simultaneously waives the evidence clause —
   the owned coarser semantics.
4. **(`spec:validate` stays green mid-market.)** While the market is open, `spec:validate`
   is green (only the structural rules exist there); the merge gate is the existing
   `pr-evidence` check.
5. **(Synthesis.)** As in all candidates: a synthesis `patch` on
   `patch/<brief-slug>/synthesis` with `synthesizes` edges to its two parents and a
   `competes-for` to the same brief; selecting it supersedes the parents; one
   `synthesizes` edge fails `synthesis_parentage`.

## Critique (spec)

**1. The specs-only-PR coverage hole is not a minor caveat — it can leave the intent's headline acceptance criterion silently unmet.** `pr-evidence.yml`/`spec:gate` (`evaluateGate`) gates on *added evidences edges* and only runs when code changes exist; C itself admits it "skips specs-only PRs." But the patch-market mutations are exactly the graph-only steps: `/propose-patches` creates `patch` nodes + `competes-for` edges (graph only), `/compare-patches` creates the `comparison` node + `compares` edges (graph only), `/select-patch` creates the `decision` + `selects` (graph only). C's defence is that "the winner's branch always carries the implementation," so the *merge that matters* hits the gate. But the intent's acceptance #3 is "a multi-patch brief whose **PR skips the comparison node** is blocked" — and a PR that adds patch nodes and edges but no code is precisely a PR that can skip the comparison node while *not* hitting C's gate. **Fix/question:** name the exact PR the gate fires on. If it is only the winner's code-bearing merge, then a graph-only PR that records a malformed/incomplete market merges green, and the intent's "blocked otherwise" guarantee is weaker than A's (always-run validate rule) or B's (always-on dedicated workflow).

**2. Folding the clause into `evaluateGate` adds a PR→brief mapping that the existing function has never needed — and corrupts a gate whose current logic is brief-agnostic.** Today `evaluateGate` (gate.ts lines 63-126) decides pass/fail purely from `addedEdgeIds`/`addedNodeIds` and an `evidences→brief→decomposes→approved-contract` walk; it never parses the head branch and never identifies "which brief this PR implements." C's new clause requires deriving that brief (graph-first via `patch.branch == head`, branch-name fallback). Bolting branch-name parsing into the one pure function that today needs no git head at all is the largest behavioural change to a load-bearing, already-tested module — and `evaluateGate`'s unit tests assert a *pure* `(spec, input)` contract with `input` = added-id sets + `today` (no branch). **Question for the selector:** does C extend `GateInput` with the head branch (changing the tested signature of the repo's central gate), and is regressing `pr-evidence` for *every* code PR an acceptable blast radius for a patch-market feature?

**3. The overloaded `waives → pr-evidence` override is coarser than the intent's model and is a real authorization regression.** C reuses the existing `waives → pr-evidence` path (checks.yaml line 12; `gate.ts` `PR_EVIDENCE_CHECK`) so "override support is already there." But that means one `override` node now waives **two unrelated concerns** at once: evidence-provenance AND patch-market resolution. A reviewer who legitimately waives a missing-evidence requirement on a routine PR would *simultaneously* suppress the patch-comparison gate if that PR happened to be a multi-patch brief — and vice versa. The intent and A/B model a *distinct* `patch-comparison` check precisely so a market waiver is scoped to the market. C's own Trade-offs own this as "coarser," but understate that it is a silent broadening of every existing `pr-evidence` waiver's authority. **Fix:** at minimum require C to document, in `CLAUDE.md`, that every `waives → pr-evidence` now also waives patch-comparison — and ask the selector whether a single shared waiver is acceptable for a Class-3 (production-sensitive-process) gate.

**4. (Shared core + literal-fidelity) C drops the intent's explicitly-named `patch-comparison.yml`, and inherits the unaddressed `selects`-widening collision.** The intent's "Validation and enforcement" section literally specifies "New workflow `patch-comparison.yml`" and acceptance #3 names it; C deliberately ships *no* such workflow and *no* `patch-comparison` check. C owns this as the Phase-7 `registry-driven` archetype — an honest deviation, not a hidden one — so this is a fidelity *question*, not a defect: does the selector read the intent's value as "the merge is blocked" (C satisfies) or as "a workflow named patch-comparison.yml exists" (only A/B satisfy)? Separately, and exactly as in A and B, C does not address that widening `selects.target` to `[contract, patch]` (edge-types.yaml line 13) makes the existing `class_market_quorum.ts` (lines 27-43, hard finding when `intentsForContract` returns `[]`) and `comparison_required.ts` (line 61) iterate over patch-typed `selects` targets. **Fix:** the shared core must guard both existing handlers to `type === "contract"` before treating a `selects` target as a contract, or every patch selection turns the graph red on a rule unrelated to the patch market.

## Critique (product)

Reviewed on the product / user-value axis only.

- **C's "specs-only PR bypass" is not a corner case — it is exactly the artifact the intent's headline acceptance describes, so C can fail the one scenario it must pass.** I verified the coverage hole against the real workflow: `pr-evidence.yml` lines 24-32 explicitly skip when the diff touches only `specs/` or `docs/` (`grep -qvE '^(specs|docs)/'` → `skip=true`). C folds the multi-patch clause into that exact gate (body lines 46-53, 100-104). Now read the intent's acceptance (lines 72-73): "a multi-patch brief whose PR **skips the comparison node** is blocked." The most natural way a contributor skips the comparison is to push the patch nodes/edges *without* the resolving `comparison`/`selects` — a **specs-only change** — which `pr-evidence.yml` skips entirely, so C's clause never runs and the merge is **not** blocked. C's own defence (Risks lines 108-113) is that "the winner's branch always carries code, so the merge that matters hits the gate," but that reasoning quietly redefines the user's guarantee from "a multi-patch state cannot merge unresolved" to "the eventual code merge is gated" — and it assumes the graph and the code always land in the same PR, which the intent never requires. **Fix / question for the selector:** construct the exact acceptance scenario (a multi-patch brief, comparison skipped) as a specs-only PR and ask whether C blocks it. If it does not, C fails the intent's headline acceptance criterion as literally stated — this is the single strongest product argument against C.

- **Overloading `pr-evidence` makes the override coarser in a way that silently weakens a *different* guarantee — the user waiving one thing waives two.** C reuses `waives → pr-evidence` for the new clause (body lines 50-53, 86-87, 135-137). I confirmed the existing semantics in `gate.ts`: an unexpired `override` + `waives → pr-evidence` passes the **whole** gate (lines 96-121). So under C, a human who waives the patch-comparison requirement *also* waives the evidence-provenance requirement in the same stroke — and vice versa. From the product/value angle this is a real regression in control granularity: a reviewer who wants to say "yes, merge this single-patch change without the usual evidence edge, just this once" cannot do so without *also* blessing any unresolved patch market in that PR, and a reviewer waiving an unresolved market unintentionally blesses missing evidence. C owns this (lines 98-100, 114-116) but frames it as a minor con; on the value axis it means the override audit trail can no longer tell you *which* guarantee a human consciously relaxed. **Fix / question:** ask C how a future reader of a `waives → pr-evidence` override distinguishes "human deliberately skipped patch comparison" from "human deliberately skipped evidence" — if the graph can't say, the override record loses the provenance value CLAUDE.md's whole "history must remain readable" rule (rule 3) exists to protect.

- **C deviates from the intent's literally named deliverable, and the user-facing cost is a documentation/expectation mismatch the feature can't paper over.** The intent says "New workflow patch-comparison.yml" (line 61-64) and acceptance names it (line 72-73). C ships **no such file and no `patch-comparison` named check** (body lines 75-77, 90-96). C handles this honestly by having CLAUDE.md document that the gate lives in `pr-evidence` (lines 67-69). But on the product axis the issue is the *next contributor*: they read the intent, the acceptance, or any prose referencing `patch-comparison.yml`, go looking for that workflow/check, and find nothing — the mental model the documentation builds does not match the artifact. C's Trade-offs (lines 90-96) correctly hand the selector the real question. **Fix / question for the selector:** is the intent's value "a multi-patch PR cannot merge unresolved" (mechanism-agnostic), in which case C's deviation is acceptable — or is part of the value a *named, discoverable* `patch-comparison` check (B's strength), in which case C drops a use-case the intent explicitly asked for?

- **Shared-core (clean on my axis): the reuse instinct genuinely lowers the contributor's cognitive load — *if* the coverage hole is closed.** Stripped of the bypass problem, C's core product virtue is real: one merge gate the team already understands (`pr-evidence`), zero new workflows to learn, the existing override path (body lines 84-89). For the *user* who has to operate this system day to day, fewer distinct gates is a genuine value. **No concern on the reuse-simplicity axis itself** — the concern is entirely the specs-only bypass (first finding) and the override coarsening (second). If C closed the bypass (e.g. by also running on graph-only PRs), its simplicity would be a strong user-value; as written, the bypass undercuts it.

## Critique (architecture)

**1. ## Critique (security-privacy)
