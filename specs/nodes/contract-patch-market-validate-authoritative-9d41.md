This intent (`intent-patch-market-synthesis-3b1e`, class 3) is itself multi-surface — it
migrates the schema (`specs/schema/node-types.yaml`, `specs/schema/edge-types.yaml`),
extends the validator (`tools/validator.ts` + new `tools/handlers/*.ts`), adds four
commands under `.claude/commands/`, edits `CLAUDE.md`, and adds CI. Under the routing
table that is a **Class 3** change, so this proposal market carries three candidate
contracts (this is candidate **A** of three). All three share an identical common core
and differ on exactly one axis: **where the multi-patch merge gate lives** — the
requirement that "a brief with >1 `competes-for` edge — merging its implementation PR
requires a comparison node + a `selects` decision; fail otherwise; overridable via an
override node." Candidate **A** realises that gate as a **third `spec:validate` rule** (a
red graph blocks the merge); `patch-comparison.yml` is a thin workflow that runs
`spec:validate`.

## Problem interpretation

The intent asks for a per-lane patch market: for one lane brief you implement the same
brief N ways on N branches, open N draft PRs, record each as a `patch` node that
`competes-for` the brief, compare them in one `comparison` node, and let a human
`/select-patch` one winner (losers superseded, branches kept, PRs closed). Within-lane
combination is a `synthesis` patch with `synthesizes` edges to its parents; across-lane
combination is integration (Phase 8), never synthesis. The market is judged **in lane
isolation** — cross-lane fit is integration's job, never patch comparison's.

The intent is unusually prescriptive: it dictates the schema, the four commands, the
CLAUDE.md prose, and **two** explicit `spec:validate` extensions. Those are the
irreducible common core, identical across all three candidates:

- **Schema — `specs/schema/node-types.yaml`.** Add node type `patch` with
  `required_fields [id, type, title, status, branch, strategy, created]`,
  `status_values [candidate, selected, superseded]`, `requires_body: true` (body = the
  patch's evidence summary). Add an optional boolean `patch_market` flag to `brief` (not
  in `required_fields`; set `true` by `/propose-patches`; unset means no market —
  modelled on the optional `lane` precedent). `nodes-status-in-enum` constrains
  `patch.status` for free; `nodes-required-fields` + `requires_body` cover presence and a
  non-empty body for free.
- **Schema — `specs/schema/edge-types.yaml`.** Add `competes-for` (`source: patch`,
  `target: brief`) and `synthesizes` (`source: patch`, `target: patch`). Widen
  `compares.target` from `contract` to `[contract, patch]` and `selects.target` from
  `contract` to `[contract, patch]`. **No handler change is needed**: `edge_endpoint_types`
  already enforces a list target (the existing `flags` edge uses `target: [evidence,
  capability]`), so union-target membership is free. (The intent's claim that `selects`
  "already allows decision → patch" is inaccurate — today it is `decision → contract`
  only; widening its target is the in-scope work that delivers the intended behaviour,
  not a scope change.)
- **Commands — `.claude/commands/`.** `/propose-patches <brief-id> <n> <strategy-list>`
  (brief must be a lane brief or single brief; per strategy create branch
  `patch/<brief-slug>/<strategy>`, run the implementation agent with the brief PLUS an
  injected strategy directive, open a draft PR, create a `patch` node carrying branch,
  strategy and an evidence summary with a `competes-for` edge to the brief, set
  `patch_market: true`). `/compare-patches <brief-id>` (per candidate branch run
  `contract-reviewer` and `/detect-drift`; invoke graph-maintainer to author one
  `comparison` node scoring contract fit, scope control, simplicity, test quality, drift
  risk and rollback safety, with `compares` edges to each candidate patch; end by asking
  for a human decision; never select). `/synthesize-patches <brief-id> <patch-id-list>
  <instruction>` (create a synthesis `patch` (status `candidate`) on branch
  `patch/<brief-slug>/synthesis`, combining the named patches per the human instruction;
  add `synthesizes` edges to each parent and a `competes-for` edge to the **same** lane
  brief; carry an evidence summary). `/select-patch <patch-id> <rationale>` (author a
  `decision` with a `selects` edge to the winner; set losing patches — including the
  parents of a selected synthesis patch — to `superseded`, close their draft PRs, keep
  their branches; the winner's branch proceeds to `/prepare-evidence`; apply the
  scope-integrity rules — if comparison exposed the contract or brief was wrong,
  supersede the brief, capture a follow-up intent, or return to human approval rather
  than widening scope inside the winner).
- **CLAUDE.md.** The patch market runs per lane brief; patches compete within one lane;
  patch comparison judges that lane in isolation; cross-lane fit is judged at integration,
  never in patch comparison. Patch market by class: Class 0–1 a single patch, no market;
  Class 2 optional per brief; Class 3 available per lane. Within-lane synthesis is a
  synthesis patch with `synthesizes` edges; across-lane combination is integration. Patch
  review carries the Phase-6 scope-integrity rules.
- **Two intent-mandated `spec:validate` extensions (structural, in every candidate).** (1)
  `synthesis_parentage` — a synthesis patch (any patch with ≥1 `synthesizes` edge) must
  carry `synthesizes` edges to **≥2** parent patches, else a finding. (2) selected-patch
  comparison coverage — a `selects`-edged `patch`'s brief must have a `comparison` node
  whose `compares` edges cover the patches that `competes-for` that brief (a set/coverage
  traversal modelled on `tools/handlers/comparison_required.ts`). Both are pure
  `(rule, spec) => Finding[]` handlers registered in the `HANDLERS` map, listed after
  `edges-references-resolve`, counting only non-superseded competitors and defensively
  skipping unresolved endpoints.

What distinguishes **candidate A**: the **merge gate** — "a brief with >1 `competes-for`
edge needs a comparison + a `selects` before its implementation can merge" — is realised
as a **third `spec:validate` rule** (`patch_market_resolved`): a brief with >1 live
`competes-for` edge is *unresolved* until a `comparison` covers those patches AND a
`selects` decision points at one of them; while unresolved the graph is **red**.
`patch-comparison.yml` is a thin workflow that runs `spec:validate` (it adds no
independent logic). Override is handled by making this one rule waiver-aware: a
`waives → patch-comparison` edge from a non-expired `override` suppresses the finding.
The graph becomes the single source of truth for every patch-market invariant.

## Scope

- **`specs/schema/node-types.yaml`, `specs/schema/edge-types.yaml`** — the schema
  migration above (patch node, `patch_market` brief flag, `competes-for`/`synthesizes`
  edges, widened `compares`/`selects` targets).
- **`.claude/commands/`** — the four commands above.
- **`CLAUDE.md`** — the patch-market prose, the by-class row, synthesis-vs-integration,
  scope-integrity in patch review.
- **`tools/handlers/` (three new handlers) + `tools/validator.ts` (register) +
  `specs/schema/validation-rules.yaml` (declare, after `edges-references-resolve`)** — the
  two structural rules AND candidate A's distinguishing third rule
  `patch_market_resolved` (multi-patch brief unresolved ⇒ red, waiver-aware). All scoped
  to live (non-superseded) nodes.
- **`.github/workflows/patch-comparison.yml`** — a thin workflow that runs
  `pnpm spec:validate` (modelled on `spec-validate.yml`); blocking comes from the red
  graph, not independent gate logic.
- **`specs/schema/checks.yaml`** — add the `patch-comparison` named check so a
  `waives → patch-comparison` override resolves.
- **Capability ownership, this PR.** The diff touches `tools/**`, `.claude/commands/**`,
  `.github/**`, `specs/schema/**` and `CLAUDE.md`; a diff touching unowned paths is
  resolved in the same PR per the Phase-8 capability-wiring flow.

## Non-scope

- **No diff-aware patch gate.** The merge gate is a graph-state validate rule, not a
  `gate.ts`/`check-diff`-style diff consumer. This is A's defining choice and is candidate
  B's mechanism, deliberately not adopted here.
- **No PR-state tracking in the graph.** A patch node records `branch`, `strategy` and an
  evidence summary; the draft PR's open/closed state lives in git/GitHub, not the graph.
- **No across-lane synthesis.** `/synthesize-patches` combines only patches competing for
  the **same** lane brief; cross-lane combination is `/integrate` (Phase 8).
- **No auto-selection.** `/compare-patches` never selects; selection is a human `decision`.
- **No new release/`includes` work.**

## Trade-offs

- **+** Single source of truth — every patch-market invariant (structural AND the merge
  gate) lives in `spec:validate`; the graph alone tells you whether a market is resolved.
- **+** Fully offline-testable via synthetic node fixtures; no PR context, branch-name
  parsing, or GitHub state needed to test the gate.
- **+** Matches the winning precedent — Phase 3's `ci-gate-spec-tool` and Phase 6's
  `work-class-validate-invariant` both pushed enforcement into `spec:` tooling with thin
  CI on top.
- **+** No PR→brief mapping problem: the rule reads `competes-for`/`compares`/`selects`
  edges directly, immune to branch renames.
- **−** `spec:validate` must become **override-aware** for this one rule (it is otherwise
  diff- and override-agnostic) — an architectural impurity; the waiver-suppression branch
  is logic validate has never carried.
- **−** A half-built market (patches proposed, not yet compared/selected) makes the graph
  **red**, so there is no green WIP graph mid-market — the winner's branch cannot show a
  green tree until selection is recorded. (Arguably correct — the market is genuinely
  unresolved — but it couples "graph well-formed" to "market finished.")
- **−** `patch-comparison.yml` nearly duplicates `spec-validate.yml`; the dedicated
  workflow the intent names earns its keep only as a separately-named required check.

## Risks

- **Status-blind competitor count (primary trap — a prior critique caught exactly this on
  the quorum rule).** If `patch_market_resolved` or the coverage rule counts competitors
  status-blind, a superseded patch could be mistaken for a live competitor. **Mitigation:**
  count only **non-superseded** patches as live competitors; mirror `comparison_required.ts`'s
  exclusion of superseded targets; unit-test a market with one superseded loser.
- **Override-awareness leaking into validate.** **Mitigation:** confine the waiver check to
  the single `patch_market_resolved` handler; keep the two structural rules pure; document
  the exception in `validation-rules.yaml`.
- **Red WIP blocks routine pushes.** An open market is red, so pushing the winner's branch
  for review before `/select-patch` fails CI. **Mitigation:** the override path exists for
  genuine WIP; document `/select-patch` as the normal resolution, not the override.
- **Rule ordering / unresolved endpoints.** **Mitigation:** list the three rules after
  `edges-references-resolve` and defensively skip absent endpoints.

## Acceptance examples

1. **(Two real candidate patches → compare → select, fully traced.)** A lane brief gets
   `/propose-patches <brief> 2 strategyA,strategyB` → two `patch` nodes each
   `competes-for` the brief, `patch_market: true` set. `/compare-patches <brief>` writes
   one `comparison` with `compares` edges to both. A human `/select-patch <winner>` writes
   a `decision` with a `selects` edge to the winner, supersedes the loser; the chain
   intent→…→brief→patches→comparison→decision is fully walkable.
2. **(Multi-patch brief unresolved ⇒ red — the merge gate.)** A brief with two live
   `competes-for` edges but **no** `comparison`/`selects` makes
   `node_modules/.bin/tsx tools/spec.ts validate` **exit non-zero** on
   `patch_market_resolved`. Adding the comparison and a `selects` flips it green;
   `patch-comparison.yml` (running `spec:validate`) blocks the merge until then.
3. **(Override.)** With the market unresolved, an `override` node + a non-expired
   `waives → patch-comparison` edge suppresses the finding and the graph is green.
4. **(Synthesis.)** `/synthesize-patches <brief> p1,p2 "<instruction>"` creates a
   synthesis `patch` on `patch/<brief-slug>/synthesis` with `synthesizes` edges to p1 and
   p2 and a `competes-for` edge to the same brief; `/select-patch <synthesis>` supersedes
   p1 and p2. A synthesis patch with only **one** `synthesizes` edge fails
   `synthesis_parentage`.
5. **(Lane isolation.)** A patch competing for lane brief X is compared only against other
   patches competing for X; a comparison never spans two briefs.

- **Acceptance trace:** the end-to-end propose→compare→select run on one real lane brief

## Critique (spec)

**1. Making `spec:validate` waiver-aware breaks the override architecture this rule set was built on — and the contract names the impurity but underplays it.** Every override today is consumed by a *diff-aware* gate (`tools/gate.ts` clause (b): `evaluateGate` checks `addedNodeIds`/`addedEdgeIds` for a `waives → pr-evidence` override and `toDateString(node.data["expires"]) >= today`). No `spec:validate` handler reads an `override` node or a `waives` edge at all — the whole `HANDLERS` map (`unique_field`…`coverage_coherence`) is pure graph-shape. Candidate A's `patch_market_resolved` would be the *first* validate rule that suppresses its own finding on the presence of a node; but `runValidation` hands handlers `(rule, spec)` with **no `today`** (compare `evaluateGate`'s `GateInput.today`, supplied by `runGate` from `new Date()`). So the handler has no run-date to compare `expires` against and must either invent one (non-deterministic validate output — a fixture-stability disaster) or ignore expiry (an *expired* override silently keeps suppressing forever). **Fix/question for the selector:** require A to specify exactly where the validate handler gets `today`, and confirm whether the team accepts the first impure, date-dependent `spec:validate` rule — or whether expiry-blind suppression is acceptable.

**2. A red-on-open-market gate inverts the lifecycle: it makes `spec:validate` fail on a graph that is correctly mid-market.** A's own Trade-offs concede "no green WIP graph mid-market," but the consequence is sharper than stated. The intent's normal flow (`/propose-patches` creates two `patch` nodes + two `competes-for` edges *before* `/compare-patches` and `/select-patch` ever run) produces, for a real interval, a graph with >1 live `competes-for` and no comparison/selects — which A declares RED. That means `pnpm spec:index && pnpm spec:validate` (CLAUDE.md's mandated post-mutation step) **fails on the very mutation `/propose-patches` is supposed to commit**, and CLAUDE.md says "must not commit on failure." So `/propose-patches` cannot land its own nodes without either an immediate override or a same-PR selection — defeating the multi-step market. Contrast every *other* gate in this graph (`class-market-quorum`, `comparison-required`) which fires only on a *`selects` edge* — i.e. only once a decision exists, never on the open market. **Fix:** key A's rule on the presence of a `selects` edge to a patch (resolved-but-wrong), not on the bare existence of >1 `competes-for` (resolved-but-incomplete), so an open market stays green.

**3. `patch-comparison.yml` that "thin-runs `spec:validate`" cannot be a meaningfully distinct required check — and overrides become coarse-grained.** A's Scope adds `.github/workflows/patch-comparison.yml` "modelled on `spec-validate.yml`" that just runs `pnpm spec:validate`, plus a `patch-comparison` entry in `checks.yaml`. But if blocking comes entirely from the red graph, then the *existing* `spec-validate` check already blocks the merge; the new workflow is a duplicate runner. Worse, A's override is a `waives → patch-comparison` edge, yet the rule it suppresses is part of `spec:validate` — so a reviewer waiving `patch-comparison` does **not** make `spec-validate` (the check that actually goes red) pass. A would have to wire the handler to read a `waives → patch-comparison` *or* `waives → spec-validate`, and `checks.yaml` (lines 8-16) lists `spec-validate` as a check that today waives the *whole* validate run. **Question for the selector:** does waiving `patch-comparison` actually un-red the graph, or does the operator still face a red `spec-validate` they must *separately* waive — re-introducing exactly the two-waivers coarseness A claims to avoid?

**4. (Shared core) The "selected-patch comparison coverage" rule cannot reuse `comparison_required.ts` as drawn, and the intent's `selects` claim is wrong in a way that breaks `class_market_quorum`.** A correctly flags that the intent's "`selects` already allows decision → patch" is false (edge-types.yaml line 13-14 is `decision → contract` only). But widening `selects.target` to `[contract, patch]` has an unaddressed downstream effect: `class_market_quorum.ts` (lines 27-43) fires on *every* `selects` edge, calls `intentsForContract(spec, target)`, and emits a hard finding when that returns `[]`. A patch never has a `proposes` edge, so `intentsForContract` returns `[]` for a selected patch → the existing quorum rule reports `selects edge … targets contract X which proposes no intent` **for every patch selection**. Likewise `comparison_required.ts` loops all `selects` edges (line 61), reads `contract.data["created"]` and `intentsForContract` on what is now a patch. A's claim that the new coverage rule is "modelled on `comparison_required.ts`" doesn't address that the *existing* two rules must be taught to skip patch-typed `selects` targets. **Fix:** require A to guard `class_market_quorum` and `comparison_required` to act only when the `selects` target's `type === "contract"`, and confirm the new patch-coverage rule is a *separate* handler, not a widening of `comparison_required`.

## Critique (product)

Reviewed on the product / user-value axis only (does the candidate solve the user problem the intent states, what use-case it silently drops, and is its value claim falsifiable). Mechanism/architecture purity is another critic's axis except where it changes user-observed behaviour.

- **The "red WIP graph" inverts the user's actual workflow and makes the normal path feel like a failure.** A's distinguishing choice (`patch_market_resolved` as a `spec:validate` rule, body lines 90-98, 148-151) means the *expected* state of a freshly-proposed market — two `competes-for` edges, no `comparison`/`selects` yet — is a **RED graph**. But the intent's own workflow is sequential: `/propose-patches` (line 24-29) happens *before* `/compare-patches` (line 30-34) which happens before `/select-patch` (line 42-48). So between propose and select — the entire window in which the human is doing the comparison work the feature exists to support — every push of the winner's branch fails CI, and the contributor cannot show a green tree for review until selection is already recorded. A's own Risks section admits this ("Red WIP blocks routine pushes", line 165-167) and waves it off with "the override path exists for genuine WIP." That is a product anti-pattern: it pushes contributors toward routinely cutting `override` nodes to make ordinary in-progress work green, training the team to treat the human waiver — the graph's strongest safety signal — as a routine un-stick button. Concrete failure: a reviewer sees a red tree on an open market, can't tell "market genuinely unresolved" from "infra broke," and the team learns to reflexively waive. **Fix / question for the selector:** does A make the *common* state (market open, comparison in progress) red, and if so, is "every open market is a CI failure until a human selects" the experience we want, versus B's design where `spec:validate` stays green mid-market (B body lines 104, 148-150) and only the *merge* is gated?

- **The override is a binary kill-switch, so the value claim "a multi-patch PR cannot merge without comparison" is only falsifiable as far as nobody waives — but A makes waiving the WIP-relief mechanism.** The value the intent is buying (acceptance line 72-73: a multi-patch PR that skips the comparison node is *blocked*) is only real if the block is hard to bypass casually. A routes both "I genuinely want to override the gate" and "I just want my open-market branch to be green while I work" through the *same* `waives → patch-comparison` edge (body line 96-97, 184-185). Once contributors use override for routine WIP relief (previous finding), an override on a market that was *never actually resolved* becomes indistinguishable from a deliberate, justified waiver. After shipping you could not tell from the graph whether a waived multi-patch merge reflected a real decision or WIP-unsticking — the headline value claim becomes unfalsifiable in practice. **Fix / question:** require A to specify, in the command prose, what distinguishes a "WIP override" from a "skip-the-comparison override," or the intent's central guarantee erodes the first time someone overrides to get a green tree.

- **Shared-core: the four commands encode the user-facing market workflow, but nothing in A's enforcement checks that a *human actually compared* — only that nodes exist.** The intent's user problem is "let a human choose among N real implementations on evidence" (intent lines 30-34, 42-48). A's `patch_market_resolved` rule and the coverage rule (body lines 80-94) check structural presence: a `comparison` node exists and `compares` edges cover the live patches, plus a `selects`. A `comparison` node with an empty or rubber-stamped scoring body satisfies the rule identically to a genuine six-axis comparison (the intent's "contract fit, scope control, simplicity, test quality, drift risk, rollback safety", line 32-33). This is the same honest-bound limitation CLAUDE.md records for integration ("a green graph asserts the node exists ... it does not prove ... the verdict is sound"). That is acceptable, but A's Trade-offs oversell it as "single source of truth — the graph alone tells you whether a market is resolved" (lines 136-138, 98): the graph tells you the *shape* is resolved, not that a human exercised judgement. **Fix / question:** confirm A's CLAUDE.md prose states plainly that `patch_market_resolved` green means "comparison node present and covering," not "comparison was substantive," so reviewers do not over-trust a green market.

- **Shared-core (clean on my axis): the corrected `selects` widening is the right user-facing call.** All three candidates correctly flag that the intent's claim "`selects` already allows decision → patch" is inaccurate — `edge-types.yaml` line 13-15 shows `selects` is `decision → contract` only (A body lines 50-53). Widening `selects.target`/`compares.target` to `[contract, patch]` is exactly what lets a human selection of a patch be recorded the same way a contract selection is, preserving one consistent "the human chose X, here's why" provenance shape across both markets. No product concern on this sub-point — it keeps the selection experience uniform.

## Critique (ux)

**1. The "red WIP graph" makes the operator's local feedback loop lie to them about what they did wrong.** Candidate A's defining choice (`patch_market_resolved`, lines 92-98, 150-151) turns a half-built market into a RED graph: the moment `/propose-patches` writes a second `competes-for` edge, `spec:validate` exits non-zero. But `/propose-patches`, per the intent (lines 24-29) and this contract (lines 54-59), itself ends by invoking graph-maintainer, whose universal convention is "regenerate indexes, and validate; commit only on green" (mirrored in `prepare-evidence.md` line 28, `decompose-lanes.md` line 19, `integrate.md` line 24). So the very command that opens the market produces the red graph that blocks its own commit. An operator running the documented happy path sees a validate failure on `patch_market_resolved` and cannot tell whether they mis-ran the command or whether red-is-expected-mid-market — the failure text is identical to a genuine defect. **Fix/question:** does `/propose-patches`'s closing report (which this contract never specifies) explicitly tell the operator "the graph is now intentionally red until /select-patch; this is not an error"? Without that one sentence, the first user of the patch market will read a correct state as a broken one. State whether `/propose-patches` is exempted from the "commit only on green" rule, and if so, where the operator is told the tree is committable-while-red.

**2. The four new commands ship no closing-report contract, so the operator is left without the "what state am I in and what's next" line every existing command guarantees.** Every command I was asked to compare against ends with an explicit operator report: `prepare-evidence.md` line 29 ("report the evidence ID and each updated node's status"), `decompose-lanes.md` line 20 ("report each brief ID and its lane"), `integrate.md` lines 25-27 (ID, draft/final status, sections written, lanes covered vs blocking, verdict, remediation). Candidate A's command descriptions (lines 54-73) reproduce the intent's mechanics verbatim but specify no closing report for any of `/propose-patches`, `/compare-patches`, `/synthesize-patches`, or `/select-patch`. The discoverability cost is sharpest under A precisely because A's red-graph model means the operator most needs to be told, after each step, "your market is unresolved (red); the next command is X." **Fix:** require each new command's spec to name its closing report — minimally `/compare-patches` must report the comparison ID and the still-required `/select-patch`, and `/select-patch` must report which patches went `superseded` and that the graph is now green.

**3. The override path is the operator's only escape from a red tree, yet its discoverability is exactly backwards.** A's override is a `waives → patch-comparison` edge (lines 96-97); the Risks section (lines 165-167) concedes "pushing the winner's branch for review before /select-patch fails CI" and offers "the override path exists for genuine WIP." But an override node is a heavyweight, dated, auditable governance artifact — making it the routine unblock-my-push mechanism trains operators to waive the gate casually, which is the one behavior a class-3 merge gate must discourage. The UX trap: the friction (red local tree, red CI on every push) pushes the user toward the override, and the override is coarse enough that habitual use erodes the gate. **Fix/question:** is there any non-override way to push the winner's branch for review while the market is legitimately mid-flight, or does A force every pre-selection push through a governance waiver? If the latter, A's UX actively incentivizes the anti-pattern it names as a risk.

**4. Shared-core: `/select-patch` inherits the intent's inaccurate `selects` premise, and the contract fixes the schema but not the operator's mental model.** The contract correctly flags (lines 50-53) that the intent's "`selects` already allows decision → patch" is false — today it is `decision → contract` only (confirmed: `edge-types.yaml` lines 13-15). A widens the target, which is right. But on my axis the consequence is unaddressed: `/select-patch` is the first command in the whole system where a `decision`/`selects` pair points at something other than a contract, and the operator's existing intuition (built by `/approve-contract`) is "selects picks a contract." Nothing in A's `/select-patch` spec (lines 67-73) tells the operator how `/select-patch` differs from contract selection — that it supersedes *patches* and *closes draft PRs* (a destructive GitHub side effect) rather than rejecting sibling contracts. **Fix:** `/select-patch`'s spec must call out the PR-closing side effect explicitly and report which draft PRs it closed, so the operator is not surprised by branches/PRs changing state outside the graph.

## Critique (architecture)

**1. The merge gate as a `spec:validate` rule forces the first override-aware handler into a validator that has no run-date and no waiver concept — the impurity is larger than the contract admits.** Every handler registered in `tools/validator.ts`'s `HANDLERS` map has the signature `(rule: Rule, spec: LoadedSpec) => Finding[]`; `LoadedSpec` carries no run date, and `runValidation` (validator.ts:59-86) never injects `today`. ## Critique (security-privacy)

Reviewed from the security-and-privacy axis only (trust boundaries, abuse/injection paths, override/waiver semantics). No personal-data storage surface exists in this change; the relevant boundary is the **merge gate as an enforcement boundary** and the **override as a privilege escalation**.

1. **The override moves a security gate from a diff-aware, same-PR-bound check into `spec:validate`, silently widening who can suppress it and from where.** Candidate A makes `patch_market_resolved` waiver-aware: a `waives → patch-comparison` edge from a non-expired `override` suppresses the RED finding (contract lines 96-97). But `evaluateGate` binds its override clause to `addedEdgeIds`/`addedNodeIds` — the `waives` edge AND the `override` node must both be *added in the same PR* (`tools/gate.ts` lines 100-108). A `spec:validate` rule has no such diff context: it runs over the whole live graph, so a `patch-comparison` waiver authored in *any* prior PR, by *anyone*, and left in the tree continues to suppress the gate on every later market until its `expires` lapses. That is a standing, repo-wide bypass token rather than a one-shot, same-PR waiver. Fix: have `patch_market_resolved` ignore the standing-graph override entirely and require the diff-bound override (i.e. keep the waiver where `evaluateGate` keeps it), or scope the waiver to the specific brief id, not the bare `patch-comparison` check name.

2. **A waiver-aware branch inside `spec:validate` collapses two trust boundaries the codebase deliberately keeps apart, and the contract admits it.** Today `spec:validate` is override-agnostic; only `gate.ts`/`checkdiff.ts` consult `override`/`waives`/`expires` (the `toDateString` + `expires >= today` machinery, `gate.ts` lines 109-120). A's own Trade-offs (line 146-147) call this "an architectural impurity… logic validate has never carried." The security consequence is concrete: the green-graph invariant that the rest of the system (e.g. `class-market-quorum`, `comparison-required`) treats as *unwaivable structural truth* now has, for the first time, a waivable hole. Any future reader or tool that assumes "validate green ⇒ all structural invariants hold" is now wrong for this one rule. Sharp question for the selector: are you willing to make `spec:validate` — the backstop the quorum/comparison rules rely on being unbypassable — carry an expiring waiver branch, and have you confirmed no other rule's "unbypassable backstop" guarantee silently weakens once override-parsing lives in the validator?

3. **Red-WIP coupling creates a denial-of-merge / pressure-to-waive dynamic that trains operators to reach for the override.** A's design makes a half-built market RED (Trade-offs line 150-151; Risks line 165-167): proposing two patches with no comparison yet turns the whole graph red, blocking *every* PR on the branch, not just the unresolved market. The documented escape is "the override path exists for genuine WIP" (line 166). Routinizing override use for ordinary mid-market WIP is exactly how a security waiver becomes background noise — operators learn to add a `waives → patch-comparison` reflexively to get a green tree, and an expiring waiver added "just to push" can outlive the WIP and silently cover a later genuinely-unresolved market (see finding 1). Fix: do not make an open market red; gate only the *merge* of the implementation, so the override is reserved for true exceptions, not routine WIP.

4. **Shared-core, on-axis: the coverage rule must count competitors status-blind-safely or a superseded patch becomes a forged comparison-coverage token.** The selected-patch coverage rule is modelled on `comparison_required.ts`, which counts only live, non-`superseded` targets that also `proposes` the market (lines 48-55). A's Risks (line 158-161) name this trap but the contract text only *promises* the mitigation. Concretely, if the new rule counts a `compares → patch` edge whose target is already `superseded`, a market could be marked "covered" by a dead patch while a live competitor goes uncompared — a real bypass of the comparison requirement. Fix: require the handler to mirror `comparison_required.ts` exactly — a covered patch counts only if it is non-superseded AND `competes-for` the same brief — and unit-test the superseded-loser case as a hard assertion, not a comment.
