This intent is multi-surface — it edits `specs/schema/node-types.yaml`, the `spec:validate` handler set under `tools/handlers/`, `CLAUDE.md`, and the `.claude/commands/` lifecycle commands. Under its own proposed routing table that is the definition of **Class 3**, which is exactly why this proposal market carries three candidate contracts rather than one. This candidate (C) delivers the full fixed common core shared by all three siblings and adds a **diff-aware approval gate** — a new pure `(spec, input) => {pass, reason}` function in the established `tools/gate.ts` / `tools/checkdiff.ts` family, run by a new `spec:` subcommand — to machine-enforce "a Class-3 intent cannot be approved until ≥2 candidate contracts exist."

## Problem interpretation

The intent has two separable obligations. The first is a **schema + routing migration** that every candidate must ship identically: make `class` a required integer field on `intent` and `contract`, add an opaque optional `produced_by` to every node type, backfill `class` onto all pre-existing intent/contract nodes in the same commit, document the Work-Class routing table and the Scope-Integrity rules in `CLAUDE.md`, and teach the lifecycle commands to read/set `class`. The second is an **enforcement question** on which the three candidates genuinely differ: how is the routing rule "Class 3 ⇒ approval requires ≥2 candidates" actually made to bite?

Candidate A leaves it to `CLAUDE.md` prose plus command discipline (no tool, no invariant). Candidate B makes it a graph-wide `spec:validate` invariant (machine-enforced but retroactive over the whole graph). This candidate, C, reads the rule as a **property of the act of approval** — the moment a `decision —selects→ contract` edge is introduced — and enforces it at that moment, diff-scoped, exactly as `spec:gate` already enforces "this PR adds evidence reaching an approved contract" and `spec:check-diff` enforces the sensitive-paths rule. A `selects` edge for a Class-3 intent backed by fewer than two `proposes` edges is precisely an approval that should never have been written, so the gate inspects only the `selects` edge the diff **adds** and fails the PR that introduces it.

This framing keeps the new constraint where the new constraint belongs (on new decisions), so it never retroactively re-judges the 4 pre-existing `selects` edges already in the graph — a property neither A (which does not enforce at all) nor B (whose invariant re-scans the whole graph on every merge) provides.

## Scope

**Fixed common core (identical across candidates A, B, C):**

- **Schema — `specs/schema/node-types.yaml`.** Add `class` to `required_fields` for both `intent` and `contract`. Add `produced_by` to NO `required_fields` list anywhere and to NO rule — it is opaque by construction (constraint 3 below). Do not model `class` via `status_values`; `status_values` is a string enum consumed by the string-only `enum_constraint` handler, and `class` is an integer (constraint 1 below).
- **Backfill — `specs/nodes/*.md`.** In the SAME commit as the schema edit, add `class:` frontmatter to every existing `intent` and `contract` node (~19 today plus this phase's own bootstrap nodes, including these three candidate contracts themselves), defaulting to `2`, with a lower class only where clearly warranted (e.g. the throwaway smoke-test intents). This is inseparable from the schema edit (constraint 2 below).
- **New integer-aware validation rule + handler.** A new `validation-rules.yaml` rule (e.g. `id: nodes-class-in-range`, `kind: int_field`, with `scope: nodes`, applied to `intent` and `contract`, allowed set `[0, 1, 2, 3]`) backed by a NEW handler `tools/handlers/int_field.ts`, registered in the `HANDLERS` map in `tools/validator.ts` alongside `list_field`. It is modeled on `tools/handlers/list_field.ts` (type-filtered, presence delegated to `required_fields`) and accepts the allowed numeric set as a rule param. It is NOT a reuse of `enum_constraint`.
- **`CLAUDE.md`.** Add the Work-Class routing table (Class 0–3) and the Scope-Integrity rules (brief-boundary-wrong ⇒ supersede the brief; contract-incomplete/behaviour-unchanged ⇒ follow-up intent via `/capture-intent`; selected-work-changes-behaviour ⇒ stop and return to human approval / new decision node). Document that contracts inherit the intent class and may revise it with rationale recorded in the contract body.
- **Commands — `.claude/commands/`.** `capture-intent` sets `class` (asks when not supplied); `propose-contracts` emits 1 candidate for Class 0–1 and ≥2 for Class 2–3; `write-brief` (and later lane/patch commands) read the contract `class` to decide lane/patch-market applicability.

**This candidate adds, on top of the core:**

- **`tools/approval-gate.ts`** exporting `evaluateApprovalGate(spec, input): GateResult` — a pure function over the HEAD graph plus the diff's added-edge set, mirroring `evaluateGate` and `evaluateCheckDiff`. Logic: for each ADDED edge of type `selects` whose target is a `contract`, follow that contract's `proposes` edge to its `intent`; read `intent.class`; if the intent is Class 3 (the rule generalizes to Class ≥ 2 per the routing table, but the hard acceptance bar is Class 3) and the number of distinct `proposes` edges targeting that intent is `< 2`, FAIL with a reason string naming the selected contract, the intent, its class, and the candidate count. Reuse `resolveBase`, `addedEdgeIds`, `nodesById`, `asString` from `gitdiff.ts`/`loader.ts` — no new git plumbing. Add a `runApprovalGate(spec)` wrapper identical in shape to `runGate`.
- **Subcommand wiring — `tools/spec.ts`.** Add `approval-gate` to `SUBCOMMANDS`, the `USAGE` string, and the dispatch block (PASS ⇒ exit 0 / `console.log`; FAIL ⇒ exit 1 / `console.error`), exactly like the existing `gate` and `check-diff` arms. Add the `spec:approval-gate` script to `package.json`.
- **CI step.** Add an `Approval gate` step running `pnpm spec:approval-gate` (with `GATE_BASE: ${{ github.event.pull_request.base.sha }}`) folded into the existing `pr-evidence.yml` job — which already does `fetch-depth: 0`, the specs-scope gate, and `pnpm install` — so no new workflow file is created. This is the one place this candidate adds CI surface beyond `spec-validate.yml`; see Trade-offs.

## Non-scope

- **No new diff-base machinery.** The gate reuses `resolveBase` / `addedEdgeIds` from `tools/gitdiff.ts` unchanged. `addedNodeIds` and `changedFiles` are not needed (the rule keys off the `selects` edge, not on changed files or new nodes).
- **No graph-wide candidate-count invariant.** This candidate deliberately does NOT add a `spec:validate` rule asserting "every Class-≥2 selected intent has ≥2 candidates" over the whole graph — that is Candidate B. C enforces only on the `selects` edge the diff introduces, leaving the 4 pre-existing `selects` edges untouched.
- **`produced_by` is never validated** — not in any `required_fields` list, not in any enum/list/int rule. Absence or any value is always fine.
- **REJECTED schema alternative (a): special-casing `enum_constraint` for a numeric field.** Rejected: `enum_constraint`'s `status` arm requires `typeof status === "string"`, and `loader.ts asString` returns `undefined` for non-strings, so a YAML `class: 2` integer is invisible to it; bending it to accept numbers would fork a single handler into two type regimes. A dedicated `int_field` handler is cleaner and is what the core scopes.
- **REJECTED schema alternative (b): a general per-field typed-field schema facility.** Rejected as YAGNI for one new field — it would enlarge `loader.ts` (`NodeTypeDef`) and spawn a multi-handler type system whose blast radius dwarfs the need. One narrow `int_field` handler is proportionate.
- **No `class` on node types other than `intent` and `contract`** (briefs, evidence, decisions, etc. are routed by the intent/contract they descend from), and no automatic re-classification of contracts that diverge from their intent's class — divergence is recorded as prose rationale in the contract body, not enforced.
- **No automated enforcement of the Scope-Integrity rules** — they are `CLAUDE.md` discipline in all three candidates.

## Trade-offs

- **+** Strongest, most isolated enforcement at exactly the moment of approval. The gate fires on the `selects` edge the PR adds, so an under-marketed Class-3 approval is blocked the instant someone tries to merge it — not discovered later by a graph scan.
- **+** **No retroactive blast radius from the enforcement mechanism.** Because the gate inspects only the ADDED `selects` edge, it never re-judges the 4 pre-existing `selects` edges. Contrast Candidate B, whose graph-wide invariant evaluates EVERY selected intent on every run and so can turn the whole graph red on merge if any historical decision predates the two-candidate norm. C carries no such hazard.
- **+** Architecturally consistent with the established gate family. `spec:gate` and `spec:check-diff` are already pure diff-aware `(spec, input) => {pass, reason}` functions sharing `resolveBase` + `addedEdgeIds` from `gitdiff.ts`; this gate is the same shape and reuses the same helpers, so it costs almost no new conceptual surface and is unit-testable without git, exactly like its siblings.
- **−** **Adds the most tool surface of the three:** a new gate module (`tools/approval-gate.ts`), a new `spec:` subcommand + `package.json` script, and a CI STEP to actually block PRs. Candidate A adds zero tool surface; Candidate B adds one rule + one handler and rides the existing `spec-validate.yml` with no new CI step.
- **−** **Mild tension with the intent's "no new workflow" line.** That line was scoped to the class-RANGE VALIDATION (which genuinely needs no new workflow — `spec-validate.yml` already runs `spec:validate`, and the new `int_field` rule rides it for free). But making this gate actually block merges requires a CI step beyond `spec-validate.yml`. This candidate keeps that honest by folding the step into the EXISTING `pr-evidence.yml` job rather than adding a workflow file — but it is still added CI surface, and reviewers should see it as a real cost.
- **−** **Diff-context only.** The gate runs in the PR/CI context off `GATE_BASE`; a `selects` edge written and committed in a purely local workflow is not caught until CI evaluates the diff. Candidate B's `spec:validate` invariant, by contrast, fires on any local `pnpm spec:validate`. C trades that local immediacy for diff-scoped precision.
- **−** **More moving parts than Candidate B's pure graph-shape check** (base-SHA resolution, the added-edge set). These are already-paid costs — the same plumbing `spec:gate` uses — but they are real failure surfaces (shallow checkout, unresolvable base) that a pure validate rule does not have.

## Risks

- **Gate authored but never wired into a blocking CI step** ⇒ enforcement silently absent (the worst failure: the rule looks enforced but nothing fails). *Mitigation:* the brief names the exact step — an `Approval gate` step running `pnpm spec:approval-gate` with `GATE_BASE: ${{ github.event.pull_request.base.sha }}`, added to the `pr-evidence` job after the existing gate step — and an acceptance example (4 below) requires observing a real PR fail that step. Treat the missing-step case as a contract failure, not a follow-up.
- **Diff-base resolution edge cases** (shallow clone, no `origin/HEAD`, a base that names no commit) ⇒ empty added-edge set and a false pass. *Mitigation:* reuse `resolveBase` from `gitdiff.ts` unchanged — it already fails CLOSED, verifying the base resolves to a real commit via `git rev-parse --verify` and throwing otherwise; and `pr-evidence.yml` already checks out with `fetch-depth: 0` and passes the PR base SHA, so the gate inherits a correct base with no new plumbing.
- **The `selects → contract → proposes → intent` walk mishandling odd graph shapes** — a `selects` edge whose target is not a `contract`, or a contract with no `proposes` edge, or a superseded sibling contract counted (or not) toward the candidate count. *Mitigation:* mirror `evaluateGate`'s defensive type guards (`asString(node.data["type"]) === "contract"`); define the candidate count as distinct `proposes` edges targeting the intent (the natural market size) and unit-test the superseded-sibling case explicitly so the counting rule is pinned, not incidental.
- **Whether the gate should also run locally.** It is exposed as a `spec:` subcommand, so a developer CAN run `pnpm spec:approval-gate` locally, but it only evaluates against a resolvable base. *Mitigation:* document in `CLAUDE.md` that `spec:approval-gate` is the PR-time enforcement and that the routing table (Class 3 ⇒ ≥2 candidates) is the human-followed rule `propose-contracts` already implements at authoring time; the gate is the backstop, not the primary workflow, so local-vs-CI divergence is acceptable and intended.
- **Over-broad class generalization** — applying the gate to Class ≥ 2 rather than only Class 3 could surprise authors mid-market. *Mitigation:* the hard acceptance bar is Class 3; if the gate is generalized to Class ≥ 2, state it explicitly in both the gate reason string and `CLAUDE.md` so the behavior is documented, not implicit.

## Acceptance examples

1. **Intent without a class is rejected by `spec:validate`.** Add an intent node with no `class:` frontmatter. `pnpm spec:validate` fails: the existing `nodes-required-fields` rule (`required_fields` handler) reports `node <id> missing required field: class`, because `class` is now in `required_fields` for `intent`. *Mechanism:* this is the core schema edit; the new `int_field` rule additionally fails any present-but-out-of-range or non-integer value (`class: "2"` string, `class: 2.0` float, `class: 4`), while it skips absent values (presence is `required_fields`' job, matching `list_field`). This holds in C identically to A and B.
2. **Class 0 intent runs end-to-end with one contract, one brief, no market.** Capture a Class-0 intent (`class: 0`); `propose-contracts` emits a single candidate; a `decision —selects→` that one contract; `brief —decomposes→ contract`. *Mechanism:* `spec:approval-gate` inspects the added `selects` edge, sees the target contract's intent is Class 0, and PASSES — the `< 2` candidate rule applies only to Class 3 (or Class ≥ 2 if generalized), so a single-candidate Class-0 approval is explicitly allowed. `spec:validate` is green throughout. No proposal market is required or produced.
3. **Class 3 intent cannot be approved with `< 2` candidates.** A PR adds a `decision —selects→ contract-X` edge where `contract-X —proposes→ intent-Y`, `intent-Y` is `class: 3`, and only ONE `proposes` edge targets `intent-Y`. *Mechanism:* `pnpm spec:approval-gate` resolves the base, derives the added-edge set, finds the new `selects` edge, walks `selects → contract-X → proposes → intent-Y`, reads `class: 3`, counts `1 < 2` candidates, and exits non-zero with a reason like `selects edge <id> approves contract-X for class-3 intent intent-Y, which has only 1 candidate contract(s) (need >=2)`. The CI `Approval gate` step in `pr-evidence.yml` fails and blocks the merge.
4. **The gate, not validate, is what blocks here (the deliberate division of labor vs Candidate B).** The SAME graph from example 3 PASSES `pnpm spec:validate` — there is no graph-wide candidate-count rule in this candidate, so the integer-range rule, required-fields rule, and all existing rules are satisfied and `spec:validate` exits 0. The block comes solely from `spec:approval-gate` exiting non-zero. *Mechanism:* this is the intended split — `spec:validate` answers "is every node well-formed and in-range," and `spec:approval-gate` answers "does this specific approval-introducing diff respect the routing table." Adding a second candidate (`contract-Z —proposes→ intent-Y`) in the same PR flips the gate to PASS while `spec:validate` stays green. Under Candidate B the same outcome would instead be a `spec:validate` failure, and a historical Class-3 single-candidate approval already in the graph would also fail B's invariant — whereas C, scoped to the added edge, never re-judges it.
5. **`produced_by` is never a validation error.** A node with `produced_by: "spec-writer"`, a node with `produced_by:` empty, and a node with no `produced_by` all pass `spec:validate` — it appears in no `required_fields` list and no rule.

## Critique

This critique is adversarial and does not rank or recommend; it tests C against the intent and the real codebase.

### 1. The CI wiring defeats the gate on exactly the PRs it must police (decisive correctness flaw)

C's enforcement lands entirely in one place: an `Approval gate` step running `pnpm spec:approval-gate`, "folded into the existing `pr-evidence.yml` job." I verified `pr-evidence.yml`. Its job opens with a **Determine scope** step whose every downstream step — including `pnpm install` and the gate run — is guarded by `if: steps.scope.outputs.skip != 'true'`. The scope logic computes the changed files and sets `skip=true` whenever the change set touches **only** `specs/` or `docs/` (it runs the job only when a non-spec/doc path is present).

A contract approval, by C's own definition, is a **specs-only** change: it adds one `selects` edge to `specs/graph/edges.yaml` plus a `decision` node under `specs/nodes/`. Nothing outside `specs/` is touched. So the scope step sets `skip=true`, and **every gated step is skipped, including `pnpm spec:approval-gate`.** The job reports success without ever running the gate.

This is not a tension; it is a correctness defect at the center of the contract. C chose `pr-evidence.yml` precisely because it "already does the specs-scope gate" — but that gate is the *evidence* gate, which is *designed to skip specs-only PRs* (its whole point is to demand evidence for code changes). C inherited a skip condition that is the exact complement of what an approval gate needs: the approval gate must fire **on** specs-only PRs (a `selects` edge is the specs-only event), and `pr-evidence.yml` skips exactly those. The gate would fire only on PRs that ALSO change code — an approval bundled with implementation, which lifecycle step 3 (approve) and step 5 (implement) deliberately separate. On a clean approval PR, C's gate is dead. (Compare `spec-validate.yml`, whose scope guard is the opposite — it runs precisely when `specs/` is touched — which is why B's `spec:validate` invariant fires on approval PRs and C's gate, in `pr-evidence.yml`, does not.)

### 2. The Acceptance examples assert an outcome the wiring cannot produce

Given finding 1, Acceptance example 3's claim — "The CI `Approval gate` step in `pr-evidence.yml` fails and blocks the merge" — is **false for the specs-only approval PR it describes**. That PR is specs-only, so `pr-evidence.yml` sets `skip=true` and the `Approval gate` step never runs, never fails, and never blocks. Example 4 inherits the same defect: it claims the gate blocks "the same graph" while `spec:validate` passes, but the gate is skipped, so the real CI outcome is that *both* checks are green and the under-marketed Class-3 approval merges unimpeded. The headline acceptance criterion of the intent ("a Class 3 intent cannot be approved until at least two candidate contracts exist") is therefore **not** delivered by C as written. The pure `evaluateApprovalGate` function would return the right verdict if invoked; the contract's failure is that CI never invokes it on the governing PR. C's own Risk 1 names "gate authored but never wired into a blocking CI step" as "the worst failure: the rule looks enforced but nothing fails" — and the wiring C specifies realizes exactly that failure mode for specs-only approvals. The brief cannot satisfy this contract by following it; the contract's named step is the bug. To actually fire, the gate must live where specs-only PRs run — its own step in a workflow with a `^specs/`-positive scope guard (i.e. `spec-validate.yml`'s shape) or a new workflow — which is more CI surface than C admits.

### 3. "No new workflow" is honored only nominally, and the honoring is what breaks it

The intent's Validation line caps the validation surface: "extend spec:validate; no new workflow — spec-validate.yml already runs it." That clause is scoped, by its own words, to the **class-range validation** — which genuinely needs no new workflow, since the new `int_field` rule rides `spec:validate`. C correctly identifies this and the core's range rule is faithful. But C's *enforcement* mechanism is a CI gate, not a validate rule, and C tries to stay "no new workflow"-compliant by folding the step into an existing job. Finding 1 shows that this nominal compliance is precisely what kills the gate: C picked the one existing job whose scope guard excludes the PRs the gate must police, and it picked that job *because* reusing it avoids a new workflow file. So C trades correctness for "no new file" compliance — the worst of both, since the result is neither a new workflow nor a working gate. A faithful reading would either (a) accept that machine-enforcing approval needs a step in a specs-scoped workflow and flag that as a real deviation from the intent's line, or (b) recognize that the intent's "no new workflow" line was written for B's validate-rule approach and that C's mechanism structurally cannot honor it. C does neither squarely.

### 4. Technical plumbing claims are accurate — but verify what that buys

I confirmed against the codebase, and C is correct here (worth recording so the brief does not re-litigate it):
- `resolveBase`, `addedEdgeIds`, `addedNodeIds` exist in `tools/gitdiff.ts` with those exact names and shapes; `resolveBase` does fail closed (it throws unless `git rev-parse --verify` confirms a real commit), so C's Risk-2 mitigation is real.
- `nodesById` and `asString` exist in `tools/loader.ts` with the claimed signatures.
- `evaluateGate` and `evaluateCheckDiff` are indeed pure `(spec, input) => GateResult` where `GateResult = {pass, reason}`, and both end with a `run*(spec)` wrapper that calls `resolveBase()` then derives the added sets — exactly the shape C proposes for `evaluateApprovalGate` / `runApprovalGate`. The defensive `asString(contract.data["type"]) === "contract"` guard C says it will mirror is real.
- The edge direction C's walk assumes is correct: `proposes` is `source: contract → target: intent`, so "follow the contract's `proposes` edge to its intent" and "count distinct `proposes` edges targeting that intent" are both right against the live graph.

But note what the accuracy buys: the *pure function* is sound and unit-testable git-free. The contract's failure is entirely at the integration seam (finding 1), which the unit tests by construction cannot catch — C's own `node --test` matrix runs `evaluateApprovalGate` directly, bypassing the CI scope guard that breaks it. So the verification plan would show green while the deployed gate is inert. The only step that would surface the defect is "A scratch PR exercising acceptance 3/4" — and per finding 1 it would surface it as the gate NOT firing, i.e. the scratch PR would reveal the contract is unsatisfiable as written.

### 5. The counting rule is raised as an open question, not pinned

C defines candidates as "distinct `proposes` edges targeting that intent" but in Risk 3 explicitly flags "a superseded sibling contract counted (or not) toward the candidate count" as an unresolved case, deferring it to "unit-test the superseded-sibling case explicitly so the counting rule is pinned." So by C's own text the rule is **not** pinned at contract time — it is an open question handed to the brief. This matters concretely: if a Class-3 intent had two candidates and one is later superseded (its successor pointing back via `supersedes`, the superseded node staying in place per CLAUDE.md rule 3), a raw "count `proposes` edges targeting the intent" still counts the superseded sibling's `proposes` edge, so the gate would pass on what is now effectively a one-live-candidate market — or, if superseded edges are excluded, a legitimately-marketed approval could fail. C does not decide which; it should, because the verdict is load-bearing for the headline acceptance bar. (B faces the same question and resolves it explicitly, a definite choice; C leaves it indefinite.)

### 6. Coverage gaps the "unbypassable" framing understates

- **Branch-direct commits and local-only flows.** C is diff-and-PR-context only. A `selects` edge committed straight to a branch with no PR, or merged through any path that does not trigger `pr-evidence.yml`, is never inspected. C frames this honestly as "the gate is the backstop, not the primary workflow" — but combined with finding 1 (the gate also does not fire on the normal specs-only approval PR), the "machine-enforced, unbypassable" selling point is substantially overstated. The honest position is: C's gate is unbypassable only for an approval PR that *also* changes code and reaches the `pr-evidence` job's non-skip branch — a narrow and atypical case.
- **Proposal-time vs approval-time.** C enforces the ≥2 rule only at *approval* (when the `selects` edge is added), never at proposal. The "Class 2–3 must emit ≥2 candidates" obligation at proposal time lives only in `propose-contracts` command prose (the fixed core), unenforced by any machine in C — identical to A's advisory posture, despite C's framing as the strongly-enforced candidate.

### 7. Scope surface — largest of the three, and the marginal surface is what fails

C adds the most surface beyond the fixed common core: a new module (`tools/approval-gate.ts`), a new `spec:` subcommand (extending `SUBCOMMANDS`/`USAGE`/dispatch in `tools/spec.ts` and a `package.json` script), and a CI step. I confirmed `tools/spec.ts` has the `SUBCOMMANDS` array, `USAGE` string, and per-subcommand dispatch arms C must extend, and `package.json` has the parallel `spec:*` scripts — so the mechanical additions are accurately scoped and small. The intent's Validation scope, however, asked only for class presence/range (a validate rule) and "no new workflow." Everything C adds past the `int_field` core rule is justified *only* by the Acceptance criterion "a Class 3 intent cannot be approved until ≥2 candidate contracts exist." That justification is legitimate in principle — but it is fatally undercut by finding 1: the very surface C adds to satisfy that criterion (the gate + the chosen CI step) does not satisfy it on the governing PR. So C pays the largest scope cost of the three and, as written, does not buy the enforcement that cost was for. The brief would have to *change the CI wiring C specifies* to make the surface earn its keep — meaning the contract under-specifies the one thing that makes its extra surface worthwhile.

### Summary

C's pure-function design, helper reuse, and core schema/range work are technically accurate against the codebase and faithfully share the common core. Its single decisive problem is that its named enforcement mechanism — `pnpm spec:approval-gate` folded into `pr-evidence.yml` — is guarded by that job's specs-only **skip** condition and therefore does not run on the specs-only approval PRs it exists to block, making Acceptance examples 3 and 4 false as written and the headline acceptance criterion undelivered. Secondary gaps: the counting rule for superseded siblings is explicitly left open rather than pinned; the "unbypassable" framing is overstated given branch-direct/local and the finding-1 skip; and C carries the largest scope surface of the three to buy enforcement that its specified wiring does not deliver.
