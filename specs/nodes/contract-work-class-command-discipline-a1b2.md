## Problem interpretation

`intent-work-class-routing-b9c4` is multi-surface: it touches the schema (`node-types.yaml`), the validator (`tools/spec.ts` + a new handler), `CLAUDE.md`, and four slash commands. Under its own routing table a multi-surface change is **Class 3**, which is precisely why this proposal market carries `>=2` candidates rather than one. All three candidates (A, B, C) ship an identical fixed core; they differ only on how the acceptance rule *"a Class 3 intent cannot be approved until `>=2` candidate contracts exist"* is enforced.

This candidate — **A, command-discipline** — delivers the full common core and enforces routing and the Class-3-needs-`>=2` rule through `CLAUDE.md` documentation plus slash-command pre-checks alone. It adds **no** new `tools/` gate function and **no** new graph invariant in `spec:validate` beyond the class-range check. The one machine-checked addition is the integer-aware class-range rule; everything else is prose and command behavior. This contract self-classifies as **Class 3** (inheriting from the intent), recorded here as required by the routing table.

Three correctness constraints bind every candidate, this one included:

1. **The integer trap.** `class: 2` parses as a JS `number`. The shared helper `asString` in `tools/loader.ts` returns `undefined` for anything that is not a non-empty string, and `enum_constraint` guards with `typeof status !== "string"` and matches against `status_values: string[]`. Reusing `enum_constraint` for `class` would therefore either reject every integer or force a string field. So the range check is a **new integer-aware handler** modeled on `tools/handlers/list_field.ts` (type-filtered loop over `spec.nodes`, scope/type/field config), taking the allowed numeric set `[0,1,2,3]` as a rule param. It must accept only a JS integer in that set and reject `class: "2"` (string), `class: 2.0` where it is not integral, and any out-of-range or non-numeric value.
2. **Atomicity.** The moment `class` joins `required_fields` for `intent` and `contract` in `node-types.yaml`, `required_fields.ts` flags every existing intent/contract node lacking it — the whole graph goes red at once. The schema edit and the full backfill are therefore **one inseparable commit**; the schema cannot be staged before the backfill.
3. **`produced_by` is never validated.** It is added to **no** `required_fields` list and is named by **no** enum/list/range rule. It stays opaque by construction: absent or any value, always valid.

## Scope

**Shared fixed core (identical across A, B, C):**

- **`specs/schema/node-types.yaml`** — add `class` to `required_fields` for `intent` and `contract`. Add `produced_by` to no list (it is optional and unvalidated; documented as a recognized free-text frontmatter key, not a schema-enforced one).
- **New handler `tools/handlers/int_range.ts`** (working name) — modeled on `list_field.ts`: config `scope: nodes`, `type: <nodeType>`, `field: class`, plus an allowed-set param (e.g. `values: [0,1,2,3]`). Skip nodes of other types and skip absent values (presence is `required_fields`' job). For a present value, emit a finding unless it is an integer (`Number.isInteger(value)`) in the allowed set — rejecting strings, floats, and out-of-range numbers. Registered in the handler map beside `enum_constraint`/`list_field`/`required_fields`.
- **`specs/schema/validation-rules.yaml`** — two rule entries (one for `intent`, one for `contract`) invoking the new handler with `field: class`, `values: [0,1,2,3]`.
- **Backfill (same commit as the schema edit)** — every existing `intent` and `contract` node (the ~19 today plus any added by migration time, including these very candidate contracts) gets `class` in frontmatter, defaulting to `2`. A lower class only where clearly warranted (e.g. throwaway smoke-test intents). This keeps `spec:validate` green graph-wide the instant the field becomes required.
- **`CLAUDE.md`** — the Work-Class routing table (Class 0–3 with their critic/lane/patch-market/human-gate obligations) and the Scope-Integrity rules (brief-boundary-wrong → supersede the brief; contract-incomplete/behaviour-unchanged → spawn a follow-up intent via `/capture-intent`; selected-work-changes-behaviour → stop, return to human approval with a new decision node). "No review may silently absorb scope drift."
- **Class-aware commands** — `/capture-intent` sets `class`, asking if not supplied; `/propose-contracts` reads the intent class and emits 1 candidate for Class 0–1, `>=2` for Class 2–3; `/write-brief` and later lane/patch commands read the contract class to decide lane/patch-market applicability.

**Distinguishing this candidate (A):**

- The Class-3-needs-`>=2` acceptance rule is enforced **in command instructions only**. `/propose-contracts` refuses to emit fewer than 2 candidates for a Class 2–3 intent. `/approve-contract` reads the target intent's class and, for a Class 3 intent, refuses to record a `selects` decision unless `>=2` candidate contracts (`proposes` edges into that intent) already exist — a pre-check the command performs before invoking graph-maintainer, surfacing a clear refusal otherwise.
- **No** `tools/gate.ts`-style function, **no** new `spec:` subcommand, and **no** second `spec:validate` graph invariant. `spec:validate` gains only the integer-aware class-range rule.

## Non-scope

- No machine gate (CI or `spec:`) enforcing the candidate-count rule. That is Candidate C's diff-aware gate and Candidate B's validate-time invariant; this candidate deliberately leaves it to command discipline.
- No new graph invariant relating `selects`/`proposes` edges to class. The four existing `selects` edges in the graph are untouched and not retroactively re-validated against candidate counts.
- No new CI workflow. `spec-validate.yml` already runs `spec:validate`; the new class-range rule rides it with zero workflow change.
- **`produced_by` is not validated** — no required-fields entry, no rule.
- **Rejected schema alternative (a):** special-casing `enum_constraint` to accept a numeric field. Rejected because of the integer trap — it would graft a numeric branch into a handler whose entire contract is `status_values: string[]` and whose existing logic guards `typeof status !== "string"`; an ugly, error-prone special case in a string-only handler.
- **Rejected schema alternative (b):** a general per-field typed-field schema facility (each field declaring its own type/enum/range in `node-types.yaml`). Rejected as YAGNI and over-engineering for a single new numeric field — it widens the loader (`NodeTypeDef`) and multiplies handler surface for no present second consumer. A focused `int_range` handler is the proportionate move.

## Trade-offs

- **+** Smallest tool surface of the three. One new handler plus two rule entries and the backfill is the entire code delta; no new gate, no new graph invariant, no new subcommand.
- **+** No retroactive effect on the four existing `selects` edges. A validate-time invariant (Candidate B) would re-judge them against candidate counts at every run; this candidate leaves the historical graph green as-is.
- **+** Consistent with today's architecture, where contract approval is already command-mediated and there is **no** machine gate at approval. This adds enforcement where enforcement already lives (the command), not a new control point.
- **+** Fastest to ship: the only code is the class-range rule + handler + backfill; the rest is `CLAUDE.md` prose and command-file edits.
- **−** Enforcement of "Class 3 needs `>=2` candidates" is **advisory**. It lives in command instructions a human or agent can bypass — most obviously by hand-editing `specs/graph/edges.yaml` to add a `selects` edge directly. Candidate B (validate-time invariant) and Candidate C (diff-aware gate) make the same rule mechanically unbypassable; this candidate does not.
- **−** The routing obligations (which critics run, whether lanes are decomposed, whether a patch market opens) are likewise process discipline, not mechanically checked. Nothing fails if a Class 3 intent is run with a single critic and no lanes.
- **−** Correctness now depends on the command files and `CLAUDE.md` staying in agreement; prose and behavior can drift apart over time.

## Risks

- **Prose/behavior drift** between the `CLAUDE.md` routing table and the actual `/propose-contracts` and `/approve-contract` behavior. *Mitigation:* encode the rule as an explicit, literal pre-check assertion in each command file (read intent class; count inbound `proposes` edges; refuse with a named reason), and cross-reference the `CLAUDE.md` table from the command so a reader sees both in one place.
- **Silent under-proposing** for a Class 3 intent — an agent emits one candidate and nothing fails at validate or CI time. *Mitigation:* `/propose-contracts` refuses to emit `<2` candidates for Class 2–3 at generation time (the first line of defense), and `/approve-contract`'s pre-check is the second; the gap remains only for someone bypassing both commands.
- **Reliance on reviewer vigilance at approval.** Because there is no machine gate, the human approval step in `/approve-contract` is the backstop, and a careless approval can still let an under-proposed Class 3 selection through. *Mitigation:* make the `/approve-contract` refusal loud and specific (naming the intent, its class, and the candidate count it found), so a human bypassing it is doing so knowingly; document that if discipline proves insufficient in practice, this contract graduates to Candidate B or C without re-doing the shared core (they share the identical schema/handler/backfill/CLAUDE.md base).
- **Backfill omission** leaving one intent/contract without `class`, turning the graph red at the moment the field becomes required. *Mitigation:* the schema edit and backfill are one commit; `pnpm spec:validate` over the whole tree in that commit is the gate, and the integer-aware rule plus `required_fields` together prove every intent/contract carries a valid `class`.

## Acceptance examples

1. **Intent without a class is rejected by `spec:validate`.** Capture an `intent` node with no `class` frontmatter. The instant `class` is in `intent.required_fields`, `required_fields.ts` emits `node <id> missing required field: class`, and `pnpm spec:validate` exits non-zero. *Mechanism: the shared core's required-field declaration — no candidate-specific machinery involved.*
2. **A non-integer or out-of-range class is rejected.** `class: "2"` (string), `class: 2.0` non-integral, or `class: 4` each produces a finding from `int_range.ts` (`Number.isInteger` + allowed-set check), and `spec:validate` fails. `class: 2` (integer) passes. *Mechanism: the new integer-aware handler, not `enum_constraint`, which would mishandle the numeric value.*
3. **A Class 0 intent runs end-to-end with one contract and one brief and no proposal market.** Capture an intent with `class: 0`. `/propose-contracts` reads the class and emits exactly one candidate; `/write-brief` reads the contract's class and decomposes it without lanes or a patch market; `spec:validate` stays green throughout. *Mechanism: `/propose-contracts` and `/write-brief` branch on class in command discipline; the single-candidate path is permitted for Class 0–1 by the routing table.*
4. **A Class 3 intent cannot be approved with `<2` candidates.** Capture an intent with `class: 3` and propose only one candidate. `/approve-contract` reads the intent's class, counts the inbound `proposes` edges, finds `<2`, and **refuses** to record the `selects` decision — naming the intent, its class, and the count — so no `decision —selects→ contract` edge is authored. Adding a second candidate, then re-running `/approve-contract`, lets approval proceed. *Mechanism: this candidate's `/approve-contract` pre-check produces this guarantee — **not** `spec:validate` and **not** a gate.*
5. **Honest limit of this candidate.** If someone hand-crafts the under-proposed Class 3 selection — editing `specs/graph/edges.yaml` to add a `selects` edge directly, bypassing `/approve-contract` — `pnpm spec:validate` still **passes**, because no graph invariant ties candidate count to class in this candidate. That is the explicit trade-off versus Candidate B (which would fail validation) and Candidate C (which would fail the PR-time diff gate). The backstop here is the human approval step, not the machine.

- `pnpm spec:validate` green on the real tree **after** the single schema-edit-plus-backfill commit, proving every intent/contract carries a valid integer `class` and no node lost its body or other required fields.
- A confirmation that `produced_by` set to an arbitrary string, and `produced_by` absent, both leave `spec:validate` green — proving it is unvalidated by construction.
- Manual/transcript verification of the command pre-checks: a Class 3 intent with one candidate is refused by `/approve-contract`; the same intent with two candidates is approved; a Class 0 intent runs single-candidate through `/propose-contracts` and `/write-brief`. (These are command-discipline behaviors, so they are verified by command transcript, not by an automated check — itself a reflection of this candidate's advisory enforcement.)
- A recorded note that the four existing `selects` edges are untouched and `spec:validate` does not re-judge them.

## Critique

Adversarial review of Candidate A (`contract-work-class-command-discipline-a1b2`) against `intent-work-class-routing-b9c4` and the real codebase. Findings are substantive; no ranking is given.

### 1. Faithfulness gap on "cannot be approved" — the contract softens the intent's hardest word, and says so only in a footnote

The intent's Acceptance is unambiguous: "a Class 3 intent **cannot be approved** until at least two candidate contracts exist." A's own acceptance example 5 concedes that a hand-edited `selects` edge in `edges.yaml` "still **passes** `pnpm spec:validate`." So under A, a Class 3 intent with one candidate *can* be approved — the `decision —selects→ contract` edge stands in a green graph. "Cannot" becomes "is asked not to, by command prose." That is a real faithfulness gap, and the contract should own it in its title/interpretation rather than only in example 5 and a trade-off bullet. The framing "the human approval step is the backstop" is not a guarantee: `/approve-contract` today (`.claude/commands/approve-contract.md`) records the decision via graph-maintainer with *no* class-aware or count-aware pre-check, and nothing outside the command's own prose obliges a human to invoke the command at all. "Backstop" here restates the gap (a human might notice) rather than closing it. The contract should not present advisory enforcement as satisfying an acceptance criterion phrased as an absolute prohibition; at minimum it should flag that this single acceptance line is met only in spirit, not mechanically.

### 2. The promised command pre-checks are not expressible in `/approve-contract` as it exists — A assumes command structure not present

A's distinguishing mechanism is that `/approve-contract` "reads the target intent's class," "counts inbound `proposes` edges," and "refuses" before invoking graph-maintainer. Checked against the actual command file: `.claude/commands/approve-contract.md` is nine lines of natural-language instruction to an agent — it has no conditional logic, no assertion mechanism, and no defined failure exit. It *does* already locate the intent (via `outgoing.yaml`) and the sibling candidates (via the intent's entry in `incoming.yaml`), so the *data* A needs is reachable — but "refuses with a named reason" is an instruction an agent may or may not follow, not a check that fails. So A's repeated verb "refuses" overstates what a markdown command can promise: it is a prompt, not a guard. Relatedly, A's risk-mitigation to "encode the rule as an explicit, literal pre-check assertion in each command file" describes a capability the command format does not have — there is no place in these files for an *assertion* that halts. This should be stated as "we instruct the agent to refuse," not "the command refuses."

Compounding this: A's verification needs admit the pre-checks "are verified by command transcript, not by an automated check." Combined with the Non-scope (no `tools/` function, no `spec:` subcommand, no new invariant), there is **no automated test that can ever catch a regression** in A's headline enforcement. If a future edit to `propose-contracts.md` or `approve-contract.md` drops the class logic, every test stays green. A names this honestly in places but does not draw the consequence: the contract's central acceptance criterion is, under A, permanently and exclusively transcript-checked. That is a verifiability gap the contract should surface as a first-class property, not bury in the last verification bullet.

### 3. Technical error: `class: 2.0` is NOT rejected by `Number.isInteger` — the claim is repeated four times

A asserts the new handler "reject[s] `class: 2.0` where it is not integral," with the same claim in Scope, acceptance example 2 ("`class: 2.0` non-integral ... produces a finding"), and verification needs ("`class: 2.0` (non-integral) ... behave correctly"). This is wrong given this repo's loader. `fromYaml` uses js-yaml `CORE_SCHEMA` (`tools/yaml.ts`), which parses the YAML scalar `2.0` to the JavaScript number `2`. JavaScript has a single number type with no integer/float distinction at runtime, so `Number.isInteger(2)` returns **`true`**. The handler A specifies (`Number.isInteger(value)` + allowed-set) would therefore **accept** `class: 2.0`, not reject it. The only inputs `Number.isInteger` actually rejects are genuinely non-integral values (e.g. `2.5`) and non-numbers. So example 2's "passes/fails" matrix is partly incorrect, and any test written to A's spec asserting `class: 2.0 ⇒ finding` would fail against the very handler A describes. The contract should either drop the `2.0` claim or specify a different mechanism (e.g. inspecting the raw YAML text), which the current `list_field`-modeled, post-parse handler shape cannot provide.

### 4. Under-specified handler registration and rule targeting

A says the new `int_range.ts` is "registered in the handler map beside `enum_constraint`/`list_field`/`required_fields`." The actual registration point is the `HANDLERS` map in `tools/validator.ts` plus an `import` at the top — A never names `validator.ts` (siblings B and C both name it explicitly). More importantly, A says the rule "targets only `intent`+`contract`" via "two rule entries (one for `intent`, one for `contract`)," modeled on `list_field`. That is consistent with `list_field`'s shape — it takes a single `type:` and loops `if (asString(node.data["type"]) !== type) continue` — so two rules is the correct mechanism. But A does not specify what config key carries the allowed set: the prose drifts between "an allowed-set param (e.g. `values: [0,1,2,3]`)" and "`values: [0,1,2,3]`" without committing, and `list_field` has no precedent param to copy (it only reads `scope`/`type`/`field`). The brief author must invent the param name and its read path with no template, which A's "modeled on `list_field`" understates. The core claim that the handler shape supports `Number.isInteger` + allowed-set is otherwise sound: `list_field` already does a type-filtered loop and skips absent values, so an integer-and-set variant is a faithful sibling — for the integer/range cases, just not for `2.0` (see finding 3).

### 5. Backfill count is wrong (~19 is short of the real ~22)

A states "the ~19 today" intent/contract nodes need `class`. Counting `specs/nodes/`: 9 intent nodes and 13 contract nodes (including the three work-class candidates) = **22**. The actual non-candidate base is 19 (9 intents + 10 prior contracts), so "~19 today" silently omits the three candidate contracts from "today" while A elsewhere says they are included. The number is at best ambiguous, at worst an undercount that a backfill executor could act on and leave a node red. Note also `intent-docs-arrow-lint-e7b3` and `intent-status-coherence-d4f2` exist and appear in no candidate's worked example — A's "default to 2 ... lower class only where clearly warranted" gives no guidance on these two, and a wrong default here is silent.

### 6. Scope obligations the intent expected — are they dropped or faithfully deferred?

A states the non-`>=2`-candidate routing obligations (which critics run, lane decomposition, patch market) are "process discipline, not mechanically checked" and "Nothing fails if a Class 3 intent is run with a single critic and no lanes." Cross-checking the intent: its routing table frames Class 3 as "the full specialist critic panel **required**; lane decomposition **required**." The intent does *not* ask for these to be machine-enforced — its Validation section scopes `spec:validate` only to `class` presence/range. So A is faithful to the intent in leaving these as process. However, A goes further than the intent in one place: the intent's Acceptance singles out the `>=2`-candidates rule as the one Class 3 obligation expected to actually *bind* approval ("cannot be approved until"), and A reduces even *that* to advisory (finding 1). So A faithfully defers the soft obligations but also quietly demotes the one obligation the intent hardened. That asymmetry should be named: A treats all routing obligations as uniformly advisory, whereas the intent treated the candidate-count one as a gate.

### 7. Ambiguity / missing cases

- **Class 1 vs Class 2 boundary in `propose-contracts`.** A says `/propose-contracts` "emits 1 candidate for Class 0–1, `>=2` for Class 2–3." The current command unconditionally instructs "produce 2-3 candidate contract nodes" — A's edit must make this conditional on class, but A does not say how the single-candidate Class 0–1 path interacts with the command's closing "asking for a human decision" / "Do NOT select a winner." For a one-candidate Class 0 intent there is no market to compare; the command's trade-off-table-and-ask ending is left undefined for that path.
- **`/capture-intent` "ask if no class supplied."** The intent requires `/capture-intent` to "ask before creating the node" when no class is given. A restates this but nothing guarantees it: the current `capture-intent.md` has no interactive prompt and A provides no mechanism that *forces* the ask — an agent can default silently. There is no validation backstop either (a defaulted class still passes range).
- **The three candidates' own class.** A says the candidates "self-classif[y] as Class 3" and are backfilled — but to which value? One place says A inherits Class 3 from the intent; the backfill defaults everything to 2. A leaves a contradiction: is `contract-work-class-command-discipline-a1b2`'s own `class` field 2 (backfill default) or 3 (self-classification)? The contract does not say which value goes in its own frontmatter.
- **`class: null` / empty.** A's verification lists `class: null` as "absent → skipped," and the handler skips `value === undefined || value === null` per the `list_field` template. But `required_fields.ts` treats `null` and `""` as absent too (`present()`), so a `null` class is caught by `required_fields`, not the range rule — A's "absent → skipped" for the range handler is correct, but the division of labor is left implicit and example 2 does not cover empty-string `class:`.

### 8. Self-described advantages over B/C that are partly overstated

- "Smallest tool surface" / "Fastest to ship" are accurate as code delta. But A pairs this with "enforcement where enforcement already lives (the command)" — and the command, as shown in finding 2, cannot *enforce*, only instruct. So the architectural-consistency claim ("contract approval is already command-mediated") is true, but it cuts both ways: today's approval being command-mediated and *unguarded* is exactly why A cannot deliver "cannot."
- "No retroactive effect on the four existing `selects` edges" is a real and correct contrast with B. But A states this as a pure plus; it is the flip side of A enforcing nothing — there is nothing to apply retroactively. Framing the absence of retroactive checks as an advantage over B, when it stems from the absence of any check at all, overstates the distinction.
