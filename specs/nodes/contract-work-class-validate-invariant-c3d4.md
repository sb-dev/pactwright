This intent (`intent-work-class-routing-b9c4`) is itself multi-surface — it edits the schema (`node-types.yaml`), the validator (`tools/`), `CLAUDE.md`, and the lifecycle commands at once. Under its own routing table that makes it a **Class 3** change, which is precisely why this proposal market carries ≥2 candidate contracts (this is candidate **B** of three). The three candidates share an identical common core and differ only in how one acceptance criterion — *a Class 3 intent cannot be approved until ≥2 candidate contracts exist* — is enforced.

## Problem interpretation

The intent asks for two schema additions, a graph-wide backfill, a CLAUDE.md routing table plus scope-integrity rules, command updates, and a validator extension, all in one PR. The irreducible common core (identical across candidates A, B, C):

- **Schema.** In `node-types.yaml`, add `class` to `required_fields` for node types `intent` and `contract` only. Add `produced_by` to **no** `required_fields` list and to **no** enum/list rule — it must remain opaque by construction (fixed constraint 3): an optional, never-validated free-text provenance field present on every node type by virtue of never being constrained.
- **The integer trap (fixed constraint 1).** `class: 2` parses as a JS number. The existing `enum_constraint` handler is string-only — its status branch does `typeof status !== "string" || !def.status_values.includes(status)` against `status_values: string[]`, and `loader.asString` returns `undefined` for non-strings. Reusing `enum_constraint` would therefore reject every legitimate integer class. So the range check must be a **new integer-aware handler** modeled on the shape of `tools/handlers/list_field.ts` (type-filtered iteration over `spec.nodes`, a numeric allowed-set `[0,1,2,3]` taken as a rule param), registered as a new `kind` in the `HANDLERS` map in `tools/validator.ts`. It must reject the string `"2"`, the float `2.0`, and any value outside the set; accept only an integer in `{0,1,2,3}`. Two schema alternatives are explicitly rejected (recorded in Non-scope): special-casing `enum_constraint` for a numeric field, and a general typed-field schema facility.
- **Atomicity (fixed constraint 2).** The instant `class` joins `required_fields` for intent/contract, `tools/handlers/required_fields.ts` flags every existing intent/contract node lacking it (`!present(node.data[field])`), turning the whole graph red. So the `node-types.yaml` edit and the full backfill of all ~19 existing intent+contract nodes (defaulting to class 2, with a lower class only where clearly warranted — e.g. `intent-capture-smoke-test-46d5` / `intent-capture-smoke-superseded-e7f8`) **must be one inseparable commit**, including these very candidate contracts. You cannot stage schema-first.
- **CLAUDE.md / commands.** The Class 0–3 routing table, the scope-integrity rules, and the command updates to `/capture-intent`, `/propose-contracts`, `/write-brief` exactly as the intent specifies.

What distinguishes **candidate B**: beyond the class-range rule, it makes the "class ≥2 cannot be approved with <2 candidates" criterion a **machine-enforced graph invariant** — a *second* new spec:validate rule + handler. It is a pure graph-shape check over the loaded spec (no git diff): for any intent of `class ≥ 2` that has an inbound `selects` decision (the chain `decision —selects→ contract —proposes→ intent`), there must be **≥2** distinct candidate `proposes` edges pointing at that intent. It rides the existing `spec-validate.yml` — no new workflow. This is the strongest faithfulness to the intent's own wording ("extend spec:validate"; "no new workflow"): the enforcement is just another rule on the CI that already runs. It contrasts with candidate A (docs + command discipline only, no machine gate) and candidate C (a diff-aware PR-time gate, `tools/gate.ts`-style, scoped to the diff rather than the whole graph).

## Scope

- **`node-types.yaml`** — add `class` to `required_fields` for `intent` and `contract`. Do not touch `produced_by` here (its optionality is achieved by *omission* from every list).
- **Backfill (same commit)** — set `class` on all existing intent+contract nodes: default 2; class 0 for the two throwaway capture-smoke intents; the three candidate contracts of this market each get `class: 2` (a Class 3 *intent* spawns Class 2 contract work unless a contract revises with rationale — this contract revises to Class 3 in its own frontmatter, recorded here).
- **New handler `class_range`** (working name) in `tools/handlers/class_range.ts`, registered in `tools/validator.ts` `HANDLERS`. Config `scope: nodes`, `types: [intent, contract]`, `allowed: [0,1,2,3]`. For each matching node, skip when the field is absent (presence is `required_fields`' job, mirroring `list_field`'s `if (value === undefined) continue`); otherwise require `Number.isInteger(value) && allowed.includes(value)`. Emit a finding for a string, a float, or an out-of-set integer.
- **New handler `class_market_quorum`** (working name) in `tools/handlers/class_market_quorum.ts`, registered in `HANDLERS`. The invariant: build `nodesById` (from `loader.ts`), then for every `selects` edge, resolve its target contract, find that contract's outgoing `proposes` edges to get the intent(s) it proposes; if the intent's `class >= 2`, count **all** `proposes` edges in the graph whose `target` is that intent; if the count is `<2`, emit a finding. Model the traversal on `references_resolve.ts` (iterate `spec.edges`, resolve via a known-id set) and `list_field.ts` (type-filtered node access).
- **`validation-rules.yaml`** — append two rules. `class-in-range` (kind `class_range`) may sit near the other node-field rules. `class-market-quorum` (kind `class_market_quorum`) **must be listed after `edges-references-resolve`** so it can assume edge endpoints resolve to real nodes before traversing them.
- **CLAUDE.md** — routing table, scope-integrity rules, command-update notes, exactly per the intent.
- **Commands** — `/capture-intent` sets/asks `class`; `/propose-contracts` emits 1 candidate for class 0–1 and ≥2 for class 2–3; `/write-brief` reads contract class for lane/patch applicability.

## Non-scope

- **No new CI workflow.** `spec-validate.yml` already runs `spec:validate`; both new rules ride it.
- **No diff-aware gate.** The quorum check is pure graph-shape, not a git-diff adapter — that is candidate C's mechanism, deliberately not adopted here (no base-ref machinery, no PR-number resolution).
- **Rejected schema alternative (a): special-casing `enum_constraint` for a numeric field** — rejected because of the integer trap; the handler's string contract (`status_values: string[]`, `asString`) would have to be forked anyway, so a clean new handler is cheaper and less risky.
- **Rejected schema alternative (b): a general per-field typed-field schema facility** — rejected as YAGNI for a single new field; it widens the loader's `NodeTypeDef`/`Rule` surface and fans out across multiple handlers for no current second consumer.
- **`produced_by` is never validated** — no rule of any kind references it; it is not added to any `required_fields` list. Adding it to a rule is out of scope and would be a regression.
- No retire/supersede changes to existing decision or contract lifecycle beyond what scope-integrity documents.

## Trade-offs

- **+** The acceptance criterion becomes a real, unbypassable invariant living in the graph itself: a hand-crafted, under-proposed class ≥2 selection turns `spec:validate` **red**. No command discipline can be skipped past it (the gap in candidate A).
- **+** Strongest faithfulness to the intent's framing ("extend spec:validate" / "no new workflow") — the enforcement is simply another validate rule on the CI that already runs `spec-validate.yml`.
- **+** Pure graph-shape (it counts edges over the loaded spec), so no git-diff adapter, no base-ref resolution, no PR-number handling — strictly simpler than candidate C's diff-aware gate.
- **+** Reuses established handler patterns (`references_resolve` traversal, `list_field` type-filtering, `nodesById`), so the blast radius is two small, unit-testable pure functions.
- **−** Evaluated **graph-wide**, so it **retroactively** constrains the existing graph. There are 4 existing `selects` edges; the default backfill sets their intents to class 2, which makes the backfill default **load-bearing** — if any selected intent had <2 candidates it would turn the *whole* graph red on the migration PR. (Mitigated below; audited.)
- **−** Slightly more validator surface than candidate A — a second rule + handler + its tests beyond the class-range rule.
- **−** A graph-shape invariant fires on the *current* graph state, not on *the diff*; it cannot say "this PR introduced the violation," only "the graph is currently invalid." That is acceptable for a green-graph gate but is less surgical than candidate C's diff-scoped message.

## Risks

- **Retroactive-red on migration (primary).** Making the quorum invariant graph-wide while backfilling class 2 risks reddening an existing selected-but-under-proposed intent. **Mitigation — audit, committed to option (i):** I audited the 4 existing `selects` edges; each resolves through its contract to an intent that already has ≥2 candidate `proposes` edges: `intent-spec-index-validate-a3f1` (2: hardcoded, schema-driven), `intent-claude-lifecycle-commands-f367` (2: thin, fat), `intent-ci-enforcement-gates-5c90` (3: spec-tool, workflow-native, checks-as-nodes), `intent-drift-detection-c7b1` (3: tool-assisted, agent-native, deterministic). All four therefore pass under the class-2 default; no per-intent reclassification to 0/1 is needed (option ii) and no narrowing to `class==3` (option iii) is required. The brief must re-run this audit programmatically as a pre-merge step and fail if any of the four drops below 2, because more selects edges may exist by migration time.
- **Edge-traversal correctness.** The count must include only `proposes` edges targeting the *same* intent, and must include the chosen contract's own `proposes` edge as one of the candidates (siblings are not separately tagged in this graph — candidacy is "shares a `proposes` target"). **Mitigation:** count `spec.edges` where `type === "proposes"` and `target === <intent id>`; unit-test the boundary at exactly 1 (fail) and exactly 2 (pass), and a multi-intent contract case.
- **Rule ordering.** If `class_market_quorum` runs before `edges-references-resolve`, it may traverse an unresolved `selects`/`proposes` endpoint. **Mitigation:** place the rule **after** `edges-references-resolve` in `validation-rules.yaml` (the file comment already states rules run "in the order listed" so each "can assume the preconditions established by the earlier ones"); have the handler defensively skip an edge whose endpoint is absent from `nodesById` rather than throw.
- **Class read must be integer-safe.** The quorum handler reads the intent's `class`; it must treat it as a number (`>= 2`), not via `asString` (which returns `undefined` for numbers). **Mitigation:** read `node.data["class"]` directly with a `typeof === "number"` guard, and rely on `class_range` having already flagged any non-integer.

## Acceptance examples

1. **Intent without a class is rejected.** Capture an intent file with no `class` frontmatter. `pnpm spec:validate` fails on rule `nodes-required-fields` with `node <id> missing required field: class` (from `required_fields.ts`, because `class` is now in the intent `required_fields`). Mechanism: the shared schema edit; no new candidate-specific code needed.
2. **String/float class is rejected.** A contract with `class: "2"` or `class: 2.0` fails on rule `class-in-range` with detail `node <id> field 'class' must be an integer in [0, 1, 2, 3]` (from the new `class_range` handler, since `enum_constraint` is deliberately not reused).
3. **Class 0 runs end-to-end with no market.** A `class: 0` intent is captured (passes validate: present and in range). `/propose-contracts` emits exactly one candidate contract (also `class: 0`); one brief decomposes it. `spec:validate` stays green throughout: the quorum invariant `class_market_quorum` only fires for intents with `class >= 2`, so a class-0 intent with a single `proposes` edge and an eventual `selects` is never required to have a second candidate.
4. **Class 3 cannot be approved with <2 candidates — concrete failure.** Construct a `class: 3` intent with exactly one `proposes` edge from one contract, then add a `decision —selects→` that one contract. `pnpm spec:validate` fails on rule **`class-market-quorum`** with detail e.g. `intent <id> (class 3) has a selected contract but only 1 candidate proposes edge (>=2 required)`. The `selects` edge cannot stand in a green graph, so the approval (which CLAUDE.md records as the `selects` decision) is blocked by CI. Adding a second candidate contract + its `proposes` edge flips the rule green. This is the candidate-B mechanism delivering the headline acceptance criterion as an unbypassable invariant.
5. **Existing graph stays green after backfill.** After the migration commit backfills all intents/contracts to class 2 (smoke intents to 0), `pnpm spec:validate` is green: the 4 pre-existing `selects` edges (`edge-selects-schema-driven-7a12`, `edge-selects-thin-commands-8069`, `edge-selects-ci-gate-spec-tool-0333`, `edge-selects-drift-tool-assisted-d5e2`) each resolve to an intent with ≥2 `proposes` edges (2, 2, 3, 3 respectively per the audit), so `class-market-quorum` emits zero findings for them.
6. **`produced_by` is unvalidated.** A node with `produced_by: spec-writer`, a node with `produced_by:` empty, and a node with no `produced_by` at all all pass `spec:validate` identically — no rule references the field.

## Critique

This critique was drafted by spec-critic against the real graph and `tools/` on 2026-06-16. It reviews candidate B adversarially; it does not rank B against A or C.

### 1. "Candidate count = raw `proposes` edges" is status-blind, and the spec never reconciles that with the intent's wording

The handler `class_market_quorum` is specified to "count **all** `proposes` edges in the graph whose `target` is that intent" — explicitly status-blind. The intent's Acceptance says a Class-3 intent "cannot be approved until at least two **candidate contracts** exist." B equates "candidate" with "shares a `proposes` target." These two definitions can diverge, and B never says which one wins when they do:

- **Supersede-and-replace double-counts.** Per CLAUDE.md rule 3, a wrong contract is *superseded*, not deleted: the new contract is added with a `supersedes` edge and the old one's status moves to `superseded` — but its `proposes` edge "remains" (B's own words). So an intent that has had exactly one *live* idea proposed, then revised once, carries **two** `proposes` edges (old superseded + new) and B's rule reports a satisfied quorum of 2 with only one genuine alternative ever on the table. That is precisely the under-proposed approval the intent wants blocked, and B's status-blind count waves it through. B resolves this case the wrong way ("count `spec.edges` where `type === 'proposes'`").
- **The hazard is real but currently untested.** Every non-selected sibling in the graph is `rejected`, and there is **no `supersedes` edge among contracts anywhere** in `edges.yaml` (the only supersedes edge is intent→intent, `edge-supersedes-capture-smoke-3f0a`). So today raw-edge-count and live-candidate-count happen to coincide (markets of 2/2/3/3 by either measure). B's status-blind design is therefore shipping its one load-bearing simplification against a graph that cannot exercise the case where it breaks. The first superseded contract created after this lands is the first test of a rule the spec leaves under-defined.

B should state explicitly whether the count filters on contract status (and if so, which statuses count) or is deliberately status-blind, and justify the choice against the `supersedes` lifecycle it cites elsewhere in the same document.

### 2. The audit is numerically correct, but the "re-run programmatically as a pre-merge step" admission is quiet scope growth

I verified B's audit against `specs/indexes/incoming.yaml`. The four `selects` edges resolve to intents with these `proposes`-edge counts: `intent-spec-index-validate-a3f1` → 2; `intent-claude-lifecycle-commands-f367` → 2; `intent-ci-enforcement-gates-5c90` → 3; `intent-drift-detection-c7b1` → 3. So B's "2, 2, 3, 3" is **accurate**, and the migration will stay green for these four (Acceptance example 5 is sound). But B then writes: "the brief must re-run this audit programmatically as a pre-merge step and fail if any of the four drops below 2." This is the tell. If `class_market_quorum` is a real graph-wide validate rule, it *already* fails the migration PR the moment any selected class-≥2 intent has <2 `proposes` edges — that is the whole point. A separate "programmatic re-run audit" as a brief pre-merge step is therefore **either redundant with the rule B is shipping, or an admission that the rule alone is not trusted to catch the regression.** Either way it is machinery beyond the advertised "a second rule + handler," and it contradicts B's repeated "rides the existing `spec-validate.yml` — no new workflow" framing. B should delete the extra audit (and rely on the rule) or explain what the rule cannot see that the audit can — it cannot have it both ways while claiming minimal surface.

### 3. Enforcing the quorum for class ≥ 2 (not just class 3) widens the retroactive blast radius beyond the literal Acceptance

The intent's Acceptance names exactly one quorum case: "a **Class 3** intent cannot be approved until at least two candidate contracts exist." B generalizes the invariant to **class ≥ 2**. B defends this as faithful to the routing table (Class 2 also requires ≥2). That is a defensible reading of the *table*, but it is strictly broader than the stated *Acceptance*, and B does not flag the consequence it creates for its own migration: the backfill defaults **every** existing intent to class 2, so a class-≥2 invariant holds **all four** selected intents — not just the ones a human would have called "Class 3" — to the ≥2-candidate bar. The audit shows they pass, so no red graph results *today*. But B's "strictly simpler... pure graph-shape" framing understates that the retroactive surface is not "class-3 intents" but "every selected intent in the graph." A class-3-only invariant (B's rejected "option iii") would have left the class-2-defaulted historical intents entirely out of scope. B should justify why widening to class ≥ 2 — and thereby pulling every backfilled intent into retroactive enforcement — is worth more than matching the literal Acceptance.

### 4. "Cannot be approved" is delivered one beat late: the `selects` edge is authored, *then* the graph turns red

B's mechanism is: graph-maintainer authors the `decision —selects→ contract` edge, and only *then* does `spec:validate` go red because the quorum rule now sees an under-proposed selection. B states this honestly in places ("the approval ... is blocked by CI"), but the headline claim "unbypassable invariant" papers over a real gap: **the approval is recorded** (the `selects` edge exists in `edges.yaml`, the contract may already be flipped to `approved`) and what is blocked is the *commit/PR going green*, not the act of approval itself. This is a genuinely different guarantee from candidate A's `/approve-contract` pre-check (refuses to author the edge at all) and candidate C's diff gate (fails on the *added* edge). B's is a *detective* control on the resulting graph state, not a *preventive* control on the approval act. The spec should not describe it as "cannot be approved"; it can honestly claim only "an under-proposed approval cannot stand in a green graph / cannot merge." The window where the `selects` edge exists but the graph is red (and the contract may already read `approved`) is a state B never names, and graph-maintainer would have to manually unwind the edge it just wrote.

### 5. Handler-correctness claims: two are confirmed, one is wrong, naming is muddied

- **Rule ordering — confirmed.** `runValidation` (`tools/validator.ts`) iterates `spec.rules` in declared order, and `validation-rules.yaml`'s header comment states rules run "in the order listed." So placing `class-market-quorum` after `edges-references-resolve` is sufficient. One caveat: ordering does **not** make endpoints safe by itself — `references_resolve` *reports* unresolved endpoints but does not *remove* them or halt the run. So a `selects` edge with a dangling endpoint still reaches B's handler. B's defensive "skip an edge whose endpoint is absent from `nodesById`" is therefore the *actual* safeguard; the ordering is cosmetic reassurance. B should lead with the defensive skip and not imply ordering alone prevents the throw.
- **`nodesById` — confirmed real.** It is exported from `loader.ts` (`nodesById(spec): Map<string, NodeRecord>`). B's "build `nodesById` (from `loader.ts`)" is correct; no helper needs to be invented.
- **Reading `class` as a number — confirmed.** `fromYaml` uses `CORE_SCHEMA`, so bare `class: 2` arrives as a JS `number`, and reading `node.data["class"]` directly with a `typeof === "number"` guard is correct; `asString` would indeed return `undefined` for it.
- **`2.0` is NOT rejected — this is a factual error.** B states (Scope: "It must reject the string `"2"`, the float `2.0`"; Acceptance example 2; Verification needs: "`2.0` (fail)") that `class: 2.0` is a validation failure. Under `CORE_SCHEMA`, `2.0` parses to the JS number `2`, and `Number.isInteger(2)` is `true` — so the value B's own handler logic (`Number.isInteger(value) && allowed.includes(value)`) **accepts** it. The YAML float `2.0` and the YAML int `2` are indistinguishable by the time the handler sees them. B's "reject `2.0`" is unachievable as specified and its acceptance test #2 / verification matrix entry for `2.0` would fail to fail. B must either drop the `2.0`-rejection claim or specify pre-parse text inspection it has not scoped.
- **Naming inconsistency — flag.** The spec uses `class_range`/`class-in-range` and `class_market_quorum`/`class-market-quorum` interchangeably, and the handler-file/rule-id/kind mapping is left as "working name" in three places. Given the validator dispatches on the exact `kind` string (`HANDLERS[kind]`), a typo there produces an "unknown kind" finding rather than enforcement, so the brief must pin the exact `kind`, rule `id`, and handler filename triple for each rule.

### 6. Missing cases the spec does not resolve

- **Multi-intent contract.** B says "find that contract's outgoing `proposes` edges to get the intent(s) it proposes" and tests "a contract proposing two intents counted independently." But every contract today proposes exactly one intent. B does not say whether such a contract counts as a candidate toward *both* intents' quorums, nor whether that is even legal. This is specified as a test case but not as a *rule*; the brief needs the rule, not just the test.
- **`open` class-3 intent, 1 candidate, no decision.** B asserts the rule fires only when a `selects` edge exists, so an undecided under-proposed intent is correctly silent — confirmed correct — but it means the rule provides **zero** signal during the window the intent is most likely under-proposed (after one contract, before the decision). The "unbypassable invariant" only bites at the selection boundary, identically to candidate C's diff gate timing, despite B framing C's timing as a weakness.
- **A `selects` whose target contract has no `proposes` edge.** The walk `selects → contract → proposes → intent` has no defined behavior if the contract has zero `proposes` edges. B's defensive skip covers a *missing node*, not a *contract with no outgoing proposes*. Undefined: does the rule silently pass such a malformed selection? That is a silent-pass hole the spec should close.

### Net

B's core technical bet — a pure `(rule, spec) => Finding[]` graph-shape handler riding the existing `spec-validate.yml`, using the real `nodesById` export and integer-safe `class` reads — is **sound and matches the codebase**. Its audit numbers are **correct**. Its three sharpest soft spots are: (a) a status-blind candidate count that is under-specified against the very `supersedes` lifecycle CLAUDE.md mandates and is currently untestable in this graph; (b) a "cannot be approved" claim that is really "cannot stay green / cannot merge," with an un-named window where the edge is authored and the graph is red; and (c) a `2.0`-rejection claim that the YAML loader makes unachievable. The "re-run the audit programmatically" line and the class-≥2 generalization both quietly enlarge B's footprint past its own "minimal: one rule + one handler" framing.
